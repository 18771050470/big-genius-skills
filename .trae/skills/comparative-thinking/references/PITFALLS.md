# 比较思维常见陷阱

## 陷阱 1：“镀金”差异 —— 将表面差异当作本质差异

### 问题描述

在比较过程中，过度关注无关紧要的表面特征（如颜色、命名、流行度），而忽略了真正决定结果的核心差异。导致结论偏差或资源浪费。

### ❌ 反例

**场景**：比较两个 REST API 设计 —— 一个使用 `POST /users` 创建用户，另一个使用 `PUT /users`。

错误比较过程：
- 差异点：HTTP 方法不同、请求体结构略有不同、返回 HTTP 码不同
- 草率将“HTTP 方法不同”列为本质差异，认为后者更长（RESTful 约定）
- 决策：选择前者，认为“更标准”

**结果**：实际两个 API 的行为完全一致（都创建新资源），`PUT` 版本只是使用了不同的路由。真正的本质差异应该是幂等性支持，而在这个场景下两者都不提供幂等性，所以 HTTP 方法只是表面约定。

### ✅ 正例

正确比较过程：
- Step 0：明确目的 —— 谁更具幂等性、安全性、可缓存性？
- Step 2：同中求异 —— 共同点：都是创建资源；差异点：HTTP 方法
- Step 4：区分本质与表面：本质差异在于幂等性？实际上 `PUT` 设计为幂等，但该 `PUT` 实现不支持全覆盖幂等（因为没有唯一 ID 在请求中），所以本质上两者都非幂等。因此 HTTP 方法差异是表面差异。
- 结论：选择更符合团队惯例的方法，差异不影响业务。

### 原因分析

- 过度依赖“直觉”而非“影响分析”
- 没有追问“改变这个差异是否改变结果？”
- 受常见知识影响（如 RESTful 规范）而没有结合具体实现

## 陷阱 2：忽略上下文 —— 将静态比较结果绝对化

### 问题描述

在一次比较后，认为某个对象“优于”另一个，忽略时间、环境、规模变化后比较结果可能反转。导致长期决策失误。

### ❌ 反例

**场景**：创业公司 CTO 比较微服务和单体架构。

比较过程：
- 维度：开发速度、部署频率、运维复杂度
- 结论：微服务更好，因为“独立部署、独立扩展、团队独立”
- 直接采用微服务，并投入大量资源拆分

**结果**：公司只有 5 人团队，微服务导致接口协调成本高、调试复杂、CI/CD 管理负担重，开发速度反而下降。一年后团队花了大量精力回迁到单体。

### ✅ 正例

正确比较过程：
- Step 0：明确比较范围 —— “在当前阶段（早期、小团队）”
- Step 3：异中求同 —— 两种架构都旨在使系统可维护可扩展，但微服务在人数不足时反而降低效率
- Step 6：综合洞见 —— “微服务提供长期扩展弹性，但单体在初创期更快。本质差异是团队规模和系统复杂度。”
- 决策：选择单体，设计良好的模块边界，保留在必要时拆分为微服务的路径。
- 额外标注：“此比较结果适用于 0-15 人团队阶段；当团队扩张到 30 人以上后需重新评估。”

### 原因分析

- 只关注了技术模式本身，忽略了组织规模和时间变量
- 没有将结论条件化（If X then A else B）
- 过度依赖静态的“最佳实践”推荐

## 陷阱 3：忽略“共同点” —— 只比较差异导致偏见

### 问题描述

比较时只列差异，忽略大量共同点，容易高估对象之间的差异程度，从而认为“完全不同”，错过基于共性的经验迁移。

### ❌ 反例

**场景**：比较 React 和 Vue 进行前端框架选型。

错误比较过程：
- 差异点：语法（JSX vs 模板）、状态管理（Redux vs Vuex）、社区（Facebook vs Evan You）
- 结论：两者“完全不同”，需要重新学习，选择哪一个都行
- 结果：团队投入大量时间学习，却忽略了它们共用虚拟DOM、组件化、声明式等核心思想，浪费了已有经验

### ✅ 正例

正确比较过程：
- Step 2：找出共同点：虚拟DOM、组件化、声明式UI、响应式数据绑定、单向数据流（都推荐）
- 差异点：实际上是语法糖和生态差异，而非范式差异
- Step 4：本质差异：开发效率差距不大，主要区别在于模板 vs JSX 对代码组织的影响
- 决策：基于团队对 JSX 的熟悉度和现有技能选择 React
- 同时指出：如果团队有较多后端模板引擎经验，Vue 的学习成本更低

### 原因分析

- 人对差异更敏感（认知偏见），忽视共同基础
- 没有强制要求同时列出共同点和差异点
- 缺乏“异中求同”的步骤，没有抽象出通用原则
