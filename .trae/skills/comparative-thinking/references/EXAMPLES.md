# 比较思维应用示例

## 示例 1：后端架构选型 —— Kotlin vs Go

**场景**：技术负责人需要为新的高并发微服务选择编程语言，选项为 Kotlin（JVM生态）和 Go（云原生生态）。

### 输入
- 团队熟悉 JVM 技术栈，但 Go 团队正在组建
- 服务要求：高并发、低延迟、可观测性好
- 部署环境：Kubernetes，重点关注启动速度和内存占用

### Step-by-Step 分析

**Step 0: 明确比较目的与范围**
- 目的：选择最适合当前需求的编程语言
- 范围：仅考虑 Kotlin（JVM）和 Go
- 关键维度：性能、开发效率、生态、运维成本、学习曲线

**Step 1: 分别理解每个对象**
- Kotlin：基于 JVM，与 Java 互操作，支持协程，丰富的库生态，但应用启动慢、内存占用高
- Go：原生并发（goroutine），编译成单一二进制，启动快，内存低，标准库完善，工具链简单

**Step 2: 同中求异——在相似中找到关键差异**
- 共同点：都是静态强类型，都支持现代语言特性，都有活跃社区
- 差异点：

| 维度 | Kotlin | Go |
|------|--------|----|
| 并发模型 | 协程（基于 JVM 线程池） | goroutine（M:N 调度） |
| 启动时间 | 200-500ms（含JVM） | <10ms |
| 内存占用 | 50-150MB（基础服务） | 5-15MB |
| 生态成熟度 | Spring Boot 完善 | Gin/Echo 较年轻 |
| 学习曲线 | 需掌握 JVM 调优 | 语法极简，快速上手 |

- 性质差异：架构层面（JVM托管运行时 vs 原生编译）

**Step 3: 异中求同——发现深层共性**
- 深层共性：都追求更高的开发效率和更好的并发处理能力
- 权衡光谱：Kotlin 侧重"开发便利+生态丰富" → Go 侧重"运行时效率+云原生适配"

**Step 4: 区分本质差异与表面差异**
- 本质差异：启动时间（影响扩缩容效率）、内存占用（影响节点密度）、并发调度模型（影响峰值性能）
- 表面差异：语法风格、工具链名称、社区活跃度数字

**Step 5: 探究差异根源**
- 启动慢是 JVM 类加载和即时编译的代价；Go 的原生编译避免了这一步
- 内存高是 JVM 运行时元空间的代价；Go 的轻量 goroutine 栈起始仅 2KB
- Go 的标准库自带 net/http、微服务制式，Kotlin 依赖 Spring Boot 等框架

**Step 6: 综合形成洞见与决策建议**
- **核心洞见**：如果当前服务对扩缩容速度和集群成本（节点数）敏感，Go 是更优选择；如果团队已有 JVM 能力且需要 Spring 生态集成，则 Kotlin 更适合。
- **决策建议**：鉴于该服务是独立微服务、对延迟和成本较为敏感，推荐使用 Go。同时建议团队补充 Go 技能。

### 最终输出

```markdown
## 比较总结表

| 维度 | Kotlin | Go | 影响等级 |
|------|--------|----|----------|
| 启动时间 | 200-500ms | <10ms | 高 |
| 内存占用 | 50-150MB | 5-15MB | 高 |
| 峰值吞吐 | 近似（协程vs goroutine） | 近似（可扩展） | 中（调优后持平） |
| 开发效率 | 高（生态丰富） | 中高（标准库简单） | 中 |
| 学习成本 | 中（JVM 知识依赖） | 低 | 低（新团队可承受） |

**推荐：Go**（理由：启动快、内存少、原生云友好，适合目标场景）
```

## 示例 2：UI 设计模式比较 —— 标签页（Tabs） vs 手风琴（Accordion）

**场景**：UX 设计师在移动端设置页面中需要选择信息组织方式。

### 输入
- 移动端屏幕尺寸有限
- 内容分为 5 个逻辑组，每组含 3-8 项设置
- 用户平均访问 2-3 个组，不希望一次性展示所有

### Step-by-Step 分析

**Step 0: 明确比较目的与范围**
- 目的：选择最佳信息组织方式以提升操作效率
- 范围：限定在标签页和手风琴两种常用模式
- 维度：空间利用率、导航效率、信息可见性、错误率

**Step 1: 分别理解每个对象**
- 标签页：顶部或底部显示标签栏，点击切换内容区
- 手风琴：垂直列表，点击展开当前项，收起其他项

**Step 2: 同中求异——在相似中找到关键差异**
- 共同点：都是通过折叠/切换节省空间，支持分组信息
- 差异点：

| 维度 | 标签页 | 手风琴 |
|------|--------|--------|
| 空间 | 标签栏固定占位，内容区全屏 | 展开项占用动态高度，需滚动 |
| 导航 | 点击标签直接切换 | 需点击展开/收起 |
| 可见性 | 所有标签名始终可见 | 只能看到标题，内容折叠 |
| 同时访问 | 只能看一个组 | 可同时展开多个（改进版） |

**Step 3: 异中求同——发现深层共性**
- 深层共性：都希望用户在有限空间快速找到并操作目标设置
- 权衡光谱：标签页侧重“导航恒定性” vs 手风琴侧重“内容沉浸感”

**Step 4: 区分本质差异与表面差异**
- 本质差异：导航方式不同（切换 vs 展开/收起）；标签页无法同时看到两个组的内容，手风琴可以但需要滚动
- 表面差异：视觉样式、动效、平台惯例

**Step 5: 探究差异根源**
- 标签页源自桌面（空间不足时采用），移动端受限于横向空间，标签文字容易遮挡
- 手风琴天然适合长列表，且符合移动端单手操作（滚动点击）

**Step 6: 综合形成洞见与决策建议**
- **核心洞见**：当用户需要频繁切换组、且组名较短时，标签页效率高；当内容高度不均、用户更多是逐项浏览时，手风琴更灵活。
- **决策建议**：本场景涉及 5 组，组名较长（如“语言与地区”），且用户倾向于逐组设置，推荐使用手风琴，并支持同时展开多个组（非默认）。

### 最终输出

```markdown
## 比较总结表

| 维度 | 标签页 | 手风琴 |
|------|--------|--------|
| 移动端导航效率 | 中（标签栏横向空间不足） | 高（垂直滚动自然） |
| 内容可见性 | 只显示当前组 | 可展开多个组对比 |
| 空间灵活度 | 固定标签栏 | 高度动态适配 |
| 适用场景 | 少量、短标题、频繁切换 | 多组、长标题、逐项操作 |

**推荐：手风琴**（理由：更适合移动端长列表设置页，且用户操作路径匹配）
```

## 反例：不该触发此 Skill 的场景

### 场景：挑选咖啡 —— “美式咖啡” vs “拿铁”

**输入**：朋友问“你今天想喝美式还是拿铁？”

**本应触发的思维**：偏好判断（基于个人口味、天气、时间），不需要系统比较。

**如果误用比较思维**：
- Step 0 目的：比较两者优劣 → 但优劣无关（纯主观）
- Step 2-5 分析成分差异、咖啡因含量、热量 → 浪费时间
- Step 6 输出：“如果你想提神且低卡，选美式；如果你喜欢奶味，选拿铁。” → 结论正确但过程过度工程化

**正确做法**：直接根据当前心情选择，或询问简单偏好即可。比较思维适用于需要客观分析、存在明确维度的场景，不适合纯个人偏好决策。
