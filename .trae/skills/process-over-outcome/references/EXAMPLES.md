# 分析示例

## 示例 1：过程优于结果——项目管理评估

**输入**：这个项目虽然延期了2周，但最终交付质量很好，应该评为优秀项目。

```
Step 1: 识别结果信号 → 产出：发现"交付质量好...评为优秀"表述，仅基于结果评价，忽视过程问题
Step 2: 评估过程质量 → 产出：延期2周反映计划或执行问题，可能不可持续
Step 3: 分析过程机制 → 产出：结果偏差：好结果掩盖过程问题；运气因素：好结果可能部分靠运气
Step 4: 校正过程vs结果 → 产出：建议分别评估过程和结果，识别可复制的成功因素
```

**输出**：
```
## 分析结论
### 核心发现
1. 结果好(质量优秀)但过程有问题(延期2周)
2. 延期原因：需求变更频繁(过程问题)+资源不足(系统问题)
3. 好结果部分靠团队加班(不可持续)和部分运气

### 建议（按优先级）
1. [高优先级] 分别评估：结果(优秀)+过程(需改进)
2. [高优先级] 识别过程问题：需求管理、资源规划
3. [中优先级] 建立过程指标，而非仅看结果

### 关键假设
- 延期反映过程问题
- 加班不可持续

### 风险
- 仅看结果可能掩盖过程问题
- 下次可能无法复制好结果
```

## 示例 2：过程优于结果——投资决策

**输入**：这只股票我随便买的，结果涨了50%，说明我投资眼光不错。

```
Step 1: 识别结果信号 → 产出：发现"涨了50%...眼光不错"表述，将好结果归因于能力
Step 2: 评估过程质量 → 产出："随便买的"反映无分析过程，好结果可能靠运气
Step 3: 分析过程机制 → 产出：结果偏差：好结果使过程被合理化；归因错误：将运气归因于能力
Step 4: 校正过程vs结果 → 产出：建议评估决策过程，而非仅看结果
```

**输出**：
```
## 分析结论
### 核心发现
1. 决策过程："随便买的"=无分析过程
2. 结果：涨50%=好结果
3. 好结果更可能来自运气而非能力

### 建议（按优先级）
1. [高优先级] 好结果≠好决策，需评估决策过程
2. [高优先级] 建立投资分析框架，而非随机选择
3. [中优先级] 记录决策过程，便于事后复盘

### 关键假设
- "随便买的"反映无分析过程
- 单次结果不能证明能力

### 风险
- 将运气归因于能力可能导致未来亏损
- 需要区分技能和运气
```

## 反例：不该触发此 Skill 的场景

**输入**：这个项目按时交付，质量达标，客户满意，评为优秀。

**原因**：这是过程和结果都好的情况，不需要过程vs结果的权衡分析。
