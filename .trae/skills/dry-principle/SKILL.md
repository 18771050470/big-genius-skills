# DRY原则思维（DRY Principle Thinking）

## 核心定义

系统中每一项知识都必须具有单一、无歧义、权威的表示——消除重复不仅是为了减少代码量，更是为了确保变更只需在一处进行，避免遗漏和不一致。

---

## 思考流程控制

### 中断处理

- **信息不足**：用 AskUserQuestion 向用户确认"这是知识重复还是偶然相似？"
- **发现矛盾**：停止思考，用 AskUserQuestion 向用户报告矛盾点（如"DRY要求合并，但合并后耦合度上升"），等待决策
- **提前终止**：若重复仅为偶然的语法相似（如两个独立业务逻辑恰好有相同循环结构），可提前终止

---

## 思考流程

### Step 1: 识别重复类型

- 识别代码重复：完全相同的代码块、结构相似但变量不同的代码、算法逻辑重复
- 识别数据重复：数据库中冗余字段、配置信息分散、常量魔法数字多处定义
- 识别文档重复：注释与代码不一致、文档复制粘贴后未更新、API文档与实现不同步
- 识别流程重复：相同的手动操作步骤、重复的审批流程、复制粘贴的业务规则
- 陷阱：将偶然的语法相似误判为知识重复（如两个独立业务逻辑都使用了for循环）

### Step 2: 评估重复成本

- 计算维护成本：当前重复导致的变更工作量（改N处 × 频率）
- 评估不一致风险：不同副本可能演化出差异，导致逻辑冲突或数据不一致
- 考虑测试成本：重复代码需要重复测试，增加测试维护负担
- 陷阱：忽视"重复但稳定"的代码（如标准库包装函数），过度抽象反而增加复杂度

### Step 3: 设计单一权威来源

- 提取公共函数/类：将重复代码提取为可复用的函数、类或模块
- 使用配置化：将变化的数据提取到配置文件、数据库或环境变量
- 代码生成：对于结构性重复（如CRUD代码），使用代码生成器而非手动复制
- 继承 vs 组合：选择合适的方式复用代码（优先组合）
- 陷阱：为消除小量重复而创建过度复杂的抽象，导致"抽象泄漏"

### Step 4: 处理耦合与内聚

- 评估抽象后的耦合度：消除重复是否引入了不自然的依赖关系？
- 检查内聚性：提取的公共模块是否职责单一？是否包含无关功能？
- 考虑变化频率：如果两个重复代码的变化频率不同，强制合并可能适得其反
- 陷阱：将变化频率不同的代码合并，导致一处变更影响另一处无关功能

### Step 5: 验证与测试

- 确保功能等价：重构后，所有原有功能是否保持不变？
- 回归测试：对受影响的所有调用点进行测试
- 文档同步：更新文档，确保新的单一权威来源被正确引用
- 陷阱：重构后未进行充分测试，导致隐藏的依赖关系被破坏

### Step 6: 总结输出

- 整合前面各步的分析结果，形成完整的DRY化方案
- 不遗漏重复类型、成本评估和抽象设计
- 明确下一步应该调用哪个 Skill（如需要重构技术则调用 refactoring）
- 输出应包含：重复清单 + 抽象方案 + 耦合评估 + 测试策略 + 风险 + 下一步

## 关键原则

1. **知识单一权威**：不仅是代码，文档、配置、流程中的知识都应只有一处权威来源。原因：避免变更时遗漏，确保一致性。
2. **三法则**：第一次写代码时直接写，第二次复制粘贴时考虑提取，第三次出现时必须提取。原因：过早抽象会增加不必要的复杂度，过晚抽象会导致技术债务累积。
3. **DRY ≠ 消除所有相似**：偶然的语法相似（如两个独立业务逻辑都使用循环）不应强制抽象。原因：过度抽象会降低代码可读性，引入不自然的耦合。
4. **抽象要符合领域语言**：提取的公共模块命名应反映业务概念，而非技术实现。原因：符合领域语言的抽象更容易被理解和维护。

## 自检清单

**执行规则**：
- **执行方**：Agent 自检，在最终输出前逐项确认
- **未通过处理**：任一检查项未通过，必须修正后重新自检，禁止跳过
- **记录方式**：自检结果写入 `self_check_result.md`（临时文件），包含：通过项、未通过项、修正记录
- **强制停止项**：以下检查项未通过时，必须停止输出并修正：
  - [ ] 已按思考流程完成全部步骤，无跳过（或已标注提前终止原因）
  - [ ] 输出具体可执行，不含模糊建议
  - [ ] 标注了关键假设和不确定性

**完整检查项**：

- [ ] 已按思考流程完成全部步骤，无跳过（或已标注提前终止原因）
- [ ] 每步思考要点已覆盖，无遗漏
- [ ] 每步结论标注了置信度和反证可能性
- [ ] 考虑了二阶/三阶效应（如果适用）
- [ ] 输出具体可执行，不含模糊建议（如"优化一下"）
- [ ] 标注了关键假设和不确定性
- [ ] 识别了最大风险点和缓解措施
- [ ] 建议按优先级排序，不是平铺罗列
- [ ] 明确了下一步应该调用哪个 Skill（如果需要）

**DRY原则特有关键检查项**：

- [ ] **DRY特供**：是否区分了"知识重复"与"偶然相似"？两个独立业务逻辑使用相同语法结构是否被误判为重复？
- [ ] **DRY特供**：是否评估了重复成本？维护成本（改N处×频率）、不一致风险、测试负担是否被量化？
- [ ] **DRY特供**：是否设计了单一权威来源？提取为函数/类/配置/代码生成器的方式是否最适合当前场景？
- [ ] **DRY特供**：是否检查了耦合与内聚？消除重复是否引入了不自然的依赖？提取的模块是否职责单一？
- [ ] **DRY特供**：是否应用了三法则？第一次直接写、第二次考虑提取、第三次必须提取的原则是否被遵循？
- [ ] **DRY特供**：是否验证了功能等价？重构后是否进行了回归测试？所有调用点是否被验证？
