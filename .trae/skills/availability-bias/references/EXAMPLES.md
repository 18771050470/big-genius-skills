## EXAMPLES.md

### 示例 1：技术团队风险估算（开发新功能）

**场景**：某后端团队计划开发一个支付微服务。最近一次线上事故（两周前）是因数据库死锁导致服务中断2小时，团队印象深刻。技术负责人根据回忆估算“支付功能开发中遇到死锁的概率”为70%。

#### 思考流程应用

| 步骤 | 输入 | 动作 | 输出 |
|------|------|------|------|
| 1. 识别判断任务 | “支付功能开发过程中遇到死锁的可能性” | 明确需评估的是“开发期间遇到死锁”的概率，非线上故障率 | 任务：估算死锁在开发期间出现的概率 |
| 2. 列出快速联想 | 记忆中最先浮现的例子：最近死锁事故；同事A提过死锁难排查 | 列出联想：① 2周前事故 ② 同事A的抱怨 ③ 网络上死锁案例多 | 初始联想：死锁很常见 |
| 3. 评估提取难度 | 上述联想 | 为什么容易想起？① 最近发生（近因效应）② 事故带来明显损失（情绪突出）③ 技术社区近期讨论过死锁 | 偏差来源：近因+情绪+媒体 |
| 4. 系统搜索反例 | 团队历史数据、同行统计 | 收集：团队过去2年共开发12个微服务，只有1次死锁发生在线上；该事故是由于遗漏索引导致，新功能已要求索引检查流程 | 反例：12个服务中仅1次死锁，且已改进流程 |
| 5. 统计校准 | 初始70% 与 Base Rate | Base Rate = 1/12 ≈ 8.3%。考虑新功能复杂度类似，校准为10%（附带索引检查后降至5%） | 校准后概率：10%（非70%） |
| 6. 记录与复盘 | 校准结果与初始差异 | 记录：本次判断差60个百分点，根本原因是近因效应。建议以后风险估算前先查历史数据清单 | 改进点：建立“发生概率基线台账” |

**最终输出**：支付功能开发中遇到死锁的概率应校准为 **10%**（而非直觉的70%）。基于此结果，团队决定投入少量时间做预防设计，而非过度设计死锁处理方案。

### 示例 2：产品经理预测市场渗透率

**场景**：某产品经理规划“AI客服聊天机器人”功能，最近接连看到3篇媒体报道称“AI客服已覆盖80%的企业”，因此预测自家产品若不快速上线AI客服，用户流失风险很高。

#### 思考流程应用

| 步骤 | 输入 | 动作 | 输出 |
|------|------|------|------|
| 1. 识别判断任务 | “目前市场上AI客服的渗透率” | 明确是整体渗透率，非头部企业案例 | 任务：估算所有中小企业中AI客服使用率 |
| 2. 列出快速联想 | 脑海中最先浮现：那3篇报道；朋友公司刚上线AI客服 | 联想：① 某大厂案例 ② 朋友公司 ③ 行业会议频繁提AI | 初始联想：AI客服很普及 |
| 3. 评估提取难度 | 上述联想 | 为什么容易想起？① 媒体报道有选材偏见 ② 朋友公司属于熟人经验 ③ 行业会议参与者属于先行者 | 偏差来源：媒体选择+熟人偏差+样本偏差 |
| 4. 系统搜索反例 | 行业调查数据、沉默多数 | 搜索：最新的第三方调研显示，中小企业AI客服渗透率仅为18%；客服外包公司反馈仍有75%客户使用传统电话 | 反例：实际渗透率不到20% |
| 5. 统计校准 | 初始估计60% 与 Base Rate | Base Rate = 18%。考虑目标客户群（中型企业）稍高，校准为25% | 校准后渗透率：25%（非60%） |
| 6. 记录与复盘 | 校准结果与初始差异 | 记录：直觉高估35个百分点，原因是对媒体案例未做代表性审查。建议以后优先引用权威统计 | 改进点：在竞争分析模板中增加“Base Rate数据源”字段 |

**最终输出**：实际AI客服渗透率约 **25%**（而非直觉的60%）。基于此，产品经理决定不盲目赶鸭子上架，而是制定合理的渐进式上线计划，并优先评估用户真实需求。

### 反例：不该触发可得性偏差的场景

**场景**：团队进行“新商业模式创意构思”的头脑风暴，主持人让大家说出联想到的任何可能性。参与者A说“共享经济模式”，B说“订阅模式”，C说“众包模式”。此时有人提出：“我们刚才只提到了常见的模式，是不是应该先用可得性偏差模型来校准我们的想法？”

**为什么不该触发**：
- 头脑风暴阶段的目标是**发散**，鼓励各种联想（包括最近、生动、容易想起的），此时越多联想越好，抑制联想反而扼杀创意。
- 可得性偏差主要在**评估概率/频率**时用来校准判断，而不是用来过滤创意。
- 正确做法：先放开联想收集所有可能，然后在**筛选/评估阶段**再应用可得性偏差来修正对某些方案概率的主观判断。

因此，在此场景中不应调用可得性偏差模型。
