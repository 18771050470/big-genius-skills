# 移动端工程师常见陷阱

## 1. 跨平台选择错误

### 反例：盲目追求跨平台
- **错误**：不考虑团队技术栈和项目需求，盲目选择跨平台框架
- **案例**：团队熟悉 Swift，却选择 Flutter 开发 iOS 独占应用
- **原因**：忽视了团队学习成本和原生性能优势

### 正例：合理选型
- **正确**：根据团队背景、项目需求、时间预算选择合适方案
- **案例**：团队有 React 背景，需要双平台 → React Native；追求极致性能 → 原生
- **要点**：没有最好的框架，只有最合适的框架

## 2. 忽视平台差异

### 反例：完全一致的 UI
- **错误**：iOS 和 Android 使用完全相同的 UI 和交互
- **案例**：Android 应用使用 iOS 风格的返回按钮，用户感到困惑
- **原因**：忽视了两个平台的设计规范和用户习惯

### 正例：平台适配
- **正确**：核心逻辑复用，UI 和交互按平台规范实现
- **案例**：iOS 使用底部 Tab Bar，Android 使用顶部 Tab + 侧滑菜单
- **要点**：尊重平台规范，用户期望一致的体验

## 3. 性能优化滞后

### 反例：上线后再优化
- **错误**：开发阶段不关注性能，上线后才想起来优化
- **案例**：应用上线后用户反馈卡顿，才发现列表没有虚拟化
- **原因**：性能问题后期修复成本远高于前期预防

### 正例：性能优先
- **正确**：开发阶段就建立性能基准，持续监控
- **案例**：使用 Flipper/Performance Monitor 监控帧率，列表超过 50 项就虚拟化
- **要点**：性能不是可选项，是必选项

## 4. 原生模块管理混乱

### 反例：随意引入原生库
- **错误**：随意引入第三方原生库，导致依赖冲突和维护困难
- **案例**：引入 5 个不同的图片处理库，包体积增加 20MB
- **原因**：没有统一的依赖管理策略

### 正例：统一管理
- **正确**：建立原生模块的评估和引入流程
- **案例**：引入新库前评估：功能是否必需？社区活跃度？包体积影响？
- **要点**：每个原生依赖都是长期维护成本

## 5. 忽视上架合规

### 反例：临时准备上架材料
- **错误**：开发完成后才临时准备上架材料，导致审核被拒
- **案例**：隐私政策不完整、截图不符合规范、描述含有敏感词
- **原因**：忽视了上架材料的准备时间和审核要求

### 正例：提前准备
- **正确**：开发阶段就准备上架材料，预留审核时间
- **案例**：开发完成前一周准备截图、描述、隐私政策，预留 3 天审核缓冲
- **要点**：上架不是最后一步，是持续的过程
