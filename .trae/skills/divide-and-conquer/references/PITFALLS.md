# 分治思维 - 常见陷阱

## 陷阱 1：拆分过度（Over-Decomposition）

**问题**：把问题拆得太细，子问题过多导致接口管理和组合成本超过拆分收益。

❌ **Before（错误做法）**
```
一个简单的CRUD用户模块，拆分为：
- UserController（10行）→ UserService → UserValidator（15行）→ UserRepository（8行）→ UserMapper（5行）
→ 5个文件总共不到100行代码，但拆成了5层
→ 改一个字段要改5个文件
```

✅ **After（正确做法）**
```
一个文件UserService.ts，100行代码，包含所有逻辑
→ 拆分是为了降低复杂度，不是为了拆分而拆分
→ 当子问题的接口管理成本 > 直接实现成本时，停止拆分
```

**原因**：Minsky在《The Society of Mind》中指出，有效的分解需要子问题的复杂度"显著低于"原问题。如果拆分没有显著降低每个子问题的认知负荷，那就是无效拆分。

## 陷阱 2：接口定义不清就开始实现

**问题**：急于开始编码，子问题间的接口边做边定，最后发现拼不起来。

❌ **Before（错误做法）**
```
"我们分头做吧，你做用户模块，我做订单模块，接口到时候再说。"
→ 3天后：用户模块返回的用户对象含10个字段，订单模块只用到3个 → 白传了7个字段
→ 而且订单模块需要的用户等级字段，用户模块根本没返回 → 返工
```

✅ **After（正确做法）**
```
接口先行：
1. 先定义接口契约（Contract）：
   GET /users/{id} → {id, name, level, status} // 4个字段，双方确认
2. 用Mock Server让双方可以并行开发
3. 接口测试先行：写集成测试后再各自实现
```

**原因**：分治的最大风险不是"拆不对"，而是"拼不起来"。接口（契约）是分治的"胶水"，胶水不好，碎片就是碎片。

## 陷阱 3：忽视子问题间的隐藏依赖

**问题**：以为子问题是独立的，但实际存在隐性耦合，导致并行开发的进度被串行依赖拖垮。

❌ **Before（错误做法）**
```
"用户模块和订单模块是独立的，可以并行开发。"
→ 实际上：订单模块需要"用户收货地址"功能，而这个功能在用户模块中排在第2周
→ 订单模块第1周干等，并行变串行
```

✅ **After（正确做法）**
```
依赖分析：
1. 画依赖关系图：订单模块 → 依赖 → 用户模块的"收货地址"API
2. 调整计划：用户模块优先交付"收货地址"API（第1周），完整"用户资料"延后
3. 或者：订单模块先用Mock地址开发，第2周再切换真实API
```

**原因**：表面的功能独立掩盖了深层的接口依赖。分治的质量取决于依赖分析的质量——没有充分依赖分析的分治是"假装并行"。
