# 分治思维 - 分析示例

> 至少 2 个示例，展示从输入到输出的完整推理链。

## 示例质量验证标准

| 检查项 | 标准 |
|--------|------|
| **完整性** | 包含输入→每步动作→每步输出→最终结论 |
| **可追踪性** | 最终结论必须能追溯到具体步骤的产出 |
| **具体性** | 输出含具体数据/代码/指标，非泛泛而谈 |
| **优先级** | 建议必须按优先级排序，说明排序依据 |
| **边界覆盖** | 包含至少 1 个"不该触发此 Skill"的失败案例 |

## 示例 1：大型需求拆分为迭代任务

**输入**：一个电商平台的"会员积分体系"需求，包含：积分获取规则、积分商城、积分过期策略、积分预警通知、积分与优惠券联动、积分排行榜。需求文档60页。

```
Step 1: 界定问题边界
→ 产出：
  - 完整范围：6个子系统，涉及前端/后端/数据/运营后台
  - 核心目标：提升用户留存率（通过积分激励）
  - 约束：3个月内上线MVP，团队5人

Step 2: 确定拆分策略
→ 产出：
  - 按功能维度拆分（MECE验证）：
    1. 积分获取引擎（行为→积分映射）
    2. 积分账户管理（余额、流水、过期）
    3. 积分商城（兑换列表、下单、发货）
    4. 积分通知（即将过期提醒、获取提醒）
    5. 积分排行榜（日榜/周榜/月榜）
    6. 运营管理后台（规则配置、数据看板）
  - 并行可能：1和6可以并行启动；2依赖1；3依赖2；4依赖2；5依赖1

Step 3: 定义子问题接口
→ 产出：
  - 接口1→2：积分获取事件 {userId, eventType, points, timestamp}
  - 接口2→3：积分余额查询 API，积分扣减 API
  - 接口2→4：积分过期事件 {userId, points, expireDate}
  - 接口1→5：积分变更事件流
  - 终止条件：每个子模块可被单人2周内完成

Step 4: 逐个解决子问题
→ 产出：
  - Sprint 1（2周）：模块1（积分获取引擎）+ 模块6（管理后台MVP）
  - Sprint 2（2周）：模块2（积分账户）+ 模块4（通知）
  - Sprint 3（2周）：模块3（商城）+ 模块5（排行榜）
  - 模块3在实现时发现商品库存管理比我预计复杂 → 进一步拆分为：商品管理 + 兑换流程

Step 5: 合并子问题结果
→ 产出：
  - 集成测试发现：模块1和模块2之间的事件格式有一个字段名不一致（eventType vs type）
    → 修正接口定义，统一为eventType
  - 模块3调用模块2的积分扣减API时，事务边界不清晰
    → 引入分布式事务方案（TCC或SAGA）
  - 验证：全部6个模块集成后，端到端业务流程跑通
```

**输出**：

```
## 分析结论

### 拆分方案
1. 6个模块按依赖关系分3个Sprint交付
2. 每个模块定义清晰的输入/输出接口
3. 并行开发：Sprint 1中模块1和6并行；Sprint 2中模块2和4并行；Sprint 3中模块3和5并行

### 关键风险
- 接口定义不一致（已通过集成测试阶段发现和处理）
- 模块3复杂度被低估（已通过递归拆分解决）

### 关键假设
- 模块间接口能被严格遵守（需要Code Review + 接口测试保障）
- 团队成员技术栈匹配各模块需求
```

## 示例 2：复杂Bug排查

**输入**：一个微服务系统的API超时率突然上升到5%，涉及8个微服务，不知道哪个环节出了问题。

```
Step 1: 界定问题
→ 产出：
  - 范围：8个微服务的整条调用链
  - 目标：定位超时的根因服务

Step 2: 拆分策略
→ 产出：按调用链路二分拆分
  - 上游服务组：网关、认证、路由
  - 下游服务组：订单、库存、支付、通知
  - 先检测上游：如果上游都正常 → 问题在下游
  - 再二分下游：订单+库存 vs 支付+通知

Step 3: 子问题接口
→ 产出：
  - 每个服务暴露：P99延迟、错误率、QPS
  - 检测标准：P99 < 200ms为正常

Step 4: 逐个排查
→ 产出：
  - 上游组正常（P99: 网关30ms, 认证50ms, 路由20ms）
  - 下游组中：订单P99=200ms正常，库存P99=3000ms异常
  - 定位：库存服务是问题源
  - 继续拆库存服务：数据库查询 vs 缓存查询 vs 第三方调用
    发现：第三方物流接口响应时间从200ms变成10s

Step 5: 合并结论
→ 产出：
  - 根因：第三方物流接口降级
  - 解决方案：对第三方调用加3s超时 + 熔断降级
```

**输出**：

```
## 分析结论
根因：第三方物流API响应时间从200ms升至10s，导致库存服务线程池耗尽，级联影响上游。
方案：加超时+熔断，库存服务降级为"物流信息稍后刷新"模式。
拆分策略：二分法沿调用链逐步缩小范围，3轮定位根因。
```

## 反例：不该触发分治思维的场景

**输入**："一个简单的登录页面，要不要拆分为：表单组件 + 验证组件 + 提交组件？"

**原因**：登录页面足够简单，拆分的沟通成本和组件间传参复杂度超过直接实现的成本。分治思维的核心价值在"大问题"，拆分复杂度本身比问题的复杂度还高时，不要分治。
