# 常见陷阱

## 陷阱 1：忽视空值/Null

**问题**：未处理空输入导致崩溃。

❌ **Before（错误做法）**
```markdown
## 分析结论
用户名不能为空，但代码未检查。
建议：用户自己注意。
```
*问题：外部输入不可控，空值会导致 NPE 或逻辑错误*

✅ **After（正确做法）**
```markdown
## 分析结论
### 空值处理
| 输入 | 处理策略 |
|------|---------|
| 用户名 null | 返回 400: "用户名不能为空" |
| 用户名 "" | 返回 400: "用户名不能为空" |
| 用户名 " " | trim 后检查，若空则报错 |

### 建议
1. 入口处统一校验空值
2. 使用 Optional 或卫语句
```
*优势：防御式校验避免运行时崩溃*

**原因**：Myers(2011)：空值是最常见的边界情况，未处理空值是软件缺陷的首要原因之一。

## 陷阱 2：假设资源无限

**问题**：未考虑内存/磁盘/连接数限制。

❌ **Before（错误做法）**
```markdown
## 分析结论
把 100 万条数据全加载到内存处理。
建议：直接 List.loadAll()。
```
*问题：大数据量会导致 OOM 或性能急剧下降*

✅ **After（正确做法）**
```markdown
## 分析结论
### 资源边界
| 资源 | 限制 | 策略 |
|------|------|------|
| 内存 | 2GB | 流式处理/分页 |
| 连接池 | 50 | 限制并发/排队 |
| 磁盘 | 80% | 监控/清理/告警 |

### 建议
1. 使用流式处理替代全量加载
2. 设置分页大小限制
```
*优势：资源感知设计避免系统崩溃*

**原因**：Nygard(2018)：资源耗尽是生产环境故障的主要原因。设计时必须考虑资源上限。

## 陷阱 3：静默失败

**问题**：错误被吞掉，问题累积后爆发。

❌ **Before（错误做法）**
```markdown
## 分析结论
catch (Exception e) { /* 忽略 */ }
建议：不影响主流程就行。
```
*问题：静默失败导致问题难以排查，可能掩盖严重错误*

✅ **After（正确做法）**
```markdown
## 分析结论
### 失败处理
| 错误类型 | 处理方式 |
|---------|---------|
| 可恢复 | 重试 + 日志 |
| 不可恢复 | 快速失败 + 告警 |
| 预期内 | 返回明确错误码 |

### 建议
1. 禁止空 catch 块
2. 所有异常必须记录日志或向上抛出
```
*优势：快速暴露问题便于排查*

**原因**：Google SRE(2016)：静默错误比显式错误更危险，因为它掩盖了系统真实状态，导致故障扩大。
