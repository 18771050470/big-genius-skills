# 分析示例

## 示例 1：边界思维——API接口设计

**输入**：设计一个用户注册API，接收用户名、邮箱、密码。

```
Step 1: 识别输入边界 → 产出：用户名：空值、超长(>50字符)、特殊字符、重复；邮箱：格式错误、已注册、超长；密码：过短(<8位)、过长(>128位)、纯数字、包含注入字符
Step 2: 识别资源边界 → 产出：并发注册：同一邮箱并发请求；数据库：连接池耗尽、写入冲突；存储：用户表大小限制
Step 3: 识别时间边界 → 产出：数据库写入超时、邮箱验证超时、并发竞争条件（同时注册同一邮箱）
Step 4: 设计边界防护 → 产出：输入验证：长度限制、格式校验、SQL注入过滤；并发控制：邮箱唯一性约束+分布式锁；超时处理：3秒超时+重试1次+失败返回明确错误
```

**输出**：
```
## 分析结论
### 核心发现
1. 输入边界：用户名/邮箱/密码各有5+种异常输入场景
2. 资源边界：并发注册可能导致唯一性冲突
3. 时间边界：数据库写入超时需处理

### 建议（按优先级）
1. [高优先级] 输入验证层：长度、格式、注入过滤，失败立即返回400
2. [高优先级] 并发控制：数据库唯一索引+应用层分布式锁
3. [中优先级] 超时处理：3秒超时+明确错误信息
4. [中优先级] 限流：单IP每分钟最多5次注册

### 关键假设
- 数据库支持唯一索引
- 分布式锁可用(Redis)

### 风险
- 验证规则过严可能误杀合法输入
- 需要平衡安全和用户体验

### 下一步
建议调用 [defensive-programming] 实现边界防护代码
```

## 示例 2：边界思维——数据处理管道

**输入**：设计一个每日处理100万条订单的数据管道。

```
Step 1: 识别输入边界 → 产出：空文件、超大文件(>10GB)、格式错误、重复数据、缺失字段、编码异常
Step 2: 识别资源边界 → 产出：内存：100万条全加载可能OOM；CPU：处理耗时超过24小时窗口；磁盘：中间结果存储不足
Step 3: 识别时间边界 → 产出：处理超时、上游延迟到达、时钟跳变导致日期计算错误
Step 4: 设计边界防护 → 产出：流式处理避免OOM；分片并行处理；断点续传；异常数据隔离
```

**输出**：
```
## 分析结论
### 核心发现
1. 输入边界：空文件、超大文件、格式错误、重复数据
2. 资源边界：100万条全加载OOM风险、24小时处理窗口
3. 时间边界：上游延迟、处理超时

### 建议（按优先级）
1. [高优先级] 流式处理：逐条/分批处理，避免全加载
2. [高优先级] 分片并行：按日期/类型分片，并行处理
3. [中优先级] 断点续传：记录处理进度，失败可从断点恢复
4. [中优先级] 异常隔离：错误数据写入dead letter queue，不阻塞正常流程

### 关键假设
- 数据可流式读取
- 分片键均匀分布

### 风险
- 流式处理增加复杂度
- 需要监控处理进度

### 下一步
建议调用 [boundary-thinking] 设计监控告警
```

## 反例：不该触发此 Skill 的场景

**输入**：这个函数接收一个正整数，返回它的平方。

**原因**：这是简单数学函数，边界情况极少（仅需处理负数和0）。边界思维适用于复杂系统、外部接口、数据处理等边界场景丰富的场景。
