# 分析示例

## 示例 1：现状偏差——技术栈选择

**输入**：我们一直用这个老框架，虽然有新框架更好，但迁移成本太高，还是继续用老的吧。

```
Step 1: 识别现状信号 → 产出：发现"一直用...还是继续用老的"表述，偏好维持现状而非改变
Step 2: 评估改变vs维持 → 产出：新框架收益：开发效率+30%、bug减少20%；迁移成本：2个月+学习成本
Step 3: 分析现状机制 → 产出：损失厌恶：迁移成本(确定损失)被过度加权；沉没成本：已投入老框架的时间影响决策
Step 4: 校正现状偏差 → 产出：建议用长期收益评估，而非短期迁移成本
```

**输出**：
```
## 分析结论
### 核心发现
1. 迁移成本2个月(确定损失)被过度加权，长期收益(年节省3个月)被低估
2. 现状偏差使维持现状被感知为"安全"，但技术债务持续积累
3. Samuelson & Zeckhauser(1988)：即使改变有明确收益，人们仍倾向维持现状

### 建议（按优先级）
1. [高优先级] 计算长期ROI：迁移成本2个月 vs 年收益3个月，8个月回本
2. [高优先级] 分阶段迁移，降低一次性成本
3. [中优先级] 评估不迁移的隐性成本：招聘难、维护成本高

### 关键假设
- 新框架收益可量化
- 迁移风险可控

### 风险
- 现状偏差可能导致技术债务积累
- 需要区分合理谨慎和过度保守
```

## 示例 2：现状偏差——默认选项

**输入**：这个系统默认配置就行，用户一般不会改，不需要提供太多自定义选项。

```
Step 1: 识别现状信号 → 产出：发现"默认配置就行...用户一般不会改"表述，依赖默认选项的力量
Step 2: 评估默认影响 → 产出：默认选项被80%+用户接受，即使有更好选择
Step 3: 分析现状机制 → 产出：默认启发：默认=推荐；决策惰性：改变需要认知努力
Step 4: 校正现状偏差 → 产出：建议审慎设计默认选项，理解其强大影响力
```

**输出**：
```
## 分析结论
### 核心发现
1. 默认选项被80%+用户接受，即使有更好选择
2. 现状偏差使默认选项成为事实标准
3. 默认选项设计应承担道德责任：是否真正符合用户利益

### 建议（按优先级）
1. [高优先级] 默认选项应基于用户最佳利益设计
2. [高优先级] 提供明显的方式让用户更改默认
3. [中优先级] 定期评估默认选项是否仍最优

### 关键假设
- 默认选项影响力大
- 设计者有责任

### 风险
- 默认选项可能不符合部分用户需求
- 需要平衡便利性和选择权
```

## 反例：不该触发此 Skill 的场景

**输入**：迁移成本500万，新框架年收益50万，10年才能回本，不划算。

**原因**：这是基于成本效益分析的理性决策(10年回本不划算)，不是现状偏差。现状偏差指即使改变有明确收益仍维持现状。
