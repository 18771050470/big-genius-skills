# 分析示例

## 示例 1：产品功能迭代策略

**输入**：产品需要持续迭代，如何设计演进机制？

```
Step 1: 定义环境和选择压力
→ 环境：竞争激烈的SaaS市场，用户需求快速变化
→ 变化趋势：AI功能需求加速、移动端使用率上升、合规要求增加
→ 选择压力：用户留存率、NPS评分、功能使用率
→ 产出：环境分析

Step 2: 设计变异机制
→ 变异来源：用户反馈（40%）、竞品分析（20%）、技术预研（20%）、团队创意（20%）
→ 变异率：每2周产生3-5个新功能提案
→ 探索/利用平衡：70%资源优化现有功能，30%探索新功能
→ 产出：变异机制设计

Step 3: 建立选择机制
→ 选择标准：用户留存影响>1%、开发成本<2周、与产品愿景一致
→ 选择速度：每2周评审一次，淘汰率60%
→ 避免过早收敛：保留10%资源给高风险高回报实验
→ 产出：选择机制

Step 4: 确保保留和传递
→ 好功能通过A/B测试验证后全量发布
→ 知识传递：每次迭代记录决策日志和效果数据
→ 反馈循环：用户行为数据影响下一轮变异方向
→ 产出：保留和传递机制
```

**输出**：
```
## 分析结论

### 核心发现
1. 当前变异率不足（每月仅1-2个提案），需要提升到每2周3-5个
2. 选择标准模糊，导致低价值功能上线
3. 知识传递缺失，每次迭代从零开始

### 建议（按优先级）
1. [高] 建立双周功能提案评审机制，明确选择标准
2. [高] 设置30%资源用于探索性实验
3. [中] 建立迭代决策日志，累积知识
4. [低] 引入用户共创机制，增加变异来源
```
