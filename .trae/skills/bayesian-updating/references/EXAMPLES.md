# 分析示例

> 贝叶斯更新 的完整分析示例

## 示例 1：软件缺陷定位

**输入**：生产环境出现间歇性 500 错误，团队怀疑是数据库连接池泄漏，但不确定。历史数据显示，类似症状由连接池泄漏导致的先验概率约为 30%。现在观察到：错误只在高并发时出现，且重启后暂时恢复。

**分析过程**：

- **Step 1: 定义先验概率**
  - P(连接池泄漏) = 0.30（基于历史数据）
  - P(非连接池泄漏) = 0.70
  - 先验来源：过去 10 次类似症状中 3 次确诊为连接池泄漏

- **Step 2: 收集新证据**
  - 证据 E1：错误只在高并发时出现
  - 证据 E2：重启后暂时恢复
  - 证据来源：生产监控日志、运维报告
  - 证据质量：高（可重复观察）

- **Step 3: 评估似然**
  - P(E1|连接池泄漏) = 0.85（连接池耗尽通常在高并发时暴露）
  - P(E1|非连接池泄漏) = 0.40（其他原因也可能在高并发时出问题）
  - P(E2|连接池泄漏) = 0.90（重启释放连接，暂时恢复）
  - P(E2|非连接池泄漏) = 0.30（重启对非连接问题效果不确定）

- **Step 4: 计算后验概率**
  - 假设 E1 和 E2 条件独立：
  - P(E|H) = 0.85 × 0.90 = 0.765
  - P(E|¬H) = 0.40 × 0.30 = 0.12
  - 后验 = (0.30 × 0.765) / (0.30 × 0.765 + 0.70 × 0.12) = 0.2295 / 0.3135 ≈ 0.732
  - 更新后 P(连接池泄漏) ≈ 73.2%

- **Step 5: 评估更新幅度**
  - 先验 30% → 后验 73.2%，更新幅度合理
  - 两个证据都强烈指向连接池泄漏，更新幅度与证据强度匹配

- **Step 6: 迭代准备**
  - 下一步证据：检查连接池监控指标（活跃连接数、等待队列长度）
  - 若监控显示连接数持续增长 → 后验将进一步提升至 90%+
  - 若连接数正常 → 后验将下降，需重新评估假设

**输出**：
```
## 分析结论
### 核心发现
1. 先验概率（连接池泄漏）= 30%，基于历史数据
2. 观察到两个关键证据：高并发触发 + 重启暂时恢复
3. 后验概率更新为 73.2%，置信度显著提升

### 建议（按优先级）
1. [高优先级] 立即检查连接池监控：活跃连接数、等待队列、泄漏检测
2. [高优先级] 若监控支持，设置连接池自动扩容和泄漏回收
3. [中优先级] 准备连接池配置调整方案（增大 maxPoolSize、减小 connectionTimeout）

### 关键假设
- 两个证据条件独立（实际可能相关，但相关性不强）
- 历史先验 30% 对当前系统仍具代表性

### 风险
- 后验 73.2% 仍有 26.8% 概率是其他原因
- 若盲目调整连接池而问题在其他地方，会浪费时间和资源

### 下一步
建议调用 [causal-thinking] 深入分析连接池泄漏的根因机制
```

## 示例 2：技术选型决策

**输入**：团队在选择前端框架，候选方案是 React 和 Vue。初步倾向 React（团队有 2 年经验），但听说 Vue 3 的 Composition API 在复杂场景下更优。如何更新判断？

**分析过程**：

- **Step 1: 定义先验概率**
  - P(React 更适合) = 0.60（基于团队经验）
  - P(Vue 更适合) = 0.40
  - 先验来源：团队过去 2 年 React 项目成功率 90%，对 Vue 仅有调研经验

- **Step 2: 收集新证据**
  - 证据：Vue 3 Composition API 在复杂场景下代码复用率更高（来源：技术博客 benchmark）
  - 证据质量：中（benchmark 可能有偏差，且场景不完全匹配）

- **Step 3: 评估似然**
  - P(Composition API 更优 | React 更适合) = 0.30（React Hooks 也能实现类似复用）
  - P(Composition API 更优 | Vue 更适合) = 0.75（Vue 的设计确实在复用性上有优势）

- **Step 4: 计算后验概率**
  - 后验(Vue) = (0.40 × 0.75) / (0.40 × 0.75 + 0.60 × 0.30) = 0.30 / 0.48 = 0.625
  - 更新后 P(Vue 更适合) = 62.5%，P(React 更适合) = 37.5%

- **Step 5: 评估更新幅度**
  - 先验 Vue 40% → 后验 62.5%，更新幅度较大
  - 但证据质量仅为"中"，需警惕过度反应
  - 保留适度惯性：实际决策应结合更多证据

- **Step 6: 迭代准备**
  - 下一步证据：POC 验证（用两个框架各实现一个核心模块）
  - 团队学习成本评估
  - 长期维护性分析

**输出**：
```
## 分析结论
### 核心发现
1. 先验倾向 React（60%），但新证据使 Vue 后验概率升至 62.5%
2. 单一证据导致概率反转，需谨慎对待
3. 证据质量中等，不应过度反应

### 建议（按优先级）
1. [高优先级] 开展 POC：用 React Hooks 和 Vue Composition API 各实现一个复杂模块
2. [高优先级] 评估团队 Vue 学习成本（预计 2-4 周上手）
3. [中优先级] 咨询有 Vue 生产经验的团队，获取一手反馈
4. [中优先级] 若 POC 结果支持 Vue，制定渐进式迁移计划

### 关键假设
- 技术博客的 benchmark 场景与我们的项目匹配
- Composition API 的优势在我们项目中可兑现

### 风险
- 单一证据导致概率反转，可能过度反应
- 团队 Vue 经验不足，初期生产力可能下降 30%
- 若 Vue 优势被高估，迁移成本无法收回

### 下一步
建议调用 [opportunity-cost] 评估迁移成本与预期收益
```

## 反例：不该触发此 Skill 的场景

**场景**："这个按钮应该用什么颜色？"

**原因**：这是设计决策，不涉及概率更新或证据积累。虽然可以主观偏好"更新"，但缺乏可量化的先验和似然，贝叶斯更新在此不适用。应调用 [design-thinking] 或 [user-centered-design]。
