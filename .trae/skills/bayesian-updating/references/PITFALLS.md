# 常见陷阱

> 贝叶斯更新的常见误用

## 陷阱 1：先验陷阱——把偏见当数据

**问题**：用主观偏见或未经检验的假设作为先验概率，导致更新过程只是强化了原有偏见。

❌ **Before（错误做法）**
```markdown
## 分析结论
先验：这个方案一定不行，P(成功) = 0.05
证据：3 个团队已成功实施类似方案
更新后：P(成功) = 0.15
结论：还是不行。
```
*问题：先验 0.05 是主观偏见，不是数据。即使强证据也只能更新到 0.15，偏见未被修正。*

✅ **After（正确做法）**
```markdown
## 分析结论
先验：基于历史数据，类似方案成功率 P(成功) = 0.45
证据：3 个团队已成功实施，且条件匹配度 80%
似然：P(证据|成功) = 0.80，P(证据|失败) = 0.30
更新后：P(成功) = 0.69
结论：方案可行，建议开展 POC 验证。
```
*优势：先验基于历史数据，更新后概率合理反映证据强度。*

**原因**：Kahneman & Tversky (1974)：人们倾向于用锚定值（偏见）作为先验，然后不充分调整。贝叶斯更新的价值在于用数据修正先验，但如果先验本身就是偏见，更新只会强化错误。

## 陷阱 2：忽视备择假设

**问题**：只评估 P(证据|假设)，不评估 P(证据|非假设)，导致更新方向错误。

❌ **Before（错误做法）**
```markdown
## 分析结论
假设：数据库慢查询导致超时
证据：超时出现在高峰期
似然：P(超时|慢查询) = 0.90
更新：后验概率很高，确定是慢查询问题。
```
*问题：未评估 P(超时|非慢查询)，如网络拥堵、连接池耗尽等也可能导致超时。*

✅ **After（正确做法）**
```markdown
## 分析结论
假设：数据库慢查询导致超时
证据：超时出现在高峰期
似然：
- P(超时|慢查询) = 0.90
- P(超时|网络拥堵) = 0.70
- P(超时|连接池耗尽) = 0.85
- P(超时|其他) = 0.20

多假设贝叶斯更新：
- P(慢查询|超时) = 0.42
- P(网络拥堵|超时) = 0.28
- P(连接池耗尽|超时) = 0.25

结论：慢查询可能性最高，但其他原因概率合计 58%，需进一步排查。
```
*优势：同时考虑多个备择假设，避免过早锁定单一解释。*

**原因**：Gelman et al. (2013)：贝叶斯推断的核心是比较假设相对于备择假设的预测能力。忽略备择假设会导致确认偏误。

## 陷阱 3：过度更新——对噪声反应过度

**问题**：对质量低或样本小的证据赋予过高权重，导致概率剧烈波动。

❌ **Before（错误做法）**
```markdown
## 分析结论
先验：P(用户喜欢新功能) = 0.50
证据：1 个用户反馈不喜欢
似然：P(不喜欢|用户不喜欢) = 0.95
更新后：P(用户喜欢) = 0.05
结论：新功能失败，立即下线。
```
*问题：单个用户反馈是噪声，不应导致概率从 50% 暴跌到 5%。*

✅ **After（正确做法）**
```markdown
## 分析结论
先验：P(用户喜欢新功能) = 0.50
证据：1 个用户反馈不喜欢（样本量 n=1，噪声大）
似然：P(不喜欢|用户不喜欢) = 0.95，但证据质量低
调整：降低更新权重（相当于 n=1 的样本只提供少量信息）
更新后：P(用户喜欢) = 0.45
结论：证据不足，继续收集反馈，样本量达到 30 后再更新。
```
*优势：对低质量证据保持适度惯性，避免噪声驱动决策。*

**原因**：Silver (2012)：信号与噪声的区分是贝叶斯更新的核心挑战。小样本证据的信息量有限，不应导致大幅更新。

## 陷阱 4：混淆条件概率方向

**问题**：将 P(证据|假设) 与 P(假设|证据) 混淆，导致计算错误。

❌ **Before（错误做法）**
```markdown
## 分析结论
假设：程序员是优秀工程师
证据：优秀工程师通常写测试
观察：该程序员写测试
计算：P(优秀工程师|写测试) = P(写测试|优秀工程师) = 0.80
结论：该程序员 80% 概率是优秀工程师。
```
*问题：混淆了 P(写测试|优秀) 和 P(优秀|写测试)。基率（优秀工程师比例）被忽略。*

✅ **After（正确做法）**
```markdown
## 分析结论
假设：程序员是优秀工程师
证据：该程序员写测试
已知：
- P(写测试|优秀) = 0.80
- P(写测试|非优秀) = 0.30
- P(优秀) = 0.20（基率：团队中 20% 是优秀工程师）

贝叶斯计算：
P(优秀|写测试) = (0.20 × 0.80) / (0.20 × 0.80 + 0.80 × 0.30) = 0.40

结论：该程序员 40% 概率是优秀工程师，高于基率 20%，但远低于 80%。
```
*优势：正确使用贝叶斯公式，不混淆条件概率方向。*

**原因**：Kahneman (2011)：基率忽视是人类最常见的概率推理错误。P(证据|假设) 和 P(假设|证据) 方向相反，必须通过贝叶斯公式转换。
