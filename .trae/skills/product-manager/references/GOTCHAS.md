# Gotchas

> 只写 Agent 容易犯错的**非显而易见的事实**。通用知识不要写。
> 每个 Gotcha 必须包含：问题描述 + ❌ Before（错误做法） + ✅ After（正确做法） + 原因解释。

## Gotcha 1：把用户想要的当成用户需要的

**问题**：直接实现用户提出的解决方案，而没有深挖背后的真实需求，导致功能上线后使用率低下。

❌ **Before（错误做法）**
```markdown
## 需求：添加导出 Excel 功能
用户反馈："我需要把数据导出成 Excel"
产品方案：开发 Excel 导出功能
```

✅ **After（正确做法）**
```markdown
## 需求分析
用户反馈："我需要把数据导出成 Excel"
追问："导出后您要做什么？"
用户："我要把数据发给财务对账"
真实需求：数据共享与协作，而非 Excel 本身

## 产品方案
方案A：直接导出 Excel（用户说的）
方案B：添加"分享给财务"功能，自动生成对账报表（用户真正需要的）
方案C：开放只读权限给财务，实时查看数据（更优解）

推荐：先实施方案A（快速满足），同时调研方案B/C的长期价值
```

**原因**：用户提出的往往是解决方案而非问题本身。产品经理必须用"5 个 Why"追溯根源，找到更高效、更根本的解决方式。

## Gotcha 2：功能堆砌导致产品臃肿

**问题**：每个需求都接，每个功能都做，导致产品复杂度过高，新用户上手困难，核心功能被淹没。

❌ **Before（错误做法）**
```markdown
## 产品路线图 V1.0
- 用户系统（注册/登录/找回密码）
- 内容管理（发布/编辑/删除/分类/标签/搜索）
- 社交功能（关注/粉丝/私信/动态/通知）
- 电商功能（商品/购物车/订单/支付/物流）
- 数据分析（报表/图表/导出/API）
- 管理后台（权限/日志/配置/备份）

上线时间：3 个月
```

✅ **After（正确做法）**
```markdown
## MVP 范围（第 1 阶段，4 周）
目标：验证核心假设"用户愿意为 X 付费"
- 用户系统（仅手机号+验证码登录）
- 核心功能（最简路径完成核心价值）
- 支付功能（仅支持微信支付）

## V1.0（第 2 阶段，6 周）
目标：提升留存和复购
- 增加功能 A（数据支撑的高需求）
- 增加功能 B（用户反馈最多的痛点）

## 明确不做（至少前 3 个月）
- 社交功能（与核心目标无关）
- 复杂的数据分析（用户量不够，数据无意义）
- 管理后台（手动操作即可满足）
```

**原因**：资源永远有限，每个功能都有开发和维护成本。MVP 的核心是验证假设，而非功能完整。明确"不做清单"和"不做的时间边界"比"做清单"更重要。

## Gotcha 3：用内部视角代替用户视角写故事

**问题**：用户故事从技术实现或业务方便出发，而非从用户目标和场景出发，导致开发出的功能用户不会用。

❌ **Before（错误做法）**
```markdown
## 用户故事
作为系统，我想要记录用户的操作日志，以便管理员可以审计。

## 验收标准
- [ ] 数据库有 operation_log 表
- [ ] 每次操作插入一条记录
- [ ] 提供 API 查询日志
```

✅ **After（正确做法）**
```markdown
## 用户故事
作为安全管理员，我想要查看用户的敏感操作记录，以便发现异常行为并及时处理。

## 验收标准
- [ ] Given 用户进行了敏感操作（如修改权限、导出数据）
  When 我进入操作审计页面
  Then 我看到该操作的时间、用户、操作类型、影响范围
- [ ] Given 发现异常操作
  When 我点击"标记异常"
  Then 系统通知相关管理员，并记录处理状态
- [ ] Given 审计页面数据量大
  When 我设置筛选条件（时间范围/用户/操作类型）
  Then 页面显示符合条件的结果，支持分页
```

**原因**：好的用户故事必须从真实用户的视角出发，描述用户的目标和场景，而非系统的行为。验收标准必须是用户可验证的，而非技术实现细节。

## Gotcha 4：优先级排序只考虑价值不考虑成本

**问题**：只按用户价值或业务价值排序，忽略实现成本，导致高价值但高成本的功能阻塞了大量低 hanging fruit。

❌ **Before（错误做法）**
```markdown
## 优先级排序（仅按用户价值）
1. AI 智能推荐（价值：极高，成本：未评估）
2. 个性化首页（价值：高，成本：未评估）
3. 一键下单（价值：高，成本：未评估）
4. 修复支付 Bug（价值：中，成本：未评估）
```

✅ **After（正确做法）**
```markdown
## RICE 评分排序

| 功能 | Reach | Impact | Confidence | Effort | RICE 分数 | 优先级 |
|------|-------|--------|-----------|--------|----------|--------|
| 一键下单 | 1000 | 3 | 90% | 2 | 1350 | P0 |
| 修复支付 Bug | 500 | 3 | 100% | 1 | 1500 | P0 |
| 个性化首页 | 800 | 2 | 70% | 5 | 224 | P1 |
| AI 智能推荐 | 500 | 3 | 40% | 20 | 30 | P2 |

## 决策
- P0：立即做（高价值 + 低成本）
- P1：计划做（中等价值，需要资源）
- P2：调研做（价值不确定，先验证假设）
```

**原因**：RICE 框架（Reach × Impact × Confidence / Effort）强迫同时考虑价值和成本。高价值但高成本的功能可能不如中等价值但低成本的功能划算，后者可以更快交付并产生实际收益。

## Gotcha 5：忽视功能的生命周期成本

**问题**：只考虑功能的开发成本，忽略后续的维护、客服、文档、培训成本，导致总成本远超预期。

❌ **Before（错误做法）**
```markdown
## 功能评估
功能：自定义工作流引擎
开发成本：2 人月
决策：做！
```

✅ **After（正确做法）**
```markdown
## 全生命周期成本评估
功能：自定义工作流引擎

| 成本类型 | 估算 | 说明 |
|---------|------|------|
| 开发成本 | 2 人月 | 核心引擎开发 |
| 测试成本 | 0.5 人月 | 复杂场景测试 |
| 文档成本 | 0.3 人月 | 用户文档 + 开发文档 |
| 客服成本 | 持续 | 预计增加 20% 的工单量 |
| 维护成本 | 0.2 人月/季度 | Bug 修复、边缘场景处理 |
| 培训成本 | 0.2 人月 | 内部团队培训 |
| **首年总成本** | **~4 人月 + 持续投入** | |

## 替代方案
使用开源工作流引擎（如 Camunda），开发成本降至 0.5 人月，维护成本由社区承担。

## 决策：采用替代方案
```

**原因**：功能的总拥有成本（TCO）通常是开发成本的 2-5 倍。产品经理必须评估全生命周期成本，包括维护、支持、文档和机会成本，避免"开发完就完事"的短视决策。
