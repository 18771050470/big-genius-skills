# 极简思维 - 常见陷阱

## 陷阱 1：为了精简而精简

**问题**：把极简当审美目标，过度削减功能以致核心价值受损。

❌ **Before（错误做法）**
```
"这个搜索页面有很多功能：搜索框、高级筛选、排序、分页、结果高亮。
我们来精简一下：只保留搜索框，把筛选、排序、分页都删了。"
→ 结果：用户搜索"手机"出来5000条结果，无法筛选，无法排序，无法翻页，体验极差
```

✅ **After（正确做法）**
```
先界定"本质"：
- 搜索页面的本质是"帮用户快速找到他们想要的东西"
- 筛选、排序、分页是支撑这个本质的必要工具（当结果>20条时）
→ 不该删除
可以精简的是：
- "高亮"功能（90%用户不感知）→ 删除
- 高级筛选中的"按注册日期"（从未使用）→ 删除
- 排序选项从8个减少到"相关度+价格+销量"3个
```

**原因**：Dieter Rams的第十原则是"Good design is as little design as possible"，但前面九条原则包括"Good design is useful"和"Good design makes a product understandable"。极简的前提是"有用"——删除无用的东西让有用的更突出，而不是删除有用的东西本身。

## 陷阱 2：低估复杂度的隐藏成本

**问题**：认为一个"小功能"的成本只是写那几行代码，忽视了它引入的交叉复杂度。

❌ **Before（错误做法）**
```
"这个'导出CSV'功能就50行代码，加了吧。"
→ 加了之后：
  - 导出格式有人要CSV有人要Excel → 又加了Excel导出
  - 大数据量导出超时 → 加了异步导出+进度条
  - 导出的数据需要脱敏 → 加了权限判断
  - 功能本来"也就几行"，不知不觉变成了一个模块
```

✅ **After（正确做法）**
```
评估"简单"功能的真实成本：
- 直接成本：50行代码
- 交互成本：新增1个按钮、1个下载流程
- 维护成本：数据模型变了导出逻辑也得变
- 连锁成本：用户要Excel → 要异步 → 要脱敏
- 认知成本：新同事需要理解这50行的上下文
→ 如果不是核心需求，推迟到"有至少3个用户明确要求"时才做
```

**原因**：软件熵——每个新元素不仅增加了自己的成本，还增加了与所有已有元素的交叉组合复杂度。系统的复杂度是O(n²)的，不是O(n)的。

## 陷阱 3：忽视安全性和可访问性

**问题**：极简主义追求被滥用，削减了安全和可访问性相关的基础设施。

❌ **Before（错误做法）**
```
"我们的注册流程太复杂了：邮箱验证、手机验证、验证码、密码强度检测、同意条款...
简化一下：只要用户名+密码就行了。"
→ 结果：垃圾注册泛滥，用户密码全是123456
```

✅ **After（正确做法）**
```
安全性是"本质"的一部分，不是"多余"的：
- 这些验证不是为了"复杂"，是为了"安全"（本质需求）
- 可以精简的是"实现方式"：比如邮箱验证改为"注册后限制功能，验证后解锁全部"，而不是挡在注册前面
- 密码强度检测可以精简为"最少8位"，而不是复杂规则
→ 精简的是用户体验的摩擦，不是安全本身
```

**原因**：W3C的可访问性原则指出，"简单"和"可访问"不是矛盾的。极简设计应该让核心功能对所有人可用，包括残障用户。删除可访问性支持是降低质量，不是提升质量。
