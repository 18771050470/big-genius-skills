# Gotchas

> 只写网络安全容易犯错的**非显而易见的事实**。通用知识不要写。

## Gotcha 1：扫描生产环境导致服务中断

**问题**：渗透测试时对生产环境执行高强度扫描或漏洞利用，导致服务中断、数据损坏或业务损失。

❌ **Before（错误做法）**
```bash
# 直接对生产环境执行全端口扫描
nmap -sS -sV -O -p- production.example.com
# 大量 SYN 包可能导致防火墙/IPS 阻断正常流量

# 执行漏洞利用
msfconsole -x "use exploit/...; set RHOSTS production.example.com; exploit"
# 可能导致服务崩溃或数据损坏
```

✅ **After（正确做法）**
```bash
# 1. 明确测试环境
# 生产环境：仅被动侦察（whois、DNS 查询）
# 测试环境：主动扫描和漏洞利用

# 2. 与运维团队协调
# - 选择低峰时段（凌晨 2-5 点）
# - 提前通知，准备回滚方案
# - 设置监控告警，发现异常立即停止

# 3. 控制扫描强度
nmap -sS -sV --max-rate 100 -p 1-1000 staging.example.com
# 限制速率，避免被误认为 DDoS

# 4. 使用非破坏性测试
# - 漏洞验证：只验证存在性，不深入利用
# - SQL 注入：使用 AND 1=1 / AND 1=2，不读取数据
# - XSS：使用 alert(document.domain)，不执行恶意代码

# 5. 准备应急预案
# - 测试人员联系方式
# - 紧急停止暗号
# - 服务恢复步骤
```

**原因**：生产环境的可用性直接影响业务。即使是"安全"的扫描也可能触发 IPS、耗尽连接池、或导致意外故障。必须在隔离环境测试，生产环境仅做被动侦察。

## Gotcha 2：忽略内部威胁和横向移动

**问题**：安全测试只关注外部边界，忽略内网安全和横向移动，导致"外紧内松"。

❌ **Before（错误做法）**
```markdown
# 安全评估报告
- 外部边界：防火墙规则完善 ✅
- WAF：已部署，规则有效 ✅
- VPN：双因素认证 ✅
- 结论：网络安全良好

（未测试内网横向移动）
```

✅ **After（正确做法）**
```markdown
# 安全评估报告

## 外部边界
- 防火墙规则完善 ✅
- WAF 有效 ✅
- VPN 安全 ✅

## 内网安全（关键发现）

### 横向移动测试
- **场景**：假设一台办公电脑被入侵
- **测试**：
  1. 从办公网段扫描内网服务
  2. 发现数据库服务器（10.0.1.50）开放 3306 端口
  3. 使用弱口令 root/123456 成功登录
  4. 从数据库服务器扫描生产网段
  5. 发现生产服务器共享相同域凭据
  6. 成功登录生产服务器

### 结果
- 从办公网到生产网：仅需 3 步
- 内网无分段，所有服务器在同一 VLAN
- 数据库无独立管理网段
- 横向移动难度：极低

### 建议
1. 网络分段：办公网、测试网、生产网隔离
2. 数据库独立管理网段，禁止从办公网访问
3. 部署内网 IDS，检测横向移动
4. 实施零信任架构，内网也要认证
```

**原因**：边界安全只是第一层。一旦单点被突破，内网"平坦"的网络拓扑会让攻击者轻易横向移动。必须测试内网分段、横向移动难度、内部服务暴露面。

## Gotcha 3：未验证备份的可恢复性

**问题**：定期备份数据，但从未测试恢复流程，真正需要恢复时才发现备份损坏或恢复流程不可行。

❌ **Before（错误做法）**
```markdown
# 备份策略
- 数据库：每日全量备份
- 文件：增量备份每小时
- 保留期：30 天
- 结论：备份完善，数据安全

（从未执行过恢复测试）
```

✅ **After（正确做法）**
```markdown
# 备份与恢复验证

## 备份策略
- 数据库：每日全量 + 每小时增量
- 文件：增量备份每小时
- 保留期：30 天
- 异地备份：是

## 恢复测试（每季度执行）

### 测试 1：数据库恢复到测试环境
- **时间**：2024-01-15
- **备份文件**：backup_20240114.sql
- **恢复步骤**：
  1. 创建新数据库实例
  2. 下载备份文件（耗时：30 分钟）
  3. 执行恢复（耗时：2 小时）
  4. 验证数据完整性（耗时：1 小时）
- **结果**：✅ 成功，数据完整
- **RTO**：3.5 小时（目标 < 4 小时）
- **RPO**：1 小时（目标 < 2 小时）

### 测试 2：文件恢复
- **结果**：✅ 成功
- **问题**：发现部分小文件（< 1KB）备份时丢失
- **修复**：调整备份工具配置，排除文件大小过滤

### 测试 3：灾难恢复（模拟机房故障）
- **场景**：主机房完全不可用
- **步骤**：
  1. 在备用机房启动备用服务器
  2. 从异地备份恢复数据
  3. 切换 DNS 到备用机房
- **结果**：⚠️ 部分成功
- **问题**：备用服务器配置过期，需手动更新
- **改进**：自动化备用环境配置同步

## 结论
- 备份策略有效，但需定期验证
- 发现 2 个问题已修复
- 下次测试：2024-04-15
```

**原因**：备份不等于可恢复。备份文件可能损坏、加密密钥可能丢失、恢复流程可能需要数小时。只有定期执行恢复测试，才能确保在真正灾难时能快速恢复。

## Gotcha 4：忽略云配置安全

**问题**：关注应用层安全，忽略云基础设施配置（S3 公开、安全组开放、IAM 过度授权），导致数据泄露。

❌ **Before（错误做法）**
```markdown
# 安全评估
- 应用层：无 SQL 注入、XSS ✅
- 认证：MFA 已启用 ✅
- 加密：TLS 1.3 ✅
- 结论：安全良好
```

✅ **After（正确做法）**
```markdown
# 安全评估（含云配置）

## 应用层
- 无 SQL 注入、XSS ✅
- MFA 已启用 ✅

## 云配置（关键发现）

### S3 存储桶
| 存储桶 | 公开访问 | 加密 | 发现 |
|--------|---------|------|------|
| company-backup | ❌ 公开 | ❌ 无 | 🔴 大量敏感数据暴露 |
| company-logs | ❌ 公开 | ✅ 有 | 🟡 日志含用户隐私 |
| company-public | ✅ 公开（预期） | ✅ 有 | ✅ 正常 |

### 安全组
| 安全组 | 规则 | 发现 |
|--------|------|------|
| sg-web | 0.0.0.0/0:80,443 | ✅ 正常 |
| sg-db | 0.0.0.0/0:3306 | 🔴 数据库公开暴露 |
| sg-ssh | 0.0.0.0/0:22 | 🟡 建议限制 IP |

### IAM
| 用户/角色 | 权限 | 发现 |
|-----------|------|------|
| dev-user | AdministratorAccess | 🔴 开发账号有管理员权限 |
| ci-cd-role | S3:*, EC2:* | 🟡 权限过于宽泛 |

## 建议
1. 立即关闭 company-backup 公开访问
2. 数据库安全组限制为应用服务器 IP
3. 实施最小权限原则，拆分 IAM 角色
4. 使用 SCP 限制根账号操作
5. 启用 CloudTrail 记录所有 API 调用
```

**原因**：云配置错误是最常见的数据泄露原因（如 Capital One 泄露 1 亿用户数据）。必须使用 CSPM（云安全态势管理）工具持续扫描配置，修复公开访问、过度授权等问题。
