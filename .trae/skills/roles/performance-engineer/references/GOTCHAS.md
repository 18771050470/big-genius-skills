# Gotchas

> 只写性能测试容易犯错的**非显而易见的事实**。通用知识不要写。

## Gotcha 1：测试数据量不足导致性能假象

**问题**：性能测试使用少量数据（如 100 条），生产环境有数百万条，导致索引失效、查询性能急剧下降。

❌ **Before（错误做法）**
```python
# 测试环境只有 100 条数据
# 查询 SELECT * FROM orders WHERE user_id = 123
# 测试结果显示 10ms，但生产环境有 1000 万条数据
```

✅ **After（正确做法）**
```python
# 准备与生产环境相当的数据量
# 数据量应达到生产环境的 50% 以上

# 数据分布也要模拟生产环境
# 生产环境：20% 用户产生 80% 订单（长尾分布）
# 测试数据也应遵循相同分布

# 测试不同数据量下的性能
# 10 万条 → 100 万条 → 1000 万条
# 观察性能拐点
```

**原因**：数据库查询性能与数据量密切相关。小数据量下全表扫描也很快，但大数据量下必须依赖索引。测试数据量不足会严重低估生产环境的性能问题。

## Gotcha 2：只关注平均响应时间忽略长尾延迟

**问题**：性能测试只报告平均响应时间，忽略 P99/P99.9，导致用户体验差但测试报告"通过"。

❌ **Before（错误做法）**
```markdown
# 性能测试报告
- 平均响应时间：50ms ✅
- 结论：性能达标
```

✅ **After（正确做法）**
```markdown
# 性能测试报告
| 指标 | 数值 | 目标 | 状态 |
|------|------|------|------|
| P50 | 30ms | < 50ms | ✅ |
| P95 | 80ms | < 100ms | ✅ |
| P99 | 500ms | < 200ms | ❌ |
| P99.9 | 3000ms | < 500ms | ❌ |
| 错误率 | 0.1% | < 0.1% | ✅ |

## P99 延迟分析
- 500ms 的请求集中在 /api/reports 接口
- 该接口执行复杂聚合查询，无缓存
- 建议：添加 Redis 缓存，优化 SQL
```

**原因**：平均响应时间容易被低延迟请求拉低，掩盖少数用户的极差体验。P99/P99.9 更能反映真实用户体验，必须纳入性能指标。

## Gotcha 3：未预热直接测试导致结果失真

**问题**：系统冷启动后立即开始性能测试，JVM 未预热、连接池未建立、缓存未填充，导致测试结果远低于实际水平。

❌ **Before（错误做法）**
```bash
# 启动服务后立即开始测试
java -jar app.jar &
# 等待 5 秒
k6 run load-test.js  # 结果：吞吐量 100/s
```

✅ **After（正确做法）**
```bash
# 1. 启动服务
java -jar app.jar &

# 2. 预热阶段（5-10 分钟）
# - JVM JIT 编译热点代码
# - 数据库连接池建立
# - 缓存填充
# - 类加载完成
k6 run --duration 10m warmup.js  # 低负载预热

# 3. 正式测试
k6 run load-test.js  # 结果：吞吐量 500/s（更真实）
```

**原因**：JVM 需要预热才能达到最佳性能（JIT 编译、堆调整）。冷启动测试会严重低估系统性能，导致过度优化或不必要的扩容。

## Gotcha 4：忽略客户端性能指标

**问题**：性能测试只关注服务端指标（QPS、响应时间），忽略客户端体验指标（FCP、LCP、CLS、TTFB）。

❌ **Before（错误做法）**
```markdown
# 性能测试报告
- API QPS：1000/s ✅
- API P95：50ms ✅
- 结论：性能优秀
```

✅ **After（正确做法）**
```markdown
# 性能测试报告

## 服务端指标
- API QPS：1000/s ✅
- API P95：50ms ✅

## 客户端 Web Vitals
| 指标 | 数值 | 目标 | 状态 |
|------|------|------|------|
| TTFB | 200ms | < 600ms | ✅ |
| FCP | 1.2s | < 1.8s | ✅ |
| LCP | 3.5s | < 2.5s | ❌ |
| CLS | 0.15 | < 0.1 | ❌ |
| INP | 280ms | < 200ms | ❌ |

## 问题分析
- LCP 3.5s：首屏大图未优化（2.1MB，未压缩）
- CLS 0.15：图片无尺寸，加载时布局偏移
- INP 280ms：主线程阻塞，JavaScript 执行过长

## 优化建议
1. 图片使用 WebP 格式，压缩到 < 200KB
2. 图片添加 width/height 属性避免布局偏移
3. JavaScript 拆包，非关键 JS 延迟加载
```

**原因**：用户感知的是端到端性能，包括网络传输、浏览器渲染、JavaScript 执行。服务端性能再好，客户端体验差用户仍会流失。
