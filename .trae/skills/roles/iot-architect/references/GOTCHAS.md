# Gotchas

> 只写 Agent 容易犯错的**非显而易见的事实**。通用知识不要写。
> 每个 Gotcha 必须包含：问题描述 + ❌ Before（错误做法） + ✅ After（正确做法） + 原因解释。

## Gotcha 1：所有设备使用相同密钥

**问题**：为简化部署，所有设备烧录相同的密钥或证书，一台设备被攻破后全部沦陷。

❌ **Before（错误做法）**
```markdown
# 设备认证
- 所有设备使用相同密钥："shared-secret-123"
- 云端验证：if (device_key == "shared-secret-123") allow
- 结果：攻击者提取一台设备的密钥，即可冒充所有设备
```

✅ **After（正确做法）**
```markdown
# 设备认证
- 每台设备出厂烧录唯一证书（X.509）
- 私钥存储在硬件安全模块（HSM）或安全元件中
- 云端验证：验证设备证书签名链 + 检查证书吊销列表
- 结果：单台设备被攻破不影响其他设备

# 证书管理
- 根CA证书：云端持有，用于签发设备证书
- 设备证书：每台设备唯一，包含设备ID和公钥
- 证书有效期：1-2年，到期前自动续期
```

**原因**：共享密钥是最严重的IoT安全漏洞。一旦密钥泄露，攻击者可控制所有设备，造成大规模安全事件。

## Gotcha 2：MQTT Topic设计不当导致消息混乱

**问题**：所有设备订阅相同的Topic，导致设备收到不属于自己的消息，或消息被错误处理。

❌ **Before（错误做法）**
```markdown
# MQTT Topic设计
- 所有设备订阅："sensor/data"
- 所有设备发布："sensor/data"
- 结果：设备A的数据被设备B收到，数据混乱
```

✅ **After（正确做法）**
```markdown
# MQTT Topic设计（分层结构）
- 设备发布："company/product/{device-id}/telemetry"
- 设备订阅："company/product/{device-id}/commands"
- 云端订阅："company/product/+/telemetry"（通配符接收所有设备数据）
- 结果：每台设备只收到自己的命令，数据不混乱

# Topic设计原则
- 包含设备ID：确保消息定向
- 分层结构：便于权限控制和通配符订阅
- 区分上行/下行：telemetry（上行）vs commands（下行）
```

**原因**：MQTT的发布/订阅模式需要合理的Topic设计来隔离设备数据。错误的Topic设计会导致数据混乱和安全风险。

## Gotcha 3：OTA升级无回滚机制

**问题**：OTA升级失败（如断电、网络中断、固件损坏），设备变砖，需要现场维修。

❌ **Before（错误做法）**
```markdown
# OTA流程
1. 下载新固件
2. 擦除旧固件
3. 写入新固件
4. 重启

# 问题
- 步骤3中断（断电）→ 固件损坏
- 设备无法启动，需要现场刷机
```

✅ **After（正确做法）**
```markdown
# OTA流程（双分区设计）
1. 下载新固件到备用分区
2. 验证新固件签名和完整性
3. 标记备用分区为启动分区
4. 重启
5. 启动成功后，标记为稳定版本
6. 启动失败（3次），回滚到原分区

# 回滚机制
- 双分区：A区（当前）+ B区（备用）
- 启动计数：新固件启动3次成功后确认
- 回滚触发：启动失败或验证失败
- 云端监控：升级进度、成功率、失败原因
```

**原因**：OTA升级过程中任何中断都可能导致固件损坏。双分区设计确保始终有一个可用固件，失败时自动回滚。

## Gotcha 4：忽略离线场景

**问题**：设计时假设设备始终在线，忽略网络中断场景，导致数据丢失或设备失控。

❌ **Before（错误做法）**
```markdown
# 数据上报
- 设备实时上报数据到云端
- 网络中断时数据丢弃
- 结果：网络恢复后数据缺失，无法完整分析
```

✅ **After（正确做法）**
```markdown
# 数据上报（支持离线）
- 本地缓存：网络中断时数据存储在本地Flash/SD卡
- 批量上报：网络恢复后按时间顺序批量上报
- 数据去重：云端根据时间戳去重
- 优先级：告警数据优先上报，普通数据可延迟

# 离线策略
- 缓存容量：根据数据量和网络稳定性计算
- 缓存策略：FIFO，满时覆盖最旧数据
- 同步机制：网络恢复后自动同步，带进度反馈
```

**原因**：IoT设备常部署在网络不稳定环境（如地下室、偏远地区）。必须设计离线缓存和同步机制，确保数据不丢失。

## Gotcha 5：数据上报无时间戳

**问题**：设备上报数据时不带时间戳，云端用接收时间作为数据时间，导致时序分析错误。

❌ **Before（错误做法）**
```markdown
# 数据格式
{
  "temperature": 25.3,
  "humidity": 60
}

# 云端处理
- 用接收时间作为数据时间
- 问题：网络延迟导致时间偏差，时序分析错误
```

✅ **After（正确做法）**
```markdown
# 数据格式（带时间戳和设备ID）
{
  "device_id": "dev-001",
  "timestamp": "2026-05-04T12:00:00Z",
  "temperature": 25.3,
  "humidity": 60,
  "seq": 12345
}

# 云端处理
- 使用设备时间戳（优先）或接收时间（备用）
- 时间同步：设备定期与NTP服务器同步
- 时序分析：基于设备时间戳，确保准确性
```

**原因**：网络延迟和批量上报会导致接收时间与实际采集时间不一致。设备时间戳是时序分析的基础，必须包含在数据中。
