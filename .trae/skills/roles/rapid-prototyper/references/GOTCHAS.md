# 快速原型师常见陷阱

## 1. 范围蔓延

### 反例：原型过于复杂
- **错误**：原型包含过多功能，失去了"快速验证"的意义
- **案例**：MVP 包含了 20 个功能，开发周期 2 个月，失去了快速验证的优势
- **原因**：没有坚持"最小化"原则

### 正例：聚焦核心
- **正确**：只构建验证核心假设所需的最小功能
- **案例**：验证"用户是否愿意为 X 付费"，只构建付费流程，其他功能 mock
- **要点**：原型不是产品，是验证工具

## 2. 完美主义

### 反例：追求 polished 原型
- **错误**：花费大量时间打磨 UI 和交互，延误验证时机
- **案例**：原型开发 3 周，其中 2 周在调整 UI 细节
- **原因**：混淆了原型和最终产品的标准

### 正例：快速粗糙
- **正确**：原型只要能验证假设即可，UI 可以粗糙
- **案例**：使用线框图风格，功能可用即可，1 周完成验证
- **要点**：速度比完美更重要

## 3. 验证目标不清

### 反例：模糊的目标
- **错误**：原型没有明确的验证目标，收集了一堆无用的反馈
- **案例**：原型上线后问用户"你觉得怎么样"，得到的反馈无法指导决策
- **原因**：没有定义清晰的假设和成功标准

### 正例：明确假设
- **正确**：每个原型验证一个具体假设，定义明确的通过/失败标准
- **案例**："假设：用户愿意花 5 分钟完成注册。标准：注册完成率 > 30%"
- **要点**：没有假设的验证是浪费时间

## 4. 忽视技术债务

### 反例：原型直接当产品
- **错误**：将验证用的原型直接作为生产代码，导致技术债务
- **案例**：MVP 使用临时架构，用户增长后系统无法扩展
- **原因**：没有规划从原型到产品的过渡

### 正例：明确边界
- **正确**：原型代码明确标记为"可丢弃"，验证通过后重新设计架构
- **案例**：MVP 使用 Firebase 快速搭建，验证通过后迁移到自研后端
- **要点**：原型是学习的工具，不是产品的起点

## 5. 用户选择偏差

### 反例：错误的用户群体
- **错误**：招募的用户不是目标用户，验证结果无效
- **案例**：验证 B2B 产品，却招募了 C 端用户测试
- **原因**：用户招募标准不清晰

### 正例：精准招募
- **正确**：严格按照目标用户画像招募测试用户
- **案例**：验证企业协作工具，招募有团队协作需求的职场人士
- **要点**：错误的用户群体得出错误的结论
