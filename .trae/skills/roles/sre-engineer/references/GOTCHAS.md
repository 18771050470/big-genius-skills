# Gotchas

> 只写 SRE 容易犯错的**非显而易见的事实**。通用知识不要写。

## Gotcha 1：SLO 目标定得太高导致无法发布

**问题**：SLO 目标设定为 99.999%（每年 5 分钟停机），导致错误预算极少，任何发布都可能耗尽预算。

❌ **Before（错误做法）**
```markdown
# SLO 定义
- 可用性：99.999%
- 错误预算：每年 5 分钟

# 结果
- 每次发布都战战兢兢
- 错误预算 1 月份就耗尽
- 团队不敢发布新功能
- 实际可用性 99.99%，但目标太高达不到
```

✅ **After（正确做法）**
```markdown
# SLO 定义（基于历史数据）
- 过去 30 天实际可用性：99.95%
- 设定 SLO：99.95%（每月 21 分钟错误预算）
- 目标：逐步提升到 99.99%

# 错误预算管理
- 每月初重置预算
- 预算消耗 > 50% 时警告
- 预算耗尽时停止非紧急发布
- 连续 3 个月未耗尽预算 → 收紧 SLO 到 99.98%
```

**原因**：SLO 不是越高越好。过高的 SLO 会束缚团队创新，且成本急剧上升（99.9% → 99.99% 成本可能翻倍）。应基于历史数据设定，逐步提升。

## Gotcha 2：SLI 选择系统内部指标而非用户感知

**问题**：SLI 选择 CPU 使用率、内存占用等系统指标，而非用户实际体验到的可用性和延迟。

❌ **Before（错误做法）**
```markdown
# SLI 定义
- CPU 使用率 < 80%
- 内存使用率 < 90%
- 磁盘空间 > 20%

# 问题
- CPU 60% 但用户请求超时 → SLO 显示正常
- 内存 50% 但页面加载 10 秒 → SLO 显示正常
- 系统指标好 ≠ 用户体验好
```

✅ **After（正确做法）**
```markdown
# SLI 定义（用户感知）
- 可用性：HTTP 200 响应比例（用户视角：服务是否可用）
- 延迟：P99 响应时间 < 500ms（用户视角：页面加载快不快）
- 错误率：5xx 比例 < 0.1%（用户视角：操作是否成功）
- 吞吐量：每秒处理订单数（用户视角：高峰期是否卡顿）

# 系统指标用于告警，但不是 SLI
- CPU > 80% → 告警运维，但不计入 SLO
```

**原因**：SLI 必须反映用户真实体验。用户不在乎 CPU 使用率，在乎页面是否加载、操作是否成功。系统指标用于诊断，不应作为 SLO。

## Gotcha 3：错误预算只监控不行动

**问题**：建立了错误预算仪表盘，但预算耗尽时没有任何行动，团队继续发布，导致可靠性持续下降。

❌ **Before（错误做法）**
```markdown
# 错误预算仪表盘
- 本月预算：43 分钟
- 已消耗：60 分钟
- 状态：❌ 已超支

# 实际情况
- 产品经理："这个功能必须这周上线"
- 开发："那就发布吧"
- SRE："预算已经超了... 但好吧"

# 结果：预算机制形同虚设
```

✅ **After（正确做法）**
```markdown
# 错误预算政策（书面化，三方签字）

## 预算状态与行动
| 状态 | 条件 | 行动 |
|------|------|------|
| 🟢 健康 | 消耗 < 50% | 正常发布 |
| 🟡 警告 | 消耗 50-100% | 发布需 SRE 审批 |
| 🔴 耗尽 | 消耗 > 100% | 停止非紧急发布，优先修复 |

## 例外流程
- 紧急安全补丁：随时可发布，事后补预算分析
- 产品 deadline 冲突：由 CTO 决策，记录决策日志

## 执行
- 预算耗尽时，CI/CD 自动阻止非紧急发布
- 每周例会 review 预算消耗趋势
```

**原因**：错误预算的价值在于指导行动。没有行动机制的预算只是数字。必须将预算状态与发布决策挂钩，并得到团队认可。

## Gotcha 4：Postmortem 变成"批斗会"

**问题**：事后复盘聚焦"谁犯了错"，导致团队成员隐瞒问题、互相指责，无法真正改进。

❌ **Before（错误做法）**
```markdown
# Postmortem 会议
- "为什么这个配置会被推上生产？"
- "谁批准的这次发布？"
- "为什么测试没发现这个问题？"
- 气氛紧张，大家互相推诿
- 结论："下次小心点"

# 结果
- 同样的问题再次发生
- 团队氛围变差
- 问题被隐瞒，不敢上报
```

✅ **After（正确做法）**
```markdown
# Blameless Postmortem

## 原则
- 假设每个人都做了当时最合理的决定
- 聚焦"系统为什么允许这个错误发生"
- 人是系统的组成部分，系统设计要考虑人的因素

## 会议结构
1. **时间线回顾**：精确到分钟，只陈述事实
2. **影响评估**：用户影响、业务损失
3. **根因分析**：5 Whys，找到系统层面的原因
4. **改进措施**：Action Items，每个有负责人和截止日期
5. **经验分享**：文档全公司可见

## 示例结论
- 不是："小王没检查配置"
- 而是："配置检查没有自动化，人工检查容易遗漏"
- 改进："CI 中添加配置校验步骤"
```

**原因**：人是系统的组成部分。如果系统设计依赖"人不会犯错"，那系统本身就有缺陷。Blameless Postmortem 才能找到真正的系统改进点。
