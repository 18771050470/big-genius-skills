# Gotchas

> 只写代码审查容易犯错的**非显而易见的事实**。通用知识不要写。

## Gotcha 1：只关注代码风格忽略架构问题

**问题**：审查者花费大量时间纠结缩进、命名风格，忽略了耦合度过高、职责不清等架构级问题。

❌ **Before（错误做法）**
```markdown
# 代码审查反馈
- 第 15 行：变量名 `data` 不够清晰，建议改为 `userData`
- 第 23 行：缺少空格，应为 `if (condition) {`
- 第 45 行：注释拼写错误

（完全没提这个 500 行的类做了 5 件不同的事）
```

✅ **After（正确做法）**
```markdown
# 代码审查反馈

## 🔴 架构问题（必须修复）
- **UserManager 类（500 行）违反单一职责原则**
  - 当前职责：用户 CRUD + 订单管理 + 邮件发送 + 日志记录 + 权限校验
  - 建议拆分为：UserService、OrderService、EmailService、AuditService
  - 参考：SOLID 原则中的 Single Responsibility Principle

## 🟡 代码质量（建议修复）
- 第 15 行：变量名 `data` 建议改为 `userData`
- 第 45 行：复杂逻辑缺少注释，建议说明业务规则

## 💭 风格问题（可选）
- 第 23 行：缩进风格与团队规范不一致
```

**原因**：代码风格问题可以通过 linter 自动解决，审查者应聚焦机器无法发现的问题：架构设计、业务逻辑、安全漏洞、性能隐患。

## Gotcha 2：审查意见只有批评没有肯定

**问题**：审查反馈全是问题和建议，没有对好的实践给予肯定，导致作者抵触情绪。

❌ **Before（错误做法）**
```markdown
# 代码审查反馈
- 第 10 行：未处理异常
- 第 25 行：魔法数字
- 第 40 行：重复代码
- 第 55 行：命名不清
```

✅ **After（正确做法）**
```markdown
# 代码审查反馈

## 👍 做得好的地方
- 使用策略模式处理不同支付方式，扩展性很好
- 单元测试覆盖了边界条件，包括空值和超长输入
- 错误处理很完善，每个异常都有明确的错误码

## 🟡 建议改进
- 第 10 行：建议添加日志记录异常信息，便于排查
- 第 25 行：魔法数字 86400 建议提取为常量 `SECONDS_PER_DAY`
- 第 40 行：这段代码与 UserService 第 120 行重复，建议提取公共方法
```

**原因**：研究表明正负反馈比例 1:1 时作者接受度最高。肯定好的实践能强化团队规范，也能让作者更愿意接受改进建议。

## Gotcha 3：在 PR 中讨论架构大改

**问题**：在代码审查阶段提出"应该重写整个模块"的建议，导致 PR 无法合并，讨论低效。

❌ **Before（错误做法）**
```markdown
# 审查反馈
这个模块的设计有问题，应该用事件驱动架构重写。
建议：
1. 引入 Kafka
2. 拆分微服务
3. 使用 CQRS 模式
```

✅ **After（正确做法）**
```markdown
# 审查反馈

## 当前 PR
- 代码逻辑正确，测试通过 ✅
- 一个小建议：第 45 行异常处理可以优化

## 后续改进（不在当前 PR 处理）
我注意到这个模块的耦合度在增加，建议后续考虑：
- 选项 A：提取订单处理为独立服务（适合高并发场景）
- 选项 B：引入消息队列解耦（适合异步处理需求）

建议在下次技术分享或架构评审中讨论，不影响当前 PR 合并。
```

**原因**：PR 审查应聚焦当前变更，大架构调整应通过专门的技术讨论或架构评审进行。在 PR 中提出大改会让作者陷入两难：接受则延期数周，不接受则无法合并。

## Gotcha 4：忽略上下文只审查 diff

**问题**：只查看 diff 中的变更行，不查看变更的上下文（ surrounding code），导致遗漏问题。

❌ **Before（错误做法）**
```markdown
# 只审查 diff 中的新增行
+ def process_payment(order_id, amount):
+     user = get_user_by_order(order_id)
+     charge(user.card_id, amount)
+     update_order_status(order_id, 'paid')
```

✅ **After（正确做法）**
```markdown
# 审查 diff 时查看完整上下文
# 查看 surrounding code 后发现：

def process_payment(order_id, amount):
    user = get_user_by_order(order_id)  # 新增
    charge(user.card_id, amount)         # 新增
    update_order_status(order_id, 'paid') # 新增
    # 问题：没有事务！
    # 如果 charge 成功但 update_order_status 失败，数据不一致
    # 建议：使用数据库事务包裹
    
    # 另外：没有验证 amount 是否为正数
    # 建议：添加参数校验
```

**原因**：代码审查不是孤立地看变更行，而是要理解变更在整体中的影响。只看 diff 会遗漏事务、并发、异常处理等上下文相关的问题。
