# Gotchas

> 只写现实检验容易犯错的**非显而易见的事实**。通用知识不要写。

## Gotcha 1：确认偏误导致片面结论

**问题**：只寻找支持自己观点的证据，主动忽略或贬低反面证据，导致结论片面。

❌ **Before（错误做法）**
```markdown
# 方案评估：采用微服务架构

## 支持证据
-  Netflix 使用微服务，扩展性很好
-  团队可以独立部署，提高效率
-  技术栈可以异构，选择更灵活

## 结论
微服务架构是最佳选择。
```

✅ **After（正确做法）**
```markdown
# 方案评估：采用微服务架构

## 支持证据
-  Netflix 使用微服务，扩展性很好
-  团队可以独立部署，提高效率
-  技术栈可以异构，选择更灵活

## 反对证据
-  团队仅 5 人，拆分服务后运维复杂度激增
-  当前单体应用 QPS < 100，无扩展瓶颈
-  团队无 Kubernetes/Docker 经验，学习成本高
-  微服务调试困难，本地开发体验差

## 平衡分析
| 维度 | 单体 | 微服务 |
|------|------|--------|
| 扩展性 | 满足当前需求 | 过度设计 |
| 运维成本 | 低 | 高（需专人） |
| 开发效率 | 高 | 低（调试困难） |
| 团队匹配 | 高 | 低（无经验） |

## 结论
当前阶段单体架构更合适。当团队 > 15 人、QPS > 1000 时再考虑微服务。
```

**原因**：确认偏误是人类本能，但会导致严重决策失误。现实检验必须强制寻找反面证据，并诚实呈现。

## Gotcha 2：锚定效应导致估计偏差

**问题**：第一个获得的信息（如初始预算、他人估计）过度影响后续判断，导致估计严重偏差。

❌ **Before（错误做法）**
```markdown
# 工期估计
- 产品经理说：这个需求很简单，2 周够了吧？
- 开发估计：那就 2 周吧

（实际开发中发现需要 6 周）
```

✅ **After（正确做法）**
```markdown
# 工期估计（使用锚定防护）

## 第一步：独立估计（不看他人估计）
- 功能拆解：10 个子任务
- 每个任务估计：1-3 天
- 独立估算总计：4-6 周

## 第二步：了解锚点（产品经理的 2 周）
- 产品经理的依据："看起来和上次那个需求差不多"
- 实际上次需求：无权限管理、无审批流、无通知

## 第三步：调整估计
- 基础功能：2 周
- 权限管理：1 周
- 审批流：1.5 周
- 通知系统：1 周
- 联调测试：1 周
- **总计：6.5 周（含缓冲）**

## 结论
2 周的锚点忽略了权限和审批的复杂度。建议按 6 周计划，分 3 个迭代交付。
```

**原因**：锚定效应在估计中极其常见。先做独立估计，再了解他人锚点，最后调整，能显著减少偏差。

## Gotcha 3：幸存者偏差导致错误归因

**问题**：只分析成功案例，忽略失败案例，导致错误的因果归因。

❌ **Before（错误做法）**
```markdown
# 分析：为什么成功创业公司都用 Ruby on Rails？
- Twitter 用 Rails 成功了
- GitHub 用 Rails 成功了
- Shopify 用 Rails 成功了
- 结论：Rails 是创业成功的关键因素
```

✅ **After（正确做法）**
```markdown
# 分析：技术栈与创业成功的关系

## 成功案例（可见的）
- Twitter：Rails → 成功
- GitHub：Rails → 成功
- Shopify：Rails → 成功

## 失败案例（不可见的）
- 使用 Rails 失败的创业公司：[数量未知，但大量存在]
- 使用 Java/PHP/Python 成功的公司：Google/Facebook/Instagram

## 正确归因
- 成功因素：市场需求、团队执行力、时机、融资
- 技术栈：影响开发效率，但不是成功/失败的决定因素
- Rails 在 2005-2010 年确实有生产力优势，但非决定性

## 结论
选择团队熟悉的技术栈，专注解决用户问题，而非追逐"成功公司的技术"。
```

**原因**：我们只看到成功案例，因为失败者不会发声。分析时必须考虑"沉默的数据"，避免错误归因。

## Gotcha 4：过度自信导致风险低估

**问题**：对自己的判断过于自信，低估不确定性和风险，导致计划外的问题。

❌ **Before（错误做法）**
```markdown
# 风险评估
- 数据库迁移风险：低（我们团队经验丰富）
- 第三方 API 变更风险：低（他们已经稳定运行 2 年）
- 性能风险：低（代码已经优化过）
- 结论：按计划执行，无重大风险
```

✅ **After（正确做法）**
```markdown
# 风险评估（使用前置概率）

## 数据库迁移
- 乐观估计（10%）：1 天完成
- 最可能（60%）：3 天，需回滚一次
- 悲观估计（30%）：1 周，数据丢失需恢复
- **预期工期**：0.1×1 + 0.6×3 + 0.3×7 = 4 天
- **应急预案**：备份策略、回滚脚本、数据校验

## 第三方 API 变更
- 历史数据：过去 2 年变更 3 次，平均影响 2 天
- 概率：30% 在接下来 3 个月内变更
- **应对**：封装适配层，监控变更公告

## 性能风险
- 测试环境数据量：1 万条
- 生产环境数据量：1000 万条
- **风险**：索引策略可能在大数据量下失效
- **应对**：使用生产数据量的 50% 进行压测

## 结论
存在多个中等风险，建议预留 30% 缓冲时间，并制定应急预案。
```

**原因**：过度自信是认知偏差中最危险的一种。使用概率估计、历史数据、最坏情况分析，能更客观地评估风险。
