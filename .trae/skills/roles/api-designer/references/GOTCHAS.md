# API 设计师常见陷阱

## 1. URI 设计不规范

### 反例：使用动词
- **错误**：URI 中使用动词，如 `/getUsers`、`/createOrder`
- **案例**：`/api/v1/getUserInfo?id=123` 不符合 RESTful 规范
- **原因**：RESTful 使用 HTTP 方法表示动作，URI 只表示资源

### 正例：使用名词
- **正确**：URI 使用名词复数，动作由 HTTP 方法表示
- **案例**：`GET /users/123` 获取用户，`POST /orders` 创建订单
- **要点**：URI 是资源的地址，不是操作的描述

## 2. 状态码滥用

### 反例：全部返回 200
- **错误**：无论成功失败都返回 HTTP 200，错误信息放在 body 中
- **案例**：用户未认证也返回 200，body 中 `{"code": 401}`
- **原因**：混淆了 HTTP 状态码和业务状态码

### 正例：正确使用状态码
- **正确**：使用恰当的 HTTP 状态码：2xx（成功）、4xx（客户端错误）、5xx（服务端错误）
- **案例**：参数错误返回 400，未认证返回 401，未授权返回 403，不存在返回 404
- **要点**：HTTP 状态码表示请求的处理结果，业务状态码表示业务逻辑结果

## 3. 忽视版本管理

### 反例：无版本控制
- **错误**：API 没有版本号，变更时直接修改现有接口
- **案例**：`/api/users` 直接修改字段，导致旧客户端崩溃
- **原因**：没有考虑 API 的兼容性和演进

### 正例：版本化管理
- **正确**：API 包含版本号，变更时发布新版本
- **案例**：`/api/v1/users`，v2 版本新增字段但不删除旧字段
- **要点**：API 是契约，变更需要兼容旧版本

## 4. 响应格式不一致

### 反例：格式混乱
- **错误**：不同接口返回不同的响应格式
- **案例**：接口 A 返回 `{"result": {...}}`，接口 B 返回 `{"data": {...}}`
- **原因**：没有统一的响应规范

### 正例：统一格式
- **正确**：全系统使用统一的响应格式
- **案例**：所有接口返回 `{"code": 0, "message": "success", "data": {...}}`
- **要点**：一致性降低调用方的理解成本

## 5. 忽视安全性

### 反例：明文传输
- **错误**：敏感接口使用 HTTP，参数明文传输
- **案例**：登录接口使用 HTTP，密码明文传输
- **原因**：忽视了 API 的安全风险

### 正例：安全设计
- **正确**：使用 HTTPS，敏感操作需认证，参数有校验
- **案例**：所有接口 HTTPS，登录需 OAuth/JWT，参数有类型和长度校验
- **要点**：安全是 API 设计的基础要求
