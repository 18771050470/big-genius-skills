# 分析示例

## 示例 1：信念坚持——负面反馈处理

**输入**：用户调研显示这个功能没人用，但我觉得他们没理解这个功能的价值，我们应该继续优化而不是砍掉。

```
Step 1: 识别信念坚持信号 → 产出：发现"他们没理解"表述，负面证据被折扣
Step 2: 评估证据权重 → 产出：用户调研是负面证据，"没理解"是信念坚持的合理化
Step 3: 分析坚持机制 → 产出：自我辩护：投入精力后不愿放弃；确认偏误：寻找支持信念的证据
Step 4: 校正信念坚持 → 产出：建议客观评估使用数据，设定继续/终止标准
```

**输出**：
```
## 分析结论
### 核心发现
1. 用户调研显示功能使用率<1%，这是强负面证据
2. "用户没理解"是信念坚持的合理化，非实证依据
3. 功能开发已投入3人月，沉没成本可能影响判断

### 建议（按优先级）
1. [高优先级] 查看实际使用数据（DAU、使用时长、留存）
2. [高优先级] 设定明确标准：若使用率<5%则砍掉
3. [中优先级] 如保留，需重新设计而非优化

### 关键假设
- 用户调研样本具有代表性
- 使用数据可准确追踪

### 风险
- 继续投入可能增加沉没成本
- 砍掉功能可能影响部分核心用户
```

## 示例 2：信念坚持——技术方案固执

**输入**：虽然压测显示这个架构扛不住10万QPS，但我认为压测场景不真实，实际不会有问题。

```
Step 1: 识别信念坚持信号 → 产出：发现"压测场景不真实"表述，负面证据被合理化
Step 2: 评估证据权重 → 产出：压测数据是客观证据，"场景不真实"是信念坚持
Step 3: 分析坚持机制 → 产出：设计者偏见：自己设计的方案不愿承认缺陷；过度自信：高估方案能力
Step 4: 校正信念坚持 → 产出：建议第三方复现压测，用数据说话
```

**输出**：
```
## 分析结论
### 核心发现
1. 压测显示架构在8万QPS时延迟飙升，这是客观数据
2. "场景不真实"需验证：实际流量模式与压测差异多少
3. 方案设计师可能有设计者偏见，不愿承认缺陷

### 建议（按优先级）
1. [高优先级] 第三方团队复现压测，消除设计者偏见
2. [高优先级] 对比压测场景与实际流量模式差异
3. [中优先级] 准备降级方案，应对高流量场景

### 关键假设
- 压测环境配置正确
- 第三方团队无利益冲突

### 风险
- 忽视压测结果可能导致线上故障
- 过度依赖压测可能忽视其他风险
```

## 反例：不该触发此 Skill 的场景

**输入**：用户说这个功能不好用，但数据显示使用率80%，可能是调研样本有偏差。

**原因**：这是基于数据质疑调研结论，有客观数据支撑（使用率80%），不是信念坚持。
