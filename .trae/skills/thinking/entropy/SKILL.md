# 熵增思维（Entropy Thinking）

## 核心定义

孤立系统必然趋向混乱，秩序需要持续投入能量来维持——将热力学第二定律映射到软件系统：代码会自然腐化，团队会自然松散，必须通过持续的工程投入（重构、文档、培训）来对抗熵增。

---

## 思考流程控制

### 中断处理

- **信息不足**：用 AskUserQuestion 向用户确认系统的"开放程度"（是否持续有外部输入/变更）
- **发现矛盾**：停止思考，用 AskUserQuestion 向用户报告矛盾点（如"要求零技术债务但要求快速交付"），等待决策
- **提前终止**：若系统为一次性脚本且不再维护，可提前终止并建议不投入抗熵资源

---

## 思考流程

### Step 1: 评估系统熵值

- 代码层面：圈复杂度、重复率、测试覆盖率、文档完整度、技术债务清单
- 架构层面：模块耦合度、接口稳定性、依赖关系清晰度、架构漂移程度
- 团队层面：知识分布（ bus factor ）、沟通成本、流程规范遵守度、人员流动率
- 基础设施：监控覆盖率、自动化程度、环境一致性、部署频率
- 陷阱：只关注代码熵，忽视团队和组织熵（后者往往是前者的根因）

### Step 2: 识别熵增来源

- 外部输入：需求变更、新功能追加、第三方库更新、法规变化
- 内部耗散：人员流动、知识流失、 shortcuts 积累、文档过时
- 正反馈：代码越乱→越难修改→越倾向于 hack →代码更乱
- 负反馈缺失：缺乏代码审查、缺乏自动化测试、缺乏重构文化
- 陷阱：将所有问题归因于外部，忽视内部流程和文化的熵增贡献

### Step 3: 计算抗熵成本

- 量化维护成本：当前技术债务的利息（额外开发时间占比）
- 评估重构收益：重构后预期节省的时间 vs 重构投入的时间
- 考虑机会成本：投入抗熵资源意味着减少新功能开发
- 设定阈值：当技术债务利息超过新功能开发时间的30%时，必须优先还债
- 陷阱：低估重构成本（冰山之下还有依赖重构）或高估收益（重构后仍可能继续腐化）

### Step 4: 设计抗熵策略

- 持续重构：将重构融入日常开发（ boy scout rule ：离开营地时比来时更干净）
- 自动化防护：CI/CD、自动化测试、静态分析、依赖扫描
- 知识管理：文档、代码注释、技术分享、结对编程
- 架构守护：架构决策记录（ADR）、接口契约、模块边界
- 陷阱：试图通过"大爆炸式重构"一次性解决所有问题，导致项目停滞

### Step 5: 建立负反馈回路

- 监控指标：代码复杂度趋势、测试覆盖率变化、缺陷密度、部署频率
- 定期审计：每季度技术债务评估、架构健康度检查
- 文化机制：奖励代码清理、惩罚技术债务累积、将质量纳入绩效考核
- 陷阱：建立了监控但不采取行动，或行动后无效果评估

### Step 6: 总结输出

- 整合前面各步的分析结果，形成完整的抗熵方案
- 不遗漏熵值评估、来源识别和抗熵策略
- 明确下一步应该调用哪个 Skill（如需要重构技术则调用 refactoring）
- 输出应包含：熵值评估 + 来源分析 + 抗熵策略 + 监控指标 + 风险 + 下一步

## 关键原则

1. **秩序需要持续投入**：代码不会自动保持整洁，必须通过持续的工程实践来维持。原因：自然趋势是混乱，不投入就等于放任腐化。
2. **开放系统才能抗熵**：封闭系统必然趋向热寂，开放系统通过引入外部能量（新思路、新工具、新人员）维持活力。原因：封闭导致知识停滞，开放带来负熵流。
3. **小步快跑优于大爆炸**：持续的小规模重构比大规模重写更有效。原因：大爆炸重构风险高、周期长，且重构期间无法响应业务需求。
4. **文化比工具更重要**：自动化工具可以检测问题，但无法改变人的行为。原因：工具是手段，文化是土壤，没有文化支撑的工具会被绕过。

## 自检清单

**执行规则**：
- **执行方**：Agent 自检，在最终输出前逐项确认
- **未通过处理**：任一检查项未通过，必须修正后重新自检，禁止跳过
- **记录方式**：自检结果写入 `self_check_result.md`（临时文件），包含：通过项、未通过项、修正记录
- **强制停止项**：以下检查项未通过时，必须停止输出并修正：
  - [ ] 已按思考流程完成全部步骤，无跳过（或已标注提前终止原因）
  - [ ] 输出具体可执行，不含模糊建议
  - [ ] 标注了关键假设和不确定性

**完整检查项**：

- [ ] 已按思考流程完成全部步骤，无跳过（或已标注提前终止原因）
- [ ] 每步思考要点已覆盖，无遗漏
- [ ] 每步结论标注了置信度和反证可能性
- [ ] 考虑了二阶/三阶效应（如果适用）
- [ ] 输出具体可执行，不含模糊建议（如"优化一下"）
- [ ] 标注了关键假设和不确定性
- [ ] 识别了最大风险点和缓解措施
- [ ] 建议按优先级排序，不是平铺罗列
- [ ] 明确了下一步应该调用哪个 Skill（如果需要）

**熵增思维特有关键检查项**：

- [ ] **熵增特供**：是否评估了系统熵值？代码复杂度、测试覆盖率、团队知识分布、架构漂移是否被量化？
- [ ] **熵增特供**：是否识别了熵增来源？外部输入（需求变更、第三方更新）、内部耗散（人员流动、 shortcuts ）、正反馈回路是否被逐一识别？
- [ ] **熵增特供**：是否计算了抗熵成本？技术债务利息（额外开发时间占比）、重构投入产出比、机会成本是否被量化？
- [ ] **熵增特供**：是否设计了抗熵策略？持续重构、自动化防护、知识管理、架构守护是否被综合考虑？是否避免了大爆炸式重构？
- [ ] **熵增特供**：是否建立了负反馈回路？监控指标（复杂度趋势、覆盖率变化）、定期审计、文化机制是否被设计？
- [ ] **熵增特供**：是否考虑了系统开放性？是否引入了外部负熵流（新工具、新思路、新人员）？封闭系统是否被识别为高风险？
