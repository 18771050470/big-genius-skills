# 决策树分析示例

## 示例1：企业是否推出新产品？

**场景**：某公司考虑投资1000万元研发新产品A。市场部预测：
- 高需求（概率40%）：年净利润500万元，持续5年
- 中等需求（概率35%）：年净利润200万元，持续5年  
- 低需求（概率25%）：年亏损200万元（仅1年即停产）

同时，公司可以选择“延缓半年推出”（投入800万元），此时概率变为：高需求30%，中需求45%，低需求25%；收益与前者一致（但只有4.5年收益期）。另一种方案是“维持现状”，收益为0。

### Step 1：定义决策问题
是否推出新产品A？目标为最大化净现值（NPV）。

### Step 2：列出备选方案
1. 立即推出（投资1000万）
2. 延缓半年推出（投资800万）
3. 不推出（收益0）

### Step 3：列出结果与概率

**方案1（立即）**：
- 高需求 40%：年收益500万×5年 = 2500万，NPV = 2500 - 1000 = 1500万
- 中需求 35%：年收益200万×5年 = 1000万，NPV = 1000 - 1000 = 0
- 低需求 25%：年亏损200万×1年 = -200万，NPV = -200 - 1000 = -1200万

**方案2（延缓）**：
- 高需求 30%：年收益500万×4.5年 = 2250万，NPV = 2250 - 800 = 1450万
- 中需求 45%：年收益200万×4.5年 = 900万，NPV = 900 - 800 = 100万
- 低需求 25%：年亏损200万×1年 = -200万，NPV = -200 - 800 = -1000万

### Step 4 & 5：计算期望值

方案1：E = 0.4×1500 + 0.35×0 + 0.25×(-1200) = 600 + 0 - 300 = **300万**
方案2：E = 0.3×1450 + 0.45×100 + 0.25×(-1000) = 435 + 45 - 250 = **230万**
方案3：E = 0

**决策**：方案1（立即推出）期望值最大，推荐。

### Step 6：敏感性分析

当高需求概率下降至30%时（保持其他比例），方案1期望值变为 0.3×1500 + 0.35×0 + 0.35×(-1200) = 450 - 420 = 30万，仍高于方案2的230万？重新计算：方案2不受影响？调整后需重新做完整分析。但发现方案1对高需求概率敏感，若高需求概率再降10%（20%），E=0.2×1500+0.35×0+0.45×(-1200)=300-540=-240万，低于方案2。因此存在临界点。

### Step 7：输出建议

推荐立即推出，但应密切关注市场初期反馈，准备在需求低于预期时及时止损。

## 示例2：技术架构选型

**场景**：某团队需选择后端语言框架，候选：Python (Django)、Go (Gin)、Java (Spring)。目标为“在满足性能需求的前提下，将总开发与运维成本控制在500万以内”。

- Python：开发成本200万，运维成本（3年）150万；性能满足概率90%（若满足则收益1000万，若不满足则需重构，收益为0）
- Go：开发成本250万，运维成本100万；性能满足概率80%（收益1000万）
- Java：开发成本300万，运维成本200万；性能满足概率95%（收益1000万）

### Step 1：定义决策问题
选择框架使期望净收益（收益-成本）最大，且不超500万成本上限。

### Step 2：备选方案
1. Python
2. Go
3. Java

### Step 3 & 4：概率与收益（注意成本为确定项）

Python：总成本350万，满足时净收益=1000-350=650万，不满足时净收益=0-350=-350万；概率0.9/0.1
Go：总成本350万，满足净收益=650万，不满足净收益=-350万；概率0.8/0.2
Java：总成本500万（正好上限），满足净收益=500万，不满足净收益=-500万；概率0.95/0.05

### Step 5：期望值计算

Python期望值=0.9×650 + 0.1×(-350)=585-35=550万
Go期望值=0.8×650 + 0.2×(-350)=520-70=450万
Java期望值=0.95×500 + 0.05×(-500)=475-25=450万

**期望值最大**：Python（550万），且成本350万未超上限。

### Step 6：敏感性分析

当Python性能满足概率低于73.1%时（计算：设p，p×650+(1-p)×(-350)=450 => p=0.8? 精确计算：650p-350+350p=1000p-350=450 => 1000p=800 => p=0.8），低于Go的期望值。需确认开发过程中能提升性能至90%以上。

### Step 7：输出建议

选择Python框架，但需预留重构缓冲（如与Go并行验证关键模块）。

## 反例：不该触发此 Skill 的场景

**场景**：用户问“午餐是吃面条还是吃米饭？我只想吃一顿简单的午餐，没什么风险，价格也差不多。”

**分析**：该决策不涉及不确定性和概率分支（如果仅仅是口味偏好），且没有显著的收益量化差异。使用决策树会过度工程化，浪费时间。正确的做法是直接凭个人偏好或随机选择即可。

**不使用决策树的理由**：缺乏不确定性（选项结果确定且可忽略差异），没有需要量化的概率或风险，决策简单无需结构化分析。
