# 分析示例

## 示例 1：守门员点球扑救决策

**输入**：我们客服团队面对用户投诉时，总是立即给出解决方案，即使有时不回复或等待更多信息可能是更好的选择。

```
Step 1: 识别行动信号 → 产出：发现"总是立即给出解决方案"表述，行动成为默认反应
Step 2: 评估行动vs不行动 → 产出：行动（立即回复）vs不行动（等待信息），分析哪种结果更好
Step 3: 分析行动机制 → 产出：规范理论：行动是规范，不行动被视为失职；后悔预期：不行动后悔更强烈
Step 4: 校正行动偏差 → 产出：建议建立决策树：何时行动、何时等待、何时不行动
```

**输出**：
```
## 分析结论
### 核心发现
1. 团队存在明显行动偏差：将"立即行动"等同于"负责任"
2. 30%的立即回复因信息不足导致二次沟通，增加总处理时间
3. 等待收集完整信息后回复，首次解决率提升40%

### 建议（按优先级）
1. [高优先级] 建立"行动vs等待"决策树，明确触发条件
2. [高优先级] 将首次解决率纳入KPI，而非响应速度
3. [中优先级] 培训团队识别需要等待的场景

### 关键假设
- 用户更关心问题被解决，而非立即回复
- 等待时间<2小时可接受

### 风险
- 过度等待可能导致用户不满
- 需要平衡响应速度和解决质量
```

## 示例 2：产品经理功能迭代决策

**输入**：这个版本我们必须做点新功能，不然老板会觉得我们没干活。

```
Step 1: 识别行动信号 → 产出：发现"必须做点新功能"表述，为行动而行动
Step 2: 评估行动vs不行动 → 产出：新功能开发成本vs优化现有功能收益
Step 3: 分析行动机制 → 产出：可见性偏好：新功能是可见产出，优化是隐性改进；汇报压力：需要向老板展示成果
Step 4: 校正行动偏差 → 产出：建议量化优化收益，建立"不行动也是决策"文化
```

**输出**：
```
## 分析结论
### 核心发现
1. "做点新功能"是为行动而行动，非用户需求驱动
2. 现有功能优化可提升留存15%，远高于新功能预期收益5%
3. 老板关注的是业务指标，而非功能数量

### 建议（按优先级）
1. [高优先级] 用数据展示优化现有功能的ROI
2. [高优先级] 向老板汇报业务指标提升，而非功能数量
3. [中优先级] 建立"优化也是产出"的团队文化

### 关键假设
- 现有功能优化数据可量化
- 老板关注业务结果而非过程

### 风险
- 长期无新功能可能影响市场竞争力
- 需要平衡优化和创新
```

## 反例：不该触发此 Skill 的场景

**输入**：服务器CPU使用率超过90%，需要立即扩容。

**原因**：这是明确的紧急响应场景，有客观阈值触发行动，不是行动偏差。行动偏差指在"行动vs不行动"无明确优劣时偏向行动。
