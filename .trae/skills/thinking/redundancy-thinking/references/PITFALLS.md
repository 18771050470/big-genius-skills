# 冗余思维 - 常见陷阱

> Agent 容易犯的错误。每个陷阱包含：问题描述 + 反例 + 正例。

## 陷阱 1：冗余越多越安全

**问题**：认为多一层冗余就多一层保护，无节制地堆叠冗余，导致系统复杂度爆炸，冗余本身成为最大故障源。

❌ **Before（错误做法）**
```
系统设计：
- 应用层：10台服务器 + 3层负载均衡
- 数据库：主从 + 从从 + 备份从 + 异地灾备
- 缓存：Redis集群 + 本地缓存 + CDN缓存
- 消息队列：Kafka集群 + 备用RabbitMQ + 日志兜底
→ 结果：运维成本爆炸，切换逻辑互相干扰，一次故障花了30分钟定位
```

✅ **After（正确做法）**
```
系统设计：
- 应用层：3台服务器（满足N+1，容忍1台故障）+ 1层LB
- 数据库：主从（半同步复制）+ 自动故障转移
- 缓存：Redis Sentinel（3节点，够用）
- 消息队列：Kafka 3节点集群
→ 理由：每层只满足N+1，够用即可。关键不是冗余层数多，而是覆盖了真正的故障模式
```

**原因**：NASA可靠性工程发现"冗余系统的管理逻辑"自身的故障率可能超过被保护组件的故障率。每增加一层冗余，系统的总复杂度呈超线性增长。冗余思维的目标是"恰好够用"，而非"越多越好"。

## 陷阱 2：忽视共因故障（Common Cause Failure）

**问题**：部署了多台"冗余"服务器，但它们共享同一个机房/电源/网络，一次断电全部瘫痪。

❌ **Before（错误做法）**
```
"我们做了高可用！3台服务器 + 负载均衡，挂了1台还有2台。"
→ 实际：3台都在同一个机柜，同一个电源排插。排插坏了全挂。
```

✅ **After（正确做法）**
```
高可用设计：
- 3台服务器分散在3个不同的可用区（物理隔离的机房）
- 每个可用区有独立电源和网络
- 验证：确认3个可用区不共享任何基础设施
→ 真正的冗余需要消除共因故障，而不只是增加节点数量
```

**原因**：Titanic的水密舱壁是共因故障的经典案例——舱壁高度不够，一次撞击击穿多个水密舱。冗余节点如果共享脆弱性，等于没有冗余。

## 陷阱 3：假设切换一定生效

**问题**：设计了主从切换方案但从不在生产环境测试，结果真故障时切换脚本有Bug。

❌ **Before（错误做法）**
```
"我们配置了MySQL自动主从切换，是从库提升为主库的脚本已经在仓库里了。"
→ 故障发生：脚本因为某个环境变量未设置而失败，手动操作花了45分钟
```

✅ **After（正确做法）**
```
高可用验证：
- 每月执行一次混沌工程：随机Kill主库，验证自动切换
- 切换脚本和生产环境绑定，每次部署后自动验证
- 记录每次切换的实际耗时：MTTD=3秒，MTTR=18秒
→ 未经验证的冗余不是冗余，是自我欺骗
```

**原因**：大量生产事故的根因是"我们以为切换会生效"。冗余机制和代码一样，需要被测试和验证。混沌工程（Chaos Engineering）正是为此而生。
