# 分析示例

## 示例 1：错误共识——技术方案选择

**输入**：这个技术方案很明显应该用微服务啊，正常人都会这么选，怎么还有人反对？

```
Step 1: 识别共识信号 → 产出：发现"正常人都会这么选"表述，假设他人与自己观点一致
Step 2: 评估共识程度 → 产出：团队5人，3人支持微服务，2人反对，共识度60%非100%
Step 3: 分析共识机制 → 产出：选择性暴露：只与支持者讨论；虚假共识：高估支持者比例
Step 4: 校正共识效应 → 产出：建议收集团队真实意见，理解反对者理由
```

**输出**：
```
## 分析结论
### 核心发现
1. "正常人都会选"是虚假共识，实际团队共识度只有60%
2. 反对者理由：团队规模小(5人)、运维能力不足、单体更简单
3. 只与支持微服务的人讨论，强化了"大家都同意"的错觉

### 建议（按优先级）
1. [高优先级] 召集团队全员讨论，收集真实意见
2. [高优先级] 理解反对者理由，评估其合理性
3. [中优先级] 基于团队实际情况决策，而非假设共识

### 关键假设
- 反对者理由可能揭示真实风险
- 全员讨论可获得更完整信息

### 风险
- 虚假共识可能导致忽视重要风险
- 强行推进可能影响团队凝聚力
```

## 示例 2：错误共识——产品功能优先级

**输入**：这个功能用户肯定需要，这还用说吗？赶紧排期开发。

```
Step 1: 识别共识信号 → 产出：发现"用户肯定需要"、"这还用说"表述，假设需求不言自明
Step 2: 评估共识程度 → 产出：无用户调研数据支撑，"肯定需要"是主观假设
Step 3: 分析共识机制 → 产出：投射偏差：将自己需求投射到用户；内部视角：团队内部讨论强化共识错觉
Step 4: 校正共识效应 → 产出：建议用用户调研验证需求，而非假设共识
```

**输出**：
```
## 分析结论
### 核心发现
1. "用户肯定需要"无数据支撑，是团队主观假设
2. 用户调研Top 10需求中无此功能
3. 团队内部讨论强化了"这很明显"的错觉

### 建议（按优先级）
1. [高优先级] 用用户调研数据验证需求优先级
2. [高优先级] 如确实需要，先做MVP验证
3. [中优先级] 建立需求验证流程，避免主观假设

### 关键假设
- 用户调研样本具有代表性
- 团队内部视角≠用户视角

### 风险
- 开发用户不需要的功能浪费资源
- 主观假设可能导致产品方向偏差
```

## 反例：不该触发此 Skill 的场景

**输入**：调研显示85%用户需要这个功能，我们可以优先开发。

**原因**：这是基于调研数据的结论(85%)，不是假设共识。错误共识指无数据支撑的共识假设。
