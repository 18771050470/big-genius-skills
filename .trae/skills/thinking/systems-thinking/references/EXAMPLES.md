# 分析示例

## 示例 1：微服务架构的团队问题

**输入**：拆了微服务后团队效率反而下降了

```
Step 1: 界定系统边界
→ 系统内：5个微服务团队、共享数据库、API网关、CI/CD流水线
→ 系统外：客户、第三方服务、公司管理层
→ 系统目标：快速交付高质量软件
→ 产出：系统边界定义

Step 2: 识别组件和连接
→ 组件：团队A(用户服务)、团队B(订单服务)、团队C(支付服务)、DBA团队、运维团队
→ 连接：A→DB(共享数据库)、B→C(同步调用)、A→B(事件通知)、所有团队→CI/CD
→ 关键发现：所有团队共享同一数据库，任何schema变更需要协调5个团队
→ 产出：组件连接图

Step 3: 发现反馈回路
→ 增强回路：服务增多→协调成本增加→交付变慢→加班→人员流失→服务更无人维护
→ 调节回路：交付变慢→管理层施压→要求加快→质量下降→线上故障→被迫返工
→ 延迟：schema变更请求到DBA执行平均延迟3天
→ 产出：2个关键回路

Step 4: 识别杠杆点
→ 系统原型：增长上限（服务数量增长受限于协调成本）
→ 杠杆点1（高阶）：改变团队组织方式，按业务域而非技术域划分（改变规则）
→ 杠杆点2（中阶）：每个服务独立数据库（改变结构）
→ 杠杆点3（低阶）：增加DBA人手（调整参数，效果有限）
→ 产出：杠杆点排序
```

**输出**：
```
## 分析结论

### 核心发现
1. 微服务拆分只拆了代码，没拆数据（共享数据库是瓶颈）
2. 协调成本随服务数量呈O(n²)增长
3. 增强回路正在加速团队效率下降

### 建议（按优先级）
1. [高] 按业务域重组团队，每个团队拥有完整数据所有权，2个月
2. [高] 服务独立数据库，消除共享依赖，1个月
3. [中] 建立服务间异步通信机制，减少同步调用，3周

### 关键假设
- 业务域边界可清晰划分
- 数据迁移可在不停机情况下完成
- 团队具备全栈能力
```

## 示例 2：用户增长停滞

**输入**：产品用户增长到10万后停滞不前

```
Step 1: 界定系统边界
→ 系统内：产品、用户获取渠道、用户留存机制、推荐系统
→ 系统外：竞品、市场环境、宏观经济
→ 系统目标：用户持续增长
→ 产出：系统边界

Step 2: 识别组件和连接
→ 组件：新用户获取、用户激活、用户留存、用户推荐、用户流失
→ 连接：营销→新用户、新用户→激活→留存、留存用户→推荐→新用户
→ 关键发现：留存率从60%降到30%，推荐率从15%降到5%
→ 产出：组件连接图

Step 3: 发现反馈回路
→ 增强回路（正向）：用户增长→更多推荐→更多新用户（但正在减弱）
→ 调节回路：用户增长→服务器压力→体验下降→留存降低→增长放缓
→ 延迟：用户体验下降到流失平均延迟2周
→ 产出：增长上限系统原型

Step 4: 识别杠杆点
→ 系统原型：增长上限（留存率下降限制了增长）
→ 杠杆点1：提升留存率（改变结构），而非继续投入获客（调整参数）
→ 杠杆点2：优化推荐机制（增强正向回路）
→ 产出：杠杆点分析
```

**输出**：
```
## 分析结论

### 核心发现
1. 增长停滞的根因是留存率下降，不是获客不足
2. 留存率下降由体验恶化引起（服务器压力+功能膨胀）
3. 当前处于"增长上限"系统原型

### 建议（按优先级）
1. [高] 暂停新功能开发，专注提升留存率，1个月
2. [高] 优化服务器性能，消除体验瓶颈，2周
3. [中] 重新设计推荐激励机制，3周

### 关键假设
- 留存率可被产品体验改善
- 服务器性能是体验下降的主因
- 推荐机制对获客有显著影响
```
