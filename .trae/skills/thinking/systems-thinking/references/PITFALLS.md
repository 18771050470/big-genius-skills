# 常见陷阱

## 陷阱 1：只见树木不见森林

**反例**：
```
只分析单个组件的性能，忽略了组件间的交互才是瓶颈
→ "服务A响应快，服务B响应快，所以整体快"（错误）
```

**正例**：
```
分析组件间的连接和反馈回路
→ "服务A和B之间的同步调用链导致整体延迟是各服务之和"
```

**原因**：系统行为来自组件互动，而非组件本身。

---

## 陷阱 2：忽略延迟效应

**反例**：
```
"改了配置后没效果，说明改错了"（可能延迟还没到）
```

**正例**：
```
"改了配置，预计2周后看到效果，因为用户行为改变有延迟"
```

**原因**：延迟导致决策者基于过时信息行动，产生振荡。

---

## 陷阱 3：在低阶杠杆点浪费精力

**反例**：
```
持续调整参数（增加服务器、优化SQL），但不改变系统结构
```

**正例**：
```
先尝试高阶杠杆点（改变规则、信息流），再考虑参数调整
```

**原因**：参数调整效果有限且短暂，结构改变效果持久。

---

## 陷阱 4：干预副作用未被评估

**反例**：
```
"解决了A问题"但引发了B问题（未预见）
```

**正例**：
```
"解决A问题，但可能导致B问题，需要同时部署C措施"
```

**原因**：系统组件相互关联，干预一个会影响其他。
