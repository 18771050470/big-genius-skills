# 分析示例

## 示例 1：项目失败复盘

**输入**：项目延期3个月交付，分析决策质量

```
Step 1: 锚定真实事件
→ 实际结果：延期3个月，预算超支40%
→ 决策时信息：需求文档完整、技术方案评审通过、团队有类似项目经验
→ 决策时间：2024年1月，结果时间：2024年7月

Step 2: 构建反事实场景
→ 场景A：如果采用敏捷迭代，每2周交付一个可用版本
→ 场景B：如果增加1名资深架构师
→ 场景C：如果砍掉30%非核心需求
→ 所有场景在决策时都是可行的

Step 3: 推演替代结果
→ 场景A：预期延期1个月（概率60%），预算超支15%
→ 场景B：预期延期1.5个月（概率50%），预算超支25%（人力成本增加）
→ 场景C：预期准时交付（概率70%），预算不超支，但功能完整性降低
→ 真实结果期望值：延期3个月，超支40%
→ 反事实最优期望值：场景C，准时交付

Step 4: 提取学习
→ 决策过程错误：过度乐观估计、未设置里程碑检查点
→ 认知偏差：规划谬误（低估时间）、确认偏误（只看支持按期交付的证据）
→ 改进清单：强制设置20%时间缓冲、每2周里程碑检查、引入外部评审
```

**输出**：
```
## 分析结论

### 核心发现
1. 决策质量差：基于过度乐观的估计，无缓冲
2. 运气成分：遇到2个关键技术难题（概率30%的事件发生了）
3. 反事实分析显示砍需求+敏捷迭代可避免80%的延期

### 建议
1. [高] 所有项目强制设置20%时间缓冲
2. [高] 采用敏捷迭代，每2周里程碑检查
3. [中] 引入外部技术评审，避免团队内部确认偏误
```
