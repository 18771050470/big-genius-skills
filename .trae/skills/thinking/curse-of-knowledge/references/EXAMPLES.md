# 分析示例

## 示例 1：技术文档编写中的知识诅咒

**输入**：这个API文档我已经写得很清楚了，按步骤操作就行，用户怎么还说看不懂？

```
Step 1: 识别知识诅咒信号 → 产出：发现"写得很清楚了"表述，专家视角替代新手视角
Step 2: 评估知识差异 → 产出：编写者知道背景知识，用户可能不知道前置概念
Step 3: 分析诅咒机制 → 产出：一旦知道就无法想象不知道的状态；专家盲点：跳过"显然"步骤
Step 4: 校正知识诅咒 → 产出：建议新手测试文档，识别缺失的前置知识
```

**输出**：
```
## 分析结论
### 核心发现
1. 文档从专家视角编写，假设用户知道前置概念
2. 新手测试显示：30%用户卡在第二步，因不知道"认证token"是什么
3. 文档缺少：前置知识说明、常见错误排查、示例代码

### 建议（按优先级）
1. [高优先级] 添加前置知识章节：需要什么背景、如何准备
2. [高优先级] 邀请3个新手试读文档，记录卡点
3. [中优先级] 添加常见错误排查章节

### 关键假设
- 目标用户包含新手
- 新手测试可识别真实卡点

### 风险
- 过度简化可能使文档冗长
- 需要平衡完整性和简洁性
```

## 示例 2：产品经理需求传达中的知识诅咒

**输入**：这个需求很简单啊，就是加个按钮，开发怎么说要3天？

```
Step 1: 识别知识诅咒信号 → 产出：发现"很简单啊"表述，产品视角替代开发视角
Step 2: 评估知识差异 → 产出：产品知道业务逻辑，开发需要理解完整上下文
Step 3: 分析诅咒机制 → 产出：产品脑中已有完整逻辑，但需求文档可能遗漏细节
Step 4: 校正知识诅咒 → 产出：建议补充需求细节，明确边界条件和异常处理
```

**输出**：
```
## 分析结论
### 核心发现
1. "加个按钮"背后涉及：权限校验、数据校验、状态同步、异常处理
2. 产品认为"简单"是因已知业务逻辑，但开发需要完整信息
3. 需求文档缺少：边界条件、异常场景、与其他模块交互

### 建议（按优先级）
1. [高优先级] 补充需求细节：权限、校验、状态、异常
2. [高优先级] 与开发对齐完整上下文，而非只描述表面功能
3. [中优先级] 建立需求模板，确保关键信息不遗漏

### 关键假设
- 开发估算基于完整实现
- 需求文档确实遗漏关键信息

### 风险
- 需求不完整可能导致返工
- 过度沟通可能降低效率
```
