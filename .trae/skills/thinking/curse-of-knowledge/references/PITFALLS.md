# 常见陷阱

## 陷阱 1：用专家视角评估易懂性

**问题**：因自己知道而认为别人也应该知道。

❌ **Before（错误做法）**
```markdown
## 分析结论
文档写得很清楚了，用户看不懂是他们的问题。
建议：不需要修改文档。
```
*问题：知识诅咒使无法从新手视角评估文档*

✅ **After（正确做法）**
```markdown
## 分析结论
### 新手测试反馈
| 步骤 | 通过率 | 主要卡点 |
|------|--------|---------|
| 第一步 | 90% | 无 |
| 第二步 | 60% | 不知道"认证token"是什么 |
| 第三步 | 40% | 缺少示例代码 |

### 建议
1. 添加前置知识说明
2. 第二步添加token获取教程
3. 每步添加示例代码
```
*优势：用新手测试数据替代专家直觉，识别真实卡点*

**原因**：Camererer等(1989)实验证明：知道答案的人无法想象不知道的状态。Birch & Bloom(2007)证明即使提醒自己有知识诅咒，偏差仍然存在。

## 陷阱 2：跳过"显然"的步骤

**问题**：专家认为"显然"的步骤，新手可能需要详细说明。

❌ **Before（错误做法）**
```markdown
## 分析结论
这个需求很简单，就是加个按钮。
建议：开发3天太多，1天就行。
```
*问题：知识诅咒使跳过"显然"的隐含步骤*

✅ **After（正确做法）**
```markdown
## 分析结论
### 完整任务拆解
| 步骤 | 耗时 | 说明 |
|------|------|------|
| 权限校验 | 0.5天 | 谁能看到/点击按钮 |
| 前端UI | 0.5天 | 按钮样式和位置 |
| 后端API | 1天 | 新增接口、数据校验 |
| 状态同步 | 0.5天 | 按钮点击后状态更新 |
| 测试 | 0.5天 | 功能测试、边界测试 |
| 合计 | 3天 | |

### 建议
1. 3天估算是合理的
2. 需求文档应包含完整任务拆解
```
*优势：完整拆解揭示"简单"背后的复杂性*

**原因**：知识诅咒使专家脑中自动补全隐含步骤，但新手需要显式说明。

## 陷阱 3：忽视受众差异

**问题**：用同一套文档/沟通方式面对不同水平受众。

❌ **Before（错误做法）**
```markdown
## 分析结论
文档就这一份，所有人都看这个。
建议：不需要区分受众。
```
*问题：知识诅咒使忽视不同受众的知识差异*

✅ **After（正确做法）**
```markdown
## 分析结论
### 受众分层
| 受众 | 知识水平 | 文档需求 |
|------|---------|---------|
| 新手 | 无背景知识 | 完整教程+前置知识 |
| 中级 | 了解基本概念 | 快速上手+常见问题 |
| 专家 | 熟练使用 | API参考+最佳实践 |

### 建议
1. 为不同受众提供不同文档
2. 新手文档占60%篇幅，专家文档占20%
```
*优势：受众分层确保各水平用户都有合适文档*

**原因**：知识诅咒使专家难以想象新手状态。受众分层强制考虑不同知识水平。
