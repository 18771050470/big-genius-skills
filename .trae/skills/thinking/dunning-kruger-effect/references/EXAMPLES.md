## 示例 1：过度自信的新手开发者

**场景**：一名刚自学 React 三个月的初级开发者，在团队 Code Review 中声称自己的组件设计是“最优方案”，并拒绝修改建议。团队 Leader 希望进行评估。

**输入数据**：
- 领域：React 前端开发
- 自我评分：9/10
- 经验年限：<1年（3个月）
- 客观表现证据：完成的2个简单组件（表单、列表），存在性能问题（未使用useMemo、大量重渲染）；同行反馈：代码可读性差、未遵循最佳实践；测试结果：单元测试覆盖率5%
- 最近能力里程碑：完成第一个个人项目（Todo App）

**Step 1 输出**：自我评分9 vs 客观表现差 → 偏差方向：高估（强烈）

**Step 2 输出**：经验 <1年，里程碑为简单个人项目 → 当前阶段：**愚昧之巅**；典型偏差：高估

**Step 3 输出**：未参与过任何对比校准（没有代码评审经验，未阅读过专业源码）→ 元认知成熟度：低

**Step 4 输出**：React前端领域复杂度中等，反馈速度中等（Code Review 即可提供反馈）→ 领域修正因子：0.8（中等偏差幅度）

**Step 5 输出**：综合判断：个体处于愚昧之巅，严重高估自身能力。建议：
- 立即参加至少3次中级开发者主导的 Code Review，记录每一条修改建议
- 完成一本系统性 React 书籍（如《React 设计原理》）并做笔记
- 自我评分每两周校准一次，同步记录学习日志
- 短期目标：写出通过的单元测试覆盖率达到50%的组件

**Step 6（可选）**：设定一个月后重新评估，期间每周与一位资深开发者结对编程。

## 示例 2：低估自己的资深架构师

**场景**：一位有10年后端经验的架构师，在技术选型会议上总是说“我的方案可能不够好”，即使被多数人认可，他仍然怀疑自己。团队希望分析这是否为达克效应。

**输入数据**：
- 领域：后端架构设计（微服务、分布式系统）
- 自我评分：6/10（自称“刚及格”）
- 经验年限：10年
- 客观表现证据：主导过3个以上高并发系统，故障率低于行业平均；同行评价：被4位CTO评为“顶尖水平”；测试结果：架构评审通过率100%
- 最近能力里程碑：设计并落地了一个日均10亿请求的网关

**Step 1 输出**：自我评分6 vs 客观表现卓越 → 偏差方向：低估（强烈）

**Step 2 输出**：经验10年，里程碑为复杂网关设计 → 当前阶段：**专家（开悟之坡晚期）**；典型偏差：低估

**Step 3 输出**：多次参与跨公司技术交流，有丰富的对比校准经历 → 元认知成熟度：高（但仍低估）

**Step 4 输出**：架构设计领域复杂度高、反馈慢（系统上线数月后才能验证）→ 领域修正因子：1.2（偏差强度增大）

**Step 5 输出**：综合判断：个体处于专家阶段，显著低估自身能力。可能原因：长期与更资深专家对比、完美主义倾向。建议：
- 创建个人成就清单，客观记录每一个成功项目的关键决策和结果
- 主动担任内部技术讲师，通过输出获得外部正反馈
- 在技术提案中强制使用“我的方案基于以下XX个成功先例”的结构化表达
- 参加行业会议分享，用听众反馈强化自身认知

**Step 6（可选）**：每季度进行一次成就回顾，并与团队信任的同事进行互评校准。

## 反例：不应触发该 Skill 的场景

**场景**：一名前端工程师在面试中要求评估自己的React能力，但公司已经进行了统一的在线编码测试，得分为92/100，同时面试官给出了“精通”的评语。

**原因**：已有客观量化标准（在线测试得分92）和明确的第三方评价（面试官评语“精通”），且该评估不包含需要元认知判断的主观成分。此时个体若自信，是合理自信，而非达克效应偏差。因此不应调用达克效应分析。

**正确做法**：直接基于数据给出结论：“根据测试得分和面试官评价，你的React能力属于精通水平，自信合理。” 无需进行Step2-5的偏差分析。
