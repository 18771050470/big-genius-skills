# 契约思维 - 分析示例

## 示例质量验证标准

| 检查项 | 标准 |
|--------|------|
| **完整性** | 包含输入→每步动作→每步输出→最终结论 |
| **可追踪性** | 最终结论必须能追溯到具体步骤 |
| **具体性** | 输出含具体代码/接口定义 |
| **边界覆盖** | 包含至少1个"不该触发"的失败案例 |

## 示例 1: 微服务间支付接口契约

**输入**：订单服务需要调用支付服务创建支付单。当前约定是口头沟通的，集成时频繁出现问题。

```
Step 1: 识别交互边界
→ 产出：订单服务→支付服务的REST调用是跨团队边界，当前假设完全隐式

Step 2: 定义前置条件
→ 产出：
  - 订单ID必须存在且状态为PENDING_PAYMENT
  - 金额必须>0且≤单笔限额50000
  - 用户ID必须已完成实名认证
  - 如果前置条件不满足 → 返回400 + 具体错误码

Step 3: 定义后置条件
→ 产出：
  - 成功时返回唯一的支付单号(paymentId)
  - 支付状态为CREATED
  - 该支付单在15分钟内可被查询（幂等保证：重复请求返回同一paymentId）

Step 4: 定义不变式
→ 产出：一个订单同时只能有一个活跃支付单（payment.status != EXPIRED/CANCELLED）

Step 5: 定义错误契约
→ 产出：
  - 网络超时 → 调用方可安全重试（幂等）
  - 余额不足 → 400 + 明确错误码，不可重试
  - 服务降级 → 503 + 建议稍后重试
  - 重试语义：POST /payments 幂等（相同订单ID→相同结果）
```

**输出**：

```
## 契约定义

### 前置条件
- orderId存在且状态=PENDING_PAYMENT
- amount ∈ (0, 50000]
- userId已完成实名认证

### 后置条件（成功）
- 返回 { paymentId, status: "CREATED" }
- 幂等：同一orderId重复请求返回相同paymentId

### 错误契约
| 场景 | HTTP | 可重试 |
|------|------|--------|
| 参数非法 | 400 | 否 |
| 余额不足 | 400(INSUFFICIENT) | 否 |
| 超时 | 503 | 是（幂等） |

### 关键假设
- 调用方会校验实名认证状态（否则被支付服务拒绝）
- 支付服务内部的事务保证扣款和返回paymentId原子性
```

## 示例 2: 函数级契约冗余校验

**输入**：一个工具函数 `divide(a, b)` 需要明确契约。

```typescript
// 不良：契约完全隐式
function divide(a: number, b: number): number {
  return a / b;  
  // 前置条件：b != 0 — 但没检查
  // 后置条件：返回有限数字 — 但b=0时返回Infinity
}

// 契约显式化方案
function divide(
  a: number, 
  b: number
): number {
  // 前置条件校验 (fail-fast)
  if (b === 0) {
    throw new Error("Precondition violated: divisor must not be zero");
  }
  // 后置条件：返回有限数字
  // 不变式：该函数纯函数，无副作用
  return a / b;
}
```

## 反例：不该触发的场景

**输入**："我们团队内部有两个开发，需要为每个Helper函数写正式的契约文档吗？"

**原因**：同团队高频沟通的小函数不需要形式化契约。口头约定/代码注释/类型签名已经足够。契约思维的强度应与"误解代价"成正比。
