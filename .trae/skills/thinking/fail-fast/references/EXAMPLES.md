# 分析示例

## 示例 1：快速失败——外部 API 依赖

**输入**：我们的服务依赖第三方支付 API，如果支付 API 挂了怎么办？

```
Step 1: 评估失败成本 → 产出：快速失败成本：用户看到错误提示，可重试；延迟失败成本：订单状态不一致、资金损失、用户投诉
Step 2: 识别失败点 → 产出：支付 API 超时、支付 API 返回错误、网络中断、签名验证失败
Step 3: 设计快速失败机制 → 产出：前置检查：支付 API 健康检查（/health）；超时设置：3 秒超时，不等待；快速返回：明确错误码，不静默失败
Step 4: 建立学习机制 → 产出：记录每次失败：时间、错误码、耗时；分析失败模式：超时占比、错误类型分布；优化：根据失败模式调整超时和重试策略
```

**输出**：
```
## 分析结论
### 核心发现
1. 快速失败成本：用户看到错误提示（可接受）
2. 延迟失败成本：订单不一致、资金损失（不可接受）
3. 支付 API 是最可能失败的外部依赖

### 建议（按优先级）
1. [高优先级] 设置 3 秒超时，超时立即返回"支付服务暂时不可用"
2. [高优先级] 调用前检查支付 API 健康状态（/health 端点）
3. [中优先级] 失败后记录详细日志：错误码、耗时、请求 ID
4. [中优先级] 设计降级方案：引导用户使用其他支付方式

### 关键假设
- 支付 API 提供健康检查端点
- 用户可接受短暂失败提示

### 风险
- 超时设置过短可能误杀正常请求
- 需要监控失败率调整策略

### 下一步
建议调用 [boundary-thinking] 设计超时和重试边界
```

## 示例 2：快速失败——数据迁移

**输入**：我们要迁移 1000 万用户数据到新表，怎么保证安全？

```
Step 1: 评估失败成本 → 产出：快速失败成本：迁移中断，可回滚；延迟失败成本：数据损坏、业务中断、恢复困难
Step 2: 识别失败点 → 产出：数据格式不兼容、外键约束冲突、磁盘空间不足、迁移脚本 bug
Step 3: 设计快速失败机制 → 产出：前置验证：抽样 1000 条数据验证格式；分批迁移：每批 1 万条，失败立即停止；回滚机制：每批可独立回滚
Step 4: 建立学习机制 → 产出：记录每批迁移结果；分析失败批次根因；优化迁移脚本
```

**输出**：
```
## 分析结论
### 核心发现
1. 快速失败成本：迁移中断，可回滚（可控）
2. 延迟失败成本：数据损坏（灾难性）
3. 数据格式不兼容是最可能失败点

### 建议（按优先级）
1. [高优先级] 迁移前抽样验证 1000 条数据格式
2. [高优先级] 分批迁移：每批 1 万条，失败立即停止
3. [中优先级] 每批迁移后验证数据一致性
4. [中优先级] 准备回滚脚本，确保可回滚

### 关键假设
- 数据可分批处理
- 回滚脚本可执行

### 风险
- 分批迁移增加时间
- 需要验证每批一致性

### 下一步
建议调用 [inverse-thinking] 设计回滚方案
```

## 反例：不该触发此 Skill 的场景

**输入**：这个功能很重要，不能有任何失败，必须 100% 可靠。

**原因**：这是容错/高可用需求，不是快速失败场景。快速失败适用于可接受失败、需要从失败中学习的场景。
