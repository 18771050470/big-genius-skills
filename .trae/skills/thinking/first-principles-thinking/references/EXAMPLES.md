# 分析示例

## 示例 1：数据库性能优化

**输入**：数据库查询慢，团队建议分库分表，但成本太高

```
Step 1: 识别并质疑现有假设
→ 假设1："慢就必须分库分表"（质疑：慢的根本原因是什么？）
→ 假设2："单库无法承载当前数据量"（质疑：真的吗？数据量多少？）
→ 假设3："分库分表是唯一解"（质疑：其他方案呢？）
→ 产出：3个关键假设

Step 2: 拆解到基本真理
→ 查询性能基本要素：索引、查询计划、数据量、并发数
→ 测量发现：90%慢查询缺少索引，5%是N+1查询问题
→ 基本真理：问题在查询设计，不在存储架构
→ 产出：性能瓶颈分解

Step 3: 从零重建解决方案
→ 方案A：添加缺失索引（成本$5K，1天）
→ 方案B：优化N+1查询（成本$10K，3天）
→ 方案C：分库分表（成本$500K，3个月）
→ 产出：3个方案，成本差异100倍

Step 4: 验证可行性
→ 方案A：立即生效，查询速度提升10倍
→ 方案B：需代码改动，查询速度提升5倍
→ 方案C：过度设计，当前数据量远未达分库分表阈值
→ 产出：ROI对比
```

**输出**：
```
## 分析结论

### 核心发现
1. 性能问题根因是缺失索引和低效查询，不是架构
2. 分库分表成本是索引优化的100倍，但收益不确定
3. 当前数据量（100GB）远未达分库分表阈值（通常>1TB）

### 建议（按优先级）
1. [高] 添加缺失索引，1天，$5K，查询速度提升10倍
2. [高] 优化N+1查询，3天，$10K，查询速度提升5倍
3. [低] 分库分表仅在数据量>500GB时考虑

### 关键假设
- 索引添加不会显著影响写入性能
- N+1查询可被批量查询替代
- 当前数据增长速率保持稳定
```

## 示例 2：产品功能设计

**输入**：用户要求添加更多功能来增加竞争力

```
Step 1: 识别并质疑现有假设
→ 假设1："功能越多竞争力越强"（质疑：用户真的需要这么多功能吗？）
→ 假设2："竞品有我们就必须有"（质疑：差异化不行吗？）
→ 假设3："用户知道他们想要什么"（质疑：用户说的和做的是否一致？）
→ 产出：3个关键假设

Step 2: 拆解到基本真理
→ 用户价值基本要素：解决核心问题的效率、使用频率、替代成本
→ 数据发现：80%用户只使用20%功能，核心功能使用率95%
→ 基本真理：用户需要的是核心功能更好用，而非更多功能
→ 产出：用户行为分析

Step 3: 从零重建解决方案
→ 方案A：精简功能，优化核心体验（成本$50K）
→ 方案B：继续添加功能（成本$200K）
→ 方案C：重新定义产品定位，聚焦细分场景（成本$100K）
→ 产出：3个方案

Step 4: 验证可行性
→ 方案A：用户满意度预期提升30%，流失率降低15%
→ 方案B：功能膨胀导致体验下降，维护成本指数增长
→ 方案C：差异化定位可避开红海竞争
→ 产出：可行性评估
```

**输出**：
```
## 分析结论

### 核心发现
1. 功能数量与用户满意度负相关（功能越多，核心功能越难找）
2. 80/20法则明显：20%功能服务80%用户
3. 竞品功能多但评分低，说明功能多≠体验好

### 建议（按优先级）
1. [高] 精简功能，优化核心体验，2个月，$50K
2. [中] 聚焦细分场景重新定位，3个月，$100K
3. [低] 停止无差别功能竞赛

### 关键假设
- 用户调研数据代表真实行为
- 核心功能优化空间足够大
- 市场存在可聚焦的细分场景
```
