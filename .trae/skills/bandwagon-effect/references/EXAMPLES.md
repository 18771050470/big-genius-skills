# 分析示例

## 示例 1：技术选型中的从众效应

**输入**：其他公司都在用Kubernetes，我们不用就落后了，赶紧上K8s。

```
Step 1: 识别从众信号 → 产出：发现"其他公司都在用"、"不用就落后"表述，流行度替代适配性分析
Step 2: 评估从众影响 → 产出：K8s流行≠适合我们，需评估团队规模、业务复杂度
Step 3: 分析从众机制 → 产出：信息级联：他人选择被视为信号；社会证明：多数人选的=正确的；FOMO：害怕落后
Step 4: 校正从众效应 → 产出：建议独立评估需求，分析K8s是否过度设计
```

**输出**：
```
## 分析结论
### 核心发现
1. "其他公司都在用"是流行度信号，非适配性依据
2. 我们团队5人、3个微服务，K8s运维成本>收益
3. 采用K8s的公司平均团队规模50+，微服务20+

### 建议（按优先级）
1. [高优先级] 评估当前需求：Docker Compose已满足
2. [高优先级] 设定K8s触发条件：团队>20人或微服务>10个
3. [中优先级] 关注轻量替代方案（如Nomad）

### 关键假设
- 团队规模和业务复杂度短期不变
- K8s运维需专职人员

### 风险
- 过早引入K8s增加运维负担
- 落后焦虑可能导致过度设计
```

## 示例 2：产品功能规划中的从众

**输入**：竞品都做了AI功能，我们也必须做，不然用户会觉得我们不行。

```
Step 1: 识别从众信号 → 产出：发现"竞品都做了"、"必须做"表述，竞争压力替代用户需求分析
Step 2: 评估从众影响 → 产出：竞品做AI≠用户需要AI，需验证真实需求
Step 3: 分析从众机制 → 产出：竞争焦虑：害怕被比下去；社会证明：竞品选择=市场趋势
Step 4: 校正从众效应 → 产出：建议调研用户真实需求，评估AI功能ROI
```

**输出**：
```
## 分析结论
### 核心发现
1. "竞品都做了"是竞争信号，非用户需求证据
2. 用户调研显示：Top 3需求是稳定性、速度、价格，AI不在前10
3. 竞品AI功能使用率平均<5%，多数为营销噱头

### 建议（按优先级）
1. [高优先级] 优先满足用户Top 3需求（稳定性、速度、价格）
2. [高优先级] 如做AI，聚焦解决用户真实痛点的场景
3. [中优先级] 小范围MVP验证AI功能需求

### 关键假设
- 用户调研样本足够代表性
- 竞品AI使用率数据可靠

### 风险
- 完全不做AI可能影响品牌感知
- 需要平衡用户真实需求和市场趋势
```
