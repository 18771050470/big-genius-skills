# 技术交付物模板

## 交付物模板 1：代码审查反馈

```markdown
# 代码审查反馈：PR #[编号]

## 审查信息
- **审查人**：[姓名]
- **审查时间**：[日期]
- **变更范围**：[文件列表]

## 总体评价
- **质量等级**：[优秀/良好/需改进]
- **是否可合并**：[是/需修改后/需讨论]

## 详细反馈

### 🔴 阻塞问题（必须修复）

#### 问题 1：[标题]
- **位置**：[文件:行号]
- **问题**：[描述]
- **风险**：[说明]
- **建议**：[具体修改方案]
- **参考**：[链接/规范]

### 🟡 建议改进（强烈建议）

#### 建议 1：[标题]
- **位置**：[文件:行号]
- **建议**：[描述]
- **原因**：[为什么]

### 💭 可选优化（供参考）

#### 优化 1：[标题]
- **位置**：[文件:行号]
- **建议**：[描述]

### 👍 做得好的地方
- [具体表扬]
```

## 交付物模板 2：代码审查检查清单

```markdown
# 代码审查检查清单

## 正确性
- [ ] 代码是否正确实现了需求
- [ ] 边界条件是否处理（空值、空集合、最大值）
- [ ] 错误处理是否完善（异常捕获、资源释放）
- [ ] 并发场景是否安全（竞态条件、死锁）

## 质量
- [ ] 命名是否清晰（变量/函数/类名）
- [ ] 函数是否单一职责（长度 < 50 行）
- [ ] 是否有重复代码（DRY 原则）
- [ ] 圈复杂度是否过高（< 10）

## 测试
- [ ] 是否有单元测试
- [ ] 测试是否覆盖核心逻辑
- [ ] 测试是否覆盖边界条件
- [ ] 测试是否独立可重复

## 安全
- [ ] 用户输入是否验证和消毒
- [ ] 是否有 SQL 注入/XSS 风险
- [ ] 是否有敏感信息泄露
- [ ] 权限控制是否正确

## 性能
- [ ] 是否有 N+1 查询
- [ ] 是否有大数据量处理问题
- [ ] 是否有内存泄漏风险
- [ ] 算法复杂度是否合理
```
