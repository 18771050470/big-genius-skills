# 常见陷阱

## 陷阱 1：停留在表面原因

**问题**：追问1-2层就停止，未找到根本原因。

❌ **Before（错误做法）**
```markdown
## 分析结论
系统宕机是因为数据库CPU 100%。
建议：重启数据库。
```
*问题：只找到直接原因，未找到系统性根因*

✅ **After（正确做法）**
```markdown
## 分析结论
### 因果链深度
| 层级 | 原因 | 类型 |
|------|------|------|
| 1 | CPU 100% | 症状 |
| 2 | 慢查询 | 直接原因 |
| 3 | 缺少索引 | 技术原因 |
| 4 | 未做审查 | 流程原因 |
| 5 | 无变更流程 | 根本原因(系统) |

### 建议
1. 建立数据库变更检查流程
2. 添加索引
```
*优势：追溯到系统性根因*

**原因**：Ohno(1988)：5Whys的核心是找到系统性原因，而非技术原因。修复系统比修复技术问题更持久。

## 陷阱 2：归咎于个人

**问题**：将根因归结为个人失误。

❌ **Before（错误做法）**
```markdown
## 分析结论
系统宕机是因为张三忘了加索引。
建议：批评张三。
```
*问题：个人原因是表面原因，系统原因才是根本*

✅ **After（正确做法）**
```markdown
## 分析结论
### 系统vs个人
| 归因 | 类型 | 效果 |
|------|------|------|
| 张三忘了 | 个人 | 临时(下次别人也会忘) |
| 无审查流程 | 系统 | 持久(流程防止任何人忘记) |

### 建议
1. 建立变更审查流程
2. 不指责个人
```
*优势：系统修复替代个人指责*

**原因**：Deming：94%的问题来自系统，6%来自个人。指责个人不解决系统问题。

## 陷阱 3：因果链断裂

**问题**：因果关系不成立，跳跃推理。

❌ **Before（错误做法）**
```markdown
## 分析结论
为什么宕机？因为代码质量差。
为什么代码质量差？因为团队能力不足。
```
*问题：因果链跳跃，缺乏证据*

✅ **After（正确做法）**
```markdown
## 分析结论
### 因果验证
| 因果链 | 证据 | 成立？ |
|--------|------|--------|
| 宕机→CPU 100% | 监控数据 | 是 |
| CPU 100%→慢查询 | 慢查询日志 | 是 |
| 慢查询→缺少索引 | EXPLAIN分析 | 是 |

### 建议
1. 每层因果必须有证据
2. 不跳跃推理
```
*优势：证据链支撑因果推理*

**原因**：Andersen & Fagerhaug(2006)：每个"为什么"的答案必须有数据或证据支持，否则是猜测而非分析。
