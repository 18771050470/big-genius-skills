# 防御性编程思维（Defensive Programming Thinking）

## 核心定义

假设所有外部输入都是恶意的、所有依赖都可能失效，通过边界验证、错误隔离和优雅降级来构建即使在异常环境下也能保持核心功能稳定的系统。

---

## 思考流程控制

### 中断处理

- **信息不足**：用 AskUserQuestion 向用户确认安全等级要求（金融级/企业级/内部工具）
- **发现矛盾**：停止思考，用 AskUserQuestion 向用户报告矛盾点（如"要求高性能但要求全面参数校验"），等待决策
- **提前终止**：若系统为纯内部工具且无外网暴露，可提前进入轻量级防御策略

## 思考流程

### Step 1: 威胁面识别

- 识别所有外部边界：用户输入、API请求、文件读取、网络响应、数据库返回、第三方服务回调
- 标注每个边界的信任等级：不可信（用户输入）→ 半可信（内部服务）→ 可信（本进程内存）
- 识别内部边界：模块间调用、函数参数、返回值
- 陷阱：只关注用户输入，忽视内部模块间的契约违反

### Step 2: 输入验证策略设计

- 对每个外部边界设计验证规则：类型检查、范围检查、格式检查、长度限制、白名单过滤
- 选择验证时机：尽早验证（入口点）vs 延迟验证（使用前）
- 确定验证严格度：拒绝（抛异常）vs 清洗（自动修正）vs 记录（告警但继续）
- 陷阱：使用黑名单（试图拦截所有恶意模式）而非白名单（只允许已知安全模式）

### Step 3: 错误处理与隔离机制

- 设计错误处理策略：快速失败（立即抛异常）vs 优雅降级（返回默认值/缓存）vs 重试（指数退避）
- 确定错误传播边界：哪些错误应该向上传播，哪些应该在模块内消化
- 设计资源清理机制：确保异常发生时资源（连接、锁、临时文件）被正确释放
- 陷阱：捕获所有异常后静默处理（吞掉错误），导致问题难以定位

### Step 4: 状态一致性保障

- 识别关键状态变更操作：数据库写入、文件修改、配置更新
- 设计事务或补偿机制：确保部分失败时能回滚或补偿
- 考虑并发场景：竞态条件、死锁、资源争用
- 陷阱：只考虑单线程场景，忽视并发下的状态不一致

### Step 5: 监控与审计

- 设计日志策略：记录什么（输入参数、关键决策、异常详情）、记录级别（DEBUG/INFO/WARN/ERROR）
- 确定告警阈值：错误率、响应时间、资源使用率
- 设计审计追踪：关键操作的可追溯性
- 陷阱：日志过多导致性能下降，或日志过少导致无法定位问题

### Step 6: 总结输出

- 整合前面各步的分析结果，形成完整防御策略
- 不遗漏关键边界、验证规则和错误处理策略
- 明确下一步应该调用哪个 Skill（如需要安全编码细节则调用 security-coding）
- 输出应包含：威胁面清单 + 验证规则 + 错误处理策略 + 监控方案 + 风险 + 下一步

## 关键原则

1. **不信任任何外部输入**：所有来自系统外部的数据都必须验证。原因：攻击者会故意构造恶意输入，正常用户也会误操作。
2. **快速失败，大声失败**：错误应该尽早暴露，而不是静默传播。原因：静默错误会在系统中扩散，导致更严重的级联故障，且难以定位根因。
3. **最小权限原则**：代码只应拥有完成其功能所必需的最小权限。原因：权限过大意味着被攻击后的影响面更大。
4. **防御层深度**：单一防御点可能被绕过，多层防御（输入验证 + 参数化查询 + 最小权限）提供冗余保护。原因：没有完美的防御，深度防御增加攻击成本。
5. **优雅降级优于完全崩溃**：当部分功能不可用时，核心功能应继续服务。原因：用户更接受功能受限，而非完全不可用。

## 自检清单

**执行规则**：
- **执行方**：Agent 自检，在最终输出前逐项确认
- **未通过处理**：任一检查项未通过，必须修正后重新自检，禁止跳过
- **记录方式**：自检结果写入 `self_check_result.md`（临时文件），包含：通过项、未通过项、修正记录
- **强制停止项**：以下检查项未通过时，必须停止输出并修正：
  - [ ] 已按思考流程完成全部步骤，无跳过（或已标注提前终止原因）
  - [ ] 输出具体可执行，不含模糊建议
  - [ ] 标注了关键假设和不确定性

**完整检查项**：

- [ ] 已按思考流程完成全部步骤，无跳过（或已标注提前终止原因）
- [ ] 每步思考要点已覆盖，无遗漏
- [ ] 每步结论标注了置信度和反证可能性
- [ ] 考虑了二阶/三阶效应（如果适用）
- [ ] 输出具体可执行，不含模糊建议（如"优化一下"）
- [ ] 标注了关键假设和不确定性
- [ ] 识别了最大风险点和缓解措施
- [ ] 建议按优先级排序，不是平铺罗列
- [ ] 明确了下一步应该调用哪个 Skill（如果需要）

**防御性编程特有关键检查项**：

- [ ] **防御性编程特供**：是否识别了所有威胁面？外部边界（用户输入、API、文件、网络、数据库、第三方服务）是否被逐一标注信任等级？
- [ ] **防御性编程特供**：是否设计了输入验证规则？每个外部边界是否有明确的类型检查、范围检查、格式检查、长度限制？是否使用白名单而非黑名单？
- [ ] **防御性编程特供**：是否避免了静默失败？所有异常是否被正确处理（抛异常、记录日志、告警），而非吞掉？是否设计了错误传播边界？
- [ ] **防御性编程特供**：是否保障了状态一致性？关键状态变更是否有事务或补偿机制？并发场景（竞态条件、死锁）是否被考虑？
- [ ] **防御性编程特供**：是否设计了优雅降级策略？当部分功能不可用时，核心功能是否能继续服务？是否有熔断、限流、回退机制？
- [ ] **防御性编程特供**：是否遵循了最小权限原则？代码是否只拥有完成功能所必需的最小权限？数据库连接、文件访问、网络请求是否被限制？
