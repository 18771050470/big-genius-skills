# Gotchas

> 只写 Agent 容易犯错的**非显而易见的事实**。通用知识不要写。
> 每个 Gotcha 必须包含：问题描述 + ❌ Before（错误做法） + ✅ After（正确做法） + 原因解释。

## Gotcha 1：灰色文字滥用导致可读性灾难

**问题**：为了视觉"简洁"大量使用浅灰色文字，导致用户阅读困难，尤其是视力不佳或在强光环境下。

❌ **Before（错误做法）**
```css
/* 正文、标签、提示全部用浅灰 */
.body-text { color: #999999; }  /* 对比度仅 2.8:1 */
.label { color: #AAAAAA; }
.hint { color: #BBBBBB; }
```

✅ **After（正确做法）**
```css
/* 建立明确的颜色层级 */
.primary-text { color: #1A1A1A; }    /* 主文本，对比度 16:1 */
.secondary-text { color: #666666; }   /* 次要文本，对比度 5.7:1 */
.disabled-text { color: #999999; }    /* 禁用状态，对比度 2.8:1（仅用于禁用） */

/* 使用场景规范 */
/* 正文 ≥ 14px：#333 或更深 */
/* 辅助说明 ≥ 12px：#666 或更深 */
/* 仅禁用/占位可用 #999 */
```

**原因**：WCAG AA 要求正文对比度 ≥ 4.5:1。浅灰文字（#999）对比度仅 2.8:1，不满足标准。应该建立颜色层级系统，主文本用深色，次要文本用中灰，浅灰仅用于禁用状态。

## Gotcha 2：按钮层次混乱导致用户行动瘫痪

**问题**：页面上多个按钮使用相同的视觉权重，用户无法识别主要行动点，导致决策困难。

❌ **Before（错误做法）**
```
[保存] [取消] [导出] [分享]

/* 所有按钮样式相同：蓝底白字，同等大小 */
/* 用户不知道哪个是主要操作 */
```

✅ **After（正确做法）**
```
[保存]    ← 主按钮：实心主色，视觉最重
[取消]    ← 次按钮：描边或文字按钮，视觉较轻
[导出] [分享]  ← 三级操作：图标按钮或文字链接，视觉最轻

/* 按钮层级规范 */
/* 主按钮：每页最多 1 个，使用 Primary Color */
/* 次按钮：取消/返回等，使用 Outline 或 Ghost 样式 */
/* 三级操作：辅助功能，使用 Text 或 Icon 样式 */
```

**原因**：视觉层级引导用户注意力。如果所有按钮同等重要，用户需要额外认知负担来判断。应该每页只有一个主按钮，其他操作降低视觉权重。

## Gotcha 3：表单标签对齐方式引发填写效率问题

**问题**：表单标签使用左对齐，导致标签与输入框距离不固定，用户视线跳跃，填写效率低下。

❌ **Before（错误做法）**
```
用户名     [________________]
电子邮箱   [________________]
手机号码   [________________]
详细地址   [________________]

/* 左对齐：标签长度不同，输入框起始位置参差不齐 */
/* 视线需要左右跳跃，扫描困难 */
```

✅ **After（正确做法）**
```
/* 方案A：顶对齐（适合长表单，移动端优先） */
用户名
[________________]

电子邮箱
[________________]

/* 方案B：右对齐（适合短表单，桌面端） */
  用户名 [________________]
电子邮箱 [________________]

/* 方案C：浮动标签（节省空间，适合紧凑布局） */
[________________]
用户名
```

**原因**：顶对齐表单填写速度最快（视线只需单向移动），右对齐次之，左对齐最慢。移动端优先顶对齐，桌面端短表单可用右对齐，空间紧张可用浮动标签。

## Gotcha 4：图标单独使用导致语义不明

**问题**：为了界面简洁只使用图标按钮，没有文字说明，导致用户无法理解功能含义，尤其是非常用功能。

❌ **Before（错误做法）**
```
[🔍] [🔔] [⚙️] [⋮]

/* 只有图标，没有文字 */
/* 心形图标：收藏？点赞？关注？ */
/* 旗帜图标：举报？标记位置？目标达成？ */
```

✅ **After（正确做法）**
```
/* 方案A：图标 + 文字（推荐） */
[🔍 搜索] [🔔 通知] [⚙️ 设置]

/* 方案B：图标 + Tooltip（空间紧张时） */
[🔍]  ← 鼠标悬停显示 "搜索"

/* 方案C：使用通用认知图标（仅限极度常见的操作） */
[＋] 新建    [✕] 关闭    [←] 返回

/* 图标使用规范 */
/* 必须文字：业务功能、非常用操作 */
/* 可纯图标：搜索、关闭、返回、菜单、用户（需用户测试确认） */
```

**原因**：图标的认知具有文化差异和主观性。研究显示，单独图标的识别率通常 < 80%。文字 + 图标的组合识别率 > 95%。只有经过广泛用户测试验证的通用图标（如搜索、关闭）才可以单独使用。

## Gotcha 5：加载状态设计缺失导致用户焦虑

**问题**：异步操作没有加载反馈，用户不知道系统是否在处理，导致重复点击或认为系统卡死。

❌ **Before（错误做法）**
```
用户点击"提交"按钮 → 按钮无变化 → 2 秒后跳转成功页

/* 用户在这 2 秒内： */
/* - 不知道是否点击成功 */
/* - 可能重复点击 */
/* - 可能认为页面卡死而刷新 */
```

✅ **After（正确做法）**
```
/* 加载状态设计 */
用户点击"提交"按钮 → 
按钮变为加载态：[⟳ 提交中...]（禁用重复点击）→ 
显示进度（如适用）：[████████░░ 80%] → 
操作完成：[✓ 提交成功]（2 秒后恢复）或跳转

/* 加载状态规范 */
/* 0-1 秒：按钮加载态（足够） */
/* 1-3 秒：按钮 + 局部加载指示器 */
/* 3 秒+：全局加载 + 预计剩余时间 + 取消选项 */
/* 10 秒+：后台处理 + 通知提醒，不阻塞用户 */
```

**原因**：用户需要系统状态的可见性（Visibility of System Status，Nielsen 启发式原则 #1）。没有加载反馈会导致用户焦虑、重复操作、甚至放弃任务。加载状态应该按等待时长分级设计。
