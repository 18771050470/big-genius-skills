# 数据工程师常见陷阱

## 1. 数据质量问题

### 反例：无数据校验
- **错误**：数据管道不做质量检查，脏数据直接流入数仓
- **案例**：用户年龄字段出现负数、空值、异常大值，导致报表错误
- **原因**：忽视了数据质量的重要性

### 正例：质量门禁
- **正确**：每个管道步骤都有数据质量检查，不合格数据拦截或标记
- **案例**：年龄字段检查范围（0-150），空值率不超过 5%，异常值告警
- **要点**：数据质量是数据管道的生命线

## 2. 数据倾斜

### 反例：忽视数据分布
- **错误**：大数据处理时不考虑数据分布，导致部分任务极慢
- **案例**：按用户 ID 分区，但头部用户数据量占 80%，某些分区过大
- **原因**：没有分析数据分布特征

### 正例：处理倾斜
- **正确**：识别倾斜键，使用加盐、拆分等策略处理
- **案例**：对热点 key 加随机前缀打散，处理后再合并结果
- **要点**：数据倾斜是大数据处理的常见问题，需要主动处理

## 3. 小文件问题

### 反例：大量小文件
- **错误**：数据管道生成大量小文件，影响 HDFS/S3 性能
- **案例**：每小时生成一个文件，一年后目录下有 8760 个小文件，LIST 操作极慢
- **原因**：忽视了存储系统的文件数量限制

### 正例：文件合并
- **正确**：定期合并小文件，控制文件大小（128MB-1GB）
- **案例**：每日合并前一天的小文件，保持文件大小在 256MB 左右
- **要点**：小文件影响存储和计算性能，需要定期合并

## 4. 无版本控制

### 反例：SQL 散落各处
- **错误**：ETL 脚本没有版本控制，修改后无法回溯
- **案例**：生产环境 SQL 被修改，出现问题后不知道改了什么
- **原因**：没有将数据转换代码纳入版本控制

### 正例：代码化管理
- **正确**：所有数据转换代码使用 Git 管理，支持 CI/CD
- **案例**：dbt 模型代码在 Git 中管理，PR 评审后自动部署
- **要点**：数据转换即代码，需要版本控制和评审

## 5. 忽视数据血缘

### 反例：数据黑盒
- **错误**：不知道数据从哪来到哪去，出现问题无法定位
- **案例**：报表数据错误，花 3 天追溯才找到是上游某个字段映射错误
- **原因**：没有建立数据血缘追踪机制

### 正例：血缘追踪
- **正确**：建立数据血缘图谱，支持上下游追溯
- **案例**：使用 Apache Atlas 或 dbt 文档自动生成血缘，出现问题 10 分钟定位
- **要点**：数据血缘是数据治理的基础，也是问题定位的利器
