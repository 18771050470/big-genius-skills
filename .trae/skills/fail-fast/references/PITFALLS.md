# 常见陷阱

## 陷阱 1：快速失败无恢复

**问题**：失败后没有恢复路径，系统直接崩溃。

❌ **Before（错误做法）**
```markdown
## 分析结论
支付 API 失败立即抛出异常。
建议：不处理，让系统崩溃。
```
*问题：快速失败不是目的，快速恢复才是*

✅ **After（正确做法）**
```markdown
## 分析结论
### 失败恢复路径
| 失败类型 | 恢复策略 |
|---------|---------|
| 超时 | 重试 1 次，仍失败则降级 |
| 服务不可用 | 切换备用支付渠道 |
| 数据错误 | 记录日志，人工介入 |

### 建议
1. 快速失败 + 明确错误
2. 设计降级/重试/人工介入路径
```
*优势：失败后可恢复*

**原因**：Nygard(2018)：快速失败必须配合快速恢复。没有恢复路径的快速失败只是崩溃的另一种说法。

## 陷阱 2：静默失败

**问题**：失败被吞掉，问题无法被发现。

❌ **Before（错误做法）**
```markdown
## 分析结论
catch (Exception e) { /* 忽略 */ }
建议：不影响主流程就行。
```
*问题：静默失败无法学习和修复*

✅ **After（正确做法）**
```markdown
## 分析结论
### 失败处理
| 要求 | 实现 |
|------|------|
| 快速失败 | 立即返回错误 |
| 明确信号 | 错误码 + 日志 |
| 可追踪 | 请求 ID + 时间戳 |

### 建议
1. 禁止空 catch 块
2. 所有失败必须记录日志
```
*优势：失败可追踪可学习*

**原因**：Hunt & Thomas(1999)：静默失败是软件设计的反模式。失败必须有明确信号才能被修复。

## 陷阱 3：失败成本误判

**问题**：低估快速失败的成本，高估延迟失败的成本。

❌ **Before（错误做法）**
```markdown
## 分析结论
快速失败会让用户看到错误，体验差。
建议：尽量隐藏错误，慢慢处理。
```
*问题：延迟失败的成本通常远高于快速失败*

✅ **After（正确做法）**
```markdown
## 分析结论
### 成本对比
| 策略 | 成本 | 影响 |
|------|------|------|
| 快速失败 | 用户看到错误提示 | 可重试，影响小 |
| 延迟失败 | 订单不一致、资金损失 | 恢复困难，影响大 |

### 建议
1. 快速失败成本通常更低
2. 用户体验可通过明确错误信息改善
```
*优势：正确评估失败成本*

**原因**：Ries(2011)：快速失败的成本通常被高估，延迟失败的成本通常被低估。尽早暴露问题是最经济的策略。
