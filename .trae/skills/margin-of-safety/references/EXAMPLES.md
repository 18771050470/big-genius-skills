# 分析示例

## 示例 1：数据库容量规划

**输入**：数据库当前使用率70%，何时需要扩容？

```
Step 1: 评估预期最坏情况
→ 风险场景：流量突增3倍（促销活动）、数据量超预期增长、主从延迟导致查询堆积
→ 合理最坏情况：流量2倍增长（基于历史促销数据）
→ 发生概率：30%（每年有大促）

Step 2: 计算基础容量
→ 当前最大QPS：10000
→ 实际可用QPS：7000（70%使用率）
→ 理论最大QPS：10000，但超过80%后性能急剧下降
→ 实际安全容量：8000 QPS

Step 3: 确定安全边际系数
→ 风险容忍度：中（影响用户体验但不危及安全）
→ 系数选择：2倍
→ 需要容量：2倍 × 8000 = 16000 QPS

Step 4: 设计缓冲机制
→ 容量缓冲：提前扩容到16000 QPS（当前10000，需增加60%）
→ 时间缓冲：在大促前2周完成扩容（不是前一天）
→ 降级方案：流量超过12000时启用读缓存和限流
→ 成本：扩容成本$5K/月 vs 宕机损失$50K/小时
```

```
## 分析结论

### 核心发现
1. 当前容量无法承受2倍流量，缺口60%
2. 安全边际要求16000 QPS，当前仅10000
3. 扩容成本远低于宕机损失

### 建议（按优先级）
1. [高] 提前2周扩容到16000 QPS，成本$5K/月
2. [高] 部署读缓存层，可临时承载额外30%流量
3. [中] 配置限流规则，保护数据库不被打垮
```
