# 常见陷阱

## 陷阱 1：无限质疑导致分析瘫痪

**反例**：
```
Step 1: 识别假设
→ 假设1："需要数据库"（质疑：为什么不用文件系统？）
→ 假设2："需要服务器"（质疑：为什么不用无服务器？）
→ 假设3："需要代码"（质疑：为什么不用配置驱动？）
→ ...无限质疑下去，永远无法进入Step 2
```

**正例**：
```
Step 1: 识别假设
→ 假设1："需要分库分表"（质疑：慢的根本原因是什么？）
→ 停止在影响最大的假设，不质疑基础架构选择
```

**原因**：第一性原理不是怀疑一切，而是质疑影响最大的假设。基础架构选择通常已被验证，质疑它们浪费精力。

---

## 陷阱 2：把个人信念当作基本真理

**反例**：
```
Step 2: 基本真理
→ "用户喜欢简洁设计"（这是个人偏好，不是基本真理）
→ "微服务比单体好"（这是观点，不是真理）
```

**正例**：
```
Step 2: 基本真理
→ "查询性能 = 索引命中率 × 数据扫描量"（可测量的工程规律）
→ "80%用户只使用20%功能"（有数据支撑的行为模式）
```

**原因**：基本真理必须有实证支撑。个人信念会导致重建方案建立在沙子上。

---

## 陷阱 3：重建时偷偷参考现有方案

**反例**：
```
Step 3: 重建方案
→ "在现有架构基础上优化..."（这已经是参考现有方案了）
→ "改进当前的微服务拆分方式..."（不是从零开始）
```

**正例**：
```
Step 3: 重建方案
→ "如果我是第一个解决这个问题的人会怎么做？"
→ "抛开现有架构，从用户需求出发重新设计..."
```

**原因**：一旦参考现有方案，就会被其约束限制，失去从零创新的机会。

---

## 陷阱 4：忽略重建方案的实现难度

**反例**：
```
Step 4: 验证
→ "理论上可行"（但没有评估实现难度）
→ "从第一性原理推导出的方案一定最优"（忽略了迁移成本）
```

**正例**：
```
Step 4: 验证
→ "理论最优，但迁移成本$500K，需6个月"
→ "次优方案迁移成本$50K，需1周，ROI更高"
```

**原因**：第一性原理不保证实现容易。理论最优方案可能因实现难度过高而不实用。
