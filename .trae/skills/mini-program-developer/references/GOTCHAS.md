# 小程序工程师常见陷阱

## 1. 包体积超标

### 反例：资源打包进项目
- **错误**：将图片、字体等资源打包进小程序项目
- **案例**：小程序包体积 3MB，超出 2MB 限制，无法上传
- **原因**：忽视了小程序的包体积限制

### 正例：资源分离
- **正确**：图片使用 CDN，字体使用在线字体，代码按需加载
- **案例**：主包 1.2MB，图片全部走 CDN，分包加载非首屏页面
- **要点**：主包 < 2MB 是硬性要求

## 2. 忽视平台差异

### 反例：一套代码不处理差异
- **错误**：使用跨端框架但不处理平台差异，导致某平台报错
- **案例**：使用 uni-app 开发，支付宝小程序上 API 报错，因为使用了微信特有 API
- **原因**：没有使用条件编译或运行时适配

### 正例：平台适配
- **正确**：使用条件编译处理平台特定代码，抽象统一接口
- **案例**：支付功能使用条件编译，微信调用 wx.requestPayment，支付宝调用 my.tradePay
- **要点**：跨端不是无差异，是管理好差异

## 3. setData 滥用

### 反例：频繁调用 setData
- **错误**：在循环中频繁调用 setData，导致性能问题
- **案例**：列表滚动时每次滚动都 setData，页面卡顿
- **原因**：不理解小程序的双线程模型和 setData 成本

### 正例：批量更新
- **正确**：合并数据更新，减少 setData 调用次数
- **案例**：将多次数据变更合并为一次 setData，使用节流控制滚动事件
- **要点**：setData 是跨线程通信，有性能开销

## 4. 审核被拒

### 反例：忽视审核规范
- **错误**：开发时不考虑审核要求，提交后频繁被拒
- **案例**：小程序包含虚拟支付，但未使用微信支付，被审核拒绝
- **原因**：没有提前了解平台审核规范

### 正例：合规开发
- **正确**：开发前了解审核规范，设计时考虑合规要求
- **案例**：虚拟支付使用微信支付，内容符合规范，一次性通过审核
- **要点**：审核规范是开发的边界条件

## 5. 用户体验差

### 反例：加载白屏
- **错误**：页面加载时没有 loading 状态，用户看到白屏
- **案例**：网络慢时页面空白 3 秒，用户以为应用卡死
- **原因**：忽视了弱网环境下的用户体验

### 正例：优雅加载
- **正确**：设计 skeleton 屏、loading 动画、错误重试机制
- **案例**：页面加载时显示 skeleton，网络错误时显示重试按钮
- **要点**：用户体验是小程序留存的关键
