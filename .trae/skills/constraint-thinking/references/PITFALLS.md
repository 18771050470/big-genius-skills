# 约束思维 - 常见陷阱

## 陷阱 1：优化非瓶颈环节

**问题**：精力花在提升"不限制产出"的环节上，系统总产出没有任何提升。

❌ **Before（错误做法）**
```
系统分析：
- 瓶颈是API响应慢（数据库全表扫描，查询耗时3s）
- 团队花2周优化了前端渲染（从200ms优化到50ms）
→ 总响应时间从3.2s变成3.05s，用户感知无变化
→ 瓶颈仍在数据库，优化前端是浪费
```

✅ **After（正确做法）**
```
TOC第一步：识别瓶颈是数据库查询
TOC第二步：为查询加索引 → 查询从3s降到50ms
TOC第三步：然后才考虑前端优化
→ 总响应从3.2s降到250ms，用户感知巨大
```

**原因**：Goldratt的核心洞察——非瓶颈环节的改善对系统总产出没有帮助，甚至可能有害（增加在制品库存、分散资源）。"瓶颈的每一分钟就是系统的每一分钟"。

## 陷阱 2：把习惯当约束

**问题**：接受自设的限制当作真约束，不去质疑。

❌ **Before（错误做法）**
```
"我们这个项目必须用微服务架构。"
"为什么？"
"因为这是公司标准。"
→ 一个内部工具，2个开发，日活<50人
→ 微服务架构的运维复杂度远超单体，纯属自缚手脚
```

✅ **After（正确做法）**
```
先区分真假约束：
- 真约束：deadline 3个月、预算20万、团队2人
- 假约束："必须用微服务" → 这是习惯/跟风，不是真约束
  → 日活50人的内部工具，单体架构足够了
  → 节省的运维时间可以用来多做一个功能
```

**原因**：很多"约束"其实是"惯性"或"跟风"。约束思维的第一步就是区分硬约束（物理/合规/合同限制）和"我们一直这么做"的假约束。

## 陷阱 3：同时优化多个瓶颈

**问题**：试图同时解决多个瓶颈，资源分散，一个都优化不好。

❌ **Before（错误做法）**
```
"我们发现系统有三个瓶颈：数据库慢、网络带宽不足、缓存命中率低。
我们同时优化这三个吧！"
→ 团队分成三组，各自优化 → 三周后三个都只优化了一半
→ 系统的最大约束一个都没真正突破
```

✅ **After（正确做法）**
```
TOC要求：一次只聚焦一个约束
1. 先解决最紧的约束（数据库慢）→ 系统产出提升
2. 让系统运行稳定，观察新瓶颈出现位置（可能是网络带宽）
3. 再解决新瓶颈（网络带宽）
按顺序逐个击破，不并行
```

**原因**：TOC第五步（回到第一步）明确要求"不要让惯性成为约束"。突破一个瓶颈后，系统会呈现新的状态，必须在新的状态下重新识别瓶颈——并行优化多个约束会导致"优化任何一步都在变动的地面上跑步"。
