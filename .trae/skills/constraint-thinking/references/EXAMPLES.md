## 示例 1：小型团队交付紧急项目

**输入**：一个5人团队被要求3周内交付一个内部数据看板系统。功能需求包括：数据接入、ETL处理、可视化图表、权限管理、导出报表。团队只有2个后端 + 1个前端 + 1个数据 + 1个PM。

```
Step 1: 识别和分类所有约束
→ 产出：
  - 硬约束：3周deadline（CEO明确要求，不可延期）、5人团队（无法加人）
  - 软约束：所有数据源必须接入（可以分批）、权限管理（可以用简单方案）
  - 假约束："必须用React" → PM提的偏好，非必需
  - 假约束："需要支持所有图表类型" → 实际核心用户只需要柱状图和折线图

Step 2: 识别核心瓶颈
→ 产出：
  - 瓶颈判断：数据接入+ETL（需要对接5个数据源，预估2周，是整条链路最慢环节）
  - 瓶颈证据：数据同事排期已经3周，后端开发只需1周

Step 3: 挖掘瓶颈最大潜力
→ 产出：
  - 瓶颈浪费：数据同事每天有2小时在开会（PM保护，减少到15分钟standup）
  - 瓶颈优化：5个数据源中3个有现成API，1个需要写爬虫，1个需要人工导出
    → 先接入3个有API的（1周），爬虫的放在第2周，人工导出的暂缓
  - 非瓶颈环节不要去"帮"数据同事 → 后端先把已接入数据的处理逻辑写好

Step 4: 非瓶颈从属于瓶颈
→ 产出：
  - 前端节奏跟数据走：数据接入一个源 → 前端做一个图表，不等全部接入
  - 后端的权限管理和导出功能排到第3周（数据接入完成后）
  - "鼓-缓冲-绳"：数据接入是鼓点，后端处理逻辑保持1天缓冲，前端消费保持1天缓冲

Step 5: 突破瓶颈
→ 产出：
  - 数据同事实在忙不过来 → 后端同事接手下最简单的那个数据源（JSON API，半天搞定）
  - 爬虫数据源简化要求：不做实时同步，每天定时拉取一次

Step 6: 约束转化为创新
→ 产出：
  - 时间约束驱使我们采用"流式交付"：接入一个数据源 → 立即上线一个图表
  - 用户反馈：这种方式反而更好用，不用等3周后一次性看一堆图表
  - 能力约束驱使我们砍掉了"花哨图表"——用户反而觉得简洁清晰
```

**输出**：

```
## 分析结论

### 核心瓶颈
数据接入+ETL是整个系统的瓶颈，所有节奏围绕它展开。

### 最大化方案
1. 消除瓶颈浪费：减少数据同事的会议时间
2. 分批接入：3个易接的先上，2个难接的后上或简化
3. 从属节奏：前端跟数据走，不超前也不落后

### 创新方案
- 流式交付：接入一个上一个，而非一次性交付
- 精简图表：只做用户真正需要的柱状图和折线图

### 关键假设
- 3个数据源的API稳定且文档齐全
- PM能成功保护数据同事的专注时间

### 风险
- 人工导出的那个数据源如果被CEO认为必须包含 → 需第3周加班或延期1个功能
```

## 示例 2：短小精悍的算法题（约束驱动的优雅）

**输入**：实现一个函数，判断字符串中的括号是否有效配对。约束：O(n)时间 + O(1)额外空间（不能另开栈）。

```
Step 1: 约束识别
→ 产出：
  - O(n)时间：必须一次遍历
  - O(1)空间：不能用栈（栈是O(n)空间）
  - 假约束："必须处理所有类型括号" → 实际需求只需要处理 ()

Step 2: 瓶颈识别
→ 约束本身就是瓶颈：O(1)空间不能另开数据结构

Step 3: 挖掘最大潜力
→ 思考：如果没有空间约束，怎么做？→ 用栈（计数器）
  → 有空间约束，能否用已有空间？→ 可以遍历时用两个指针"就地"操作

→ 既然不能用额外空间，那就在原输入上操作：
  但是字符串不可变 → 改用计数器模拟栈的深度
```

**输出**：

```
## 分析结论
O(1)空间约束催生了"计数器"方案：
- 用一个变量depth代替整个栈，depth表示"当前有多少个未配对的左括号"
- 遇到 '(' → depth++；遇到 ')' → 如果depth==0则非法，否则depth--
- 最后检查depth==0 → 配对成功
- 本质：用单个整数代替了栈的全部功能（因为括号只有一种类型，不需要记录类型）

约束思维的价值：O(1)空间限制让解法从"通用栈"退化为"专用计数器"——反而更简单高效。
```

## 反例：不该触发约束思维的场景

**输入**："这个需求要加一个导出Excel功能，但Excel库300KB太大了，能不能自己写一个？"

**原因**：300KB的包体积不是真正的瓶颈——用户下载的页面总大小已经是2MB。这是一个"假约束"（完美主义倾向），不是需要TOC分析的真实瓶颈。直接用现成库即可。
