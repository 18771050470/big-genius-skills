# 批判性思维 – 分析示例

## 示例 1：评估商业趋势文章

### 场景
阅读一篇博客文章，标题为"人工智能将取代所有程序员，三年内实现"。作者声称：
- "StackOverflow 上关于低代码问题的数量增长了 300%"
- "Google 工程师透露内部测试表明 AI 可以自动完成 90% 的代码"
- "咨询公司预测到 2027 年程序员数量将减少 80%"

需要判断该主张是否可信。

### Step 1：明确议题和立场
- **议题**："人工智能将在三年内取代所有程序员"这一主张是否成立。
- **核心概念**："取代所有程序员"需要界定——是完全替代所有编程岗位，还是大幅度减少常规编码工作？
- **输出**：`清晰议题` = "到 2027 年，AI 将导致程序员就业人数减少 80% 以上，且绝大多数编程任务由 AI 自动完成。"

### Step 2：识别假设和前提
- 假设 1："低代码问题数量增长 300%"意味着开发者在放弃传统编码，转向低代码工具。
- 假设 2：Google 工程师的"内部测试"代表业界平均水平，且测试结果可推广。
- 假设 3：咨询公司的预测模型准确且无利益冲突（咨询公司常为客户宣传趋势以推动购买）。
- 假设 4：技术发展呈线性可预测，无重大意外。
- 输出：`identified_assumptions` = [假设1（有争议）, 假设2（需进一步验证）, 假设3（有争议）, 假设4（有争议）]

### Step 3：评估证据质量
- 证据 1：StackOverflow 增长数据 → 来源：博客引用单一截图，无原始数据链接。可追溯性差 → 评分: 2/5
- 证据 2：Google 工程师透露 → 来源：匿名引述，无姓名、无日期、无外部验证。不可靠 → 评分: 1/5
- 证据 3：咨询公司预测 → 来源：文中未提供公司名称，只写"知名咨询公司"。无法验证 → 评分: 1/5
- 进一步搜索：实际市场研究报告（如 Gartner、麦肯锡）预测 AI 会改变程序员工作内容，但不会完全取代，且就业增长仍在。多源一致 → 评分: 4/5
- 输出：`evidence_quality_matrix` = "文章证据极弱（平均 1.3/5），对立证据较强（4/5）"

### Step 4：分析逻辑结构
- 推理形式：归纳（从三个趋势推断必然结果）+ 诉诸权威（Google 工程师、咨询公司）
- 逻辑谬误：
  - 轻率概括：三个孤立数据点不足以支撑全局结论
  - 虚假权威：匿名"工程师"和"咨询公司"不具备可验证权威性
  - 滑坡谬误：由"AI 能完成 90% 代码"直接跳到"程序员被取代"缺乏中间环节（如企业实际采用速度、社会因素等）
  - 幸存者偏差：可能只选取支持结论的数据，忽略反例（如大量新项目仍在招聘程序员）
- 输出：`logic_report` = "发现 4 处明显逻辑谬误，论证结构严重缺陷。"

### Step 5：考虑替代解释
- 解释 A：文章是广告软文，旨在推销某低代码产品（搜索作者背景发现其关联销售低代码平台） → 证据支持度：高
- 解释 B：作者过度简化趋势，实际发展是程序员工作内容转变而非取代（历史类比：计算机取代算盘计量人，但新增无数其他岗位） → 证据支持度：高
- 解释 C：文章中提到的数字可能存在夸大或误差（如 300% 增长可能是从极少基数开始） → 证据支持度：中
- 输出：`alternative_explanations` = [解释A（证据充分）、解释B（强证据）、解释C（中等证据）]

### Step 6：形成独立判断
- 结论：文章主张"AI 将在三年内取代所有程序员"极不可信。核心证据薄弱，逻辑谬误众多，且存在利益冲突。
- 较合理的判断是：AI 会深刻改变编程工作，但程序员就业不会消失，而是在转型中创造新的岗位。
- 置信度：高（90%）
- 剩余不确定性：如果出现突破性技术奇点，但概率极低（<1%），暂不考虑。
- 输出：`final_judgment` = "拒绝原始主张；采纳替代解释 B 为主要立场。"

### 最终输出（可交付格式）

```
批判性思维分析结果：
议题：AI在三年内取代所有程序员
得分：证据强度 1.3/5，逻辑谬误 4处，替代解释合理性 高
结论：不可信。分析表明文章存在利益冲突和严重逻辑缺陷。
建议：采用替代解释——AI促进程序员技能转型，而非取代。
置信度：高（90%）
```

## 示例 2：评估产品需求

### 场景
产品经理提出需求："用户调研显示 80% 的受访者希望增加"暗黑模式"，所以我们应该在下一个版本推出暗黑模式。"

作为开发者，你打算用批判性思维评估这个需求。

### Step 1：明确议题和立场
- **议题**：基于"80% 受访者希望"推出暗黑模式是否合理。
- **核心概念**："用户调研"需要明确：样本量、人群覆盖、提问方式。
- **输出**：`清晰议题` = "当前用户调研结果是否足以支持将暗黑模式列为下个版本的开发项。"

### Step 2：识别假设和前提
- 假设 1："80%" 数据无偏差，样本能代表全体用户。
- 假设 2：受访者真实需要暗黑模式（而非社交期望偏差）。
- 假设 3：实现暗黑模式是最高优先级，优于其他改进。
- 假设 4：暗黑模式对用户满意度提升远大于其开发成本。
- 输出：`identified_assumptions` = [假设1（需验证）, 假设2（需验证）, 假设3（有争议）, 假设4（有争议）]

### Step 3：评估证据质量
- 询问产品经理调研细节后，得到信息：
  - 样本数：100 人（产品内弹窗问卷）
  - 用户筛选：只针对已激活用户
  - 提问方式："您是否希望增加暗黑模式？"（是/否）
- 评估：
  - 样本量不足（100/100,000 用户 = 0.1%），且未随机抽样（仅活跃用户）→ 偏差高风险
  - 提问方式诱导"是"（通常用户倾向于对改进选项回答"是"，而不会考虑成本）
  - 未提供其他选项比较，无法判断优先级
  - 评分：2/5
- 输出：`evidence_quality_matrix` = "证据质量低，存在严重抽样偏差和提问偏差。"

### Step 4：分析逻辑结构
- 推理形式：归纳（80% 数据频次 → 用户需求）
- 逻辑谬误：
  - 轻率概括：100 人不代表整个用户群体
  - 偷换概念："希望"不等于"愿意付出成本"或"对提升留存有帮助"
  - 假两难：隐含认为只有做/不做两个选项，忽略了其他更重要的需求
- 输出：`logic_report` = "存在 3 处逻辑缺陷，证据不足以支撑决策。"

### Step 5：考虑替代解释
- 解释 A：用户期望暗黑模式，但若与修复关键 Bug 或提升性能相比，优先级较低 → 证据支持度：高
- 解释 B：调研中肯定回答率 80% 是"默认偏好"现象，实际会使用的用户比例远低 → 证据支持度：高
- 解释 C：暗黑模式是竞品标配，用户因其他不满而迁移，但调研未涉及根本原因 → 证据支持度：中
- 输出：`alternative_explanations` = [解释A（强证据）, 解释B（强证据）, 解释C（中等证据）]

### Step 6：形成独立判断
- 结论：基于当前调研数据，不能直接将"开发暗黑模式"定为首要任务。需要补充更严谨的调研（扩大样本、加入优先级比较、测量实际使用意愿）。
- 置信度：中高（85%）
- 建议：先进行 A/B 测试或细化调研，同时与其他需求（如性能改进）做优先级投票。
- 输出：`final_judgment` = "延迟决策，先补充数据。"

### 最终输出

```
批判性思维分析结果：
议题：基于80%调研结果推出暗黑模式
证据质量：2/5，偏差风险高
逻辑缺陷：轻率概括、偷换概念、假两难
替代解释更合理：用户偏好不等于真实高优先级
结论：不应立即采纳，建议补充跨群体调研和优先级比较。
置信度：85%
```

## 反例：不应触发批判性思维

### 场景
同事说："我刚看到窗外下雨了，你出门记得带伞。"

### 为什么不该调用批判性思维
这属于**简单事实陈述**，直接来自个人即时感知，且为善意的提醒。批判性思维不适用于以下场景：
- 事实传递（窗户可观察到的降雨）
- 无争议的日常提醒
- 不需要评估逻辑或证据

若在此场景中启动 6 步分析：
- 你会浪费时间质疑同事的观察准确性
- 假设同事故意戏弄你
- 检查外部的天气预报作为证据
- 分析是否有逻辑谬误——完全没有
- 最终结论"我信任同事的话，需要带伞"与直接接受并无二致

**正确做法**：直接采纳提醒，不再分析。只有当你怀疑同事误报时才需要调用（例如：所在楼层很高，无法直接看到地面下雨，但同事说下雨了，此时需要简单核实）。但即使这样也只需快速核实，不必完整执行批判性思维流程。
