# Gotchas

> 只写安全测试容易犯错的**非显而易见的事实**。通用知识不要写。

## Gotcha 1：安全扫描误报未验证直接报告

**问题**：将自动化扫描工具的结果直接作为漏洞报告，未进行人工验证，导致大量误报浪费开发团队时间。

❌ **Before（错误做法）**
```markdown
# 安全扫描报告（直接导出）

## 高危漏洞：SQL 注入
- **URL**：https://example.com/api/search?q=test
- **工具**：OWASP ZAP
- **证据**：参数 q 可能存在注入

## 高危漏洞：XSS
- **URL**：https://example.com/profile?name=test
- **工具**：OWASP ZAP
- **证据**：参数 name 可能存在 XSS
```

✅ **After（正确做法）**
```markdown
# 安全测试报告（验证后）

## 确认漏洞：SQL 注入
- **URL**：https://example.com/api/search?q=test
- **验证过程**：
  1. 发送 `q=test' OR '1'='1`，返回所有数据
  2. 发送 `q=test; DROP TABLE users--`，表被删除（测试环境）
  3. 确认存在 UNION 注入，可提取任意数据
- **风险**：数据库完全暴露，可提取、修改、删除任意数据
- **修复建议**：使用参数化查询，禁止字符串拼接 SQL

## 误报排除：XSS
- **URL**：https://example.com/profile?name=test
- **验证过程**：
  1. 发送 `name=<script>alert(1)</script>`
  2. 响应中 `<script>` 被转义为 `&lt;script&gt;`
  3. 浏览器未执行脚本
- **结论**：不存在 XSS，输出已正确编码
```

**原因**：自动化扫描工具有 30-50% 的误报率。未验证的误报会严重损害安全团队的可信度，并浪费开发资源。每个高危漏洞必须手动验证并记录验证过程。

## Gotcha 2：渗透测试后未清理后门

**问题**：渗透测试过程中创建的后门账号、WebShell、持久化机制在测试结束后未清理，成为真实安全隐患。

❌ **Before（错误做法）**
```bash
# 渗透测试时创建后门账号
useradd pentest_backdoor
# 测试结束后直接离场，未删除账号
```

✅ **After（正确做法）**
```bash
# 1. 测试前记录所有将要创建的后门/账号
# backdoors.txt
- 账号：pentest_backdoor（192.168.1.100）
- WebShell：/tmp/shell.php（192.168.1.200）
- 计划任务：/etc/cron.d/pentest（192.168.1.100）

# 2. 测试结束后逐项清理
userdel pentest_backdoor
rm -f /tmp/shell.php
rm -f /etc/cron.d/pentest

# 3. 验证清理结果
cat /etc/passwd | grep pentest  # 应无输出
find / -name "shell.php" 2>/dev/null  # 应无输出
crontab -l | grep pentest  # 应无输出

# 4. 提交清理确认报告
```

**原因**：渗透测试创建的后门如果未清理，会成为真实的攻击入口。曾有案例因测试后门未清理导致真实入侵。必须建立后门清单、逐项清理、验证确认。

## Gotcha 3：只关注技术漏洞忽略业务逻辑漏洞

**问题**：安全测试只关注 SQL 注入、XSS 等技术漏洞，忽略业务逻辑漏洞（如支付绕过、越权访问），而这些往往危害更大。

❌ **Before（错误做法）**
```markdown
# 安全测试报告
- SQL 注入：0 个
- XSS：1 个（低危）
- CSRF：0 个
- 结论：系统安全
```

✅ **After（正确做法）**
```markdown
# 安全测试报告

## 技术漏洞
- SQL 注入：0 个
- XSS：1 个（低危）
- CSRF：0 个

## 业务逻辑漏洞（高危）

### 支付绕过
- **问题**：修改订单金额为 0 可成功支付
- **复现**：
  1. 创建订单，金额 1000 元
  2. 拦截支付请求，将 amount 改为 0
  3. 支付成功，订单状态变为已支付
- **风险**：可零元购任意商品
- **修复**：服务端校验订单金额，不以客户端传入为准

### 水平越权
- **问题**：修改订单 ID 可查看他人订单
- **复现**：
  1. 登录用户 A，查看订单 /orders/1001
  2. 修改 URL 为 /orders/1002（用户 B 的订单）
  3. 成功查看用户 B 的订单详情
- **风险**：用户数据泄露
- **修复**：服务端校验订单归属，只能访问自己的订单
```

**原因**：业务逻辑漏洞（支付绕过、越权、并发竞争）无法通过自动化扫描发现，但危害往往比技术漏洞更大。安全测试必须包含业务逻辑测试。

## Gotcha 4：安全测试未覆盖供应链和第三方组件

**问题**：只测试自研代码，忽略第三方组件、开源库、供应链的安全风险。

❌ **Before（错误做法）**
```markdown
# 安全测试范围
- 自研代码：已测试
- 第三方组件：未测试（假设安全）
```

✅ **After（正确做法）**
```markdown
# 安全测试范围
- 自研代码：已测试
- 第三方组件：已扫描

## 第三方组件扫描结果
| 组件 | 版本 | 已知漏洞 | 风险等级 |
|------|------|---------|---------|
| log4j | 2.14.0 | CVE-2021-44228 (Log4Shell) | 🔴 严重 |
| spring-boot | 2.5.0 | CVE-2022-22965 | 🔴 严重 |
| lodash | 4.17.20 | CVE-2021-23337 | 🟡 中等 |

## 修复建议
1. 立即升级 log4j 到 2.17.1+
2. 升级 Spring Boot 到 2.6.6+
3. 升级 lodash 到 4.17.21+
```

**原因**：第三方组件漏洞（如 Log4Shell、Spring4Shell）已成为主要攻击向量。必须使用 SCA（软件成分分析）工具扫描依赖，及时修复已知漏洞。
