# 技术交付物模板

> 提供可直接使用的 Markdown/代码模板。

## 交付物模板 1：前端架构决策记录（ADR）

```markdown
# ADR-XXX: [决策标题]

## 状态
- 提议 / 已通过 / 已废弃

## 背景
[描述促使做出此决策的问题或需求]

## 决策
[明确描述做出的决策]

## 候选方案对比

| 维度 | 方案A: [名称] | 方案B: [名称] | 方案C: [名称] |
|------|--------------|--------------|--------------|
| 学习成本 | 高/中/低 | 高/中/低 | 高/中/低 |
| 开发效率 | 高/中/低 | 高/中/低 | 高/中/低 |
| 性能 | 优/良/差 | 优/良/差 | 优/良/差 |
| 生态成熟度 | 高/中/低 | 高/中/低 | 高/中/低 |
| 团队熟悉度 | 高/中/低 | 高/中/低 | 高/中/低 |
| 风险 | [描述] | [描述] | [描述] |

## 影响
- **正面影响**：...
- **负面影响**：...
- **风险**：...

## 相关决策
- [ADR-XXX: 相关决策]
```

## 交付物模板 2：组件设计规范

```markdown
# 组件设计规范: [组件名称]

## 组件定位
- **类型**：基础组件 / 业务组件 / 页面组件
- **职责**：一句话描述组件职责
- **复用范围**：全局 / 模块内 / 页面内

## Props 接口

| 属性 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| prop1 | string | 是 | - | 属性说明 |
| prop2 | number | 否 | 0 | 属性说明 |
| onEvent | () => void | 否 | - | 事件回调 |

## 状态管理
- **内部状态**：使用 useState 管理 ...
- **外部状态**：通过 Props 接收 ...

## 使用示例

```tsx
<ComponentName
  prop1="value"
  prop2={42}
  onEvent={() => console.log('triggered')}
/>
```

## 注意事项
- [ ] 边界条件处理
- [ ] 性能优化点
- [ ] 可访问性要求
```

## 交付物模板 4：代码审查清单

```markdown
# 代码审查: [PR/分支名称]

## 总体印象
[🟢 通过 / 🟡 有条件通过 / 🔴 需要修改]

## 架构设计
- [ ] 组件拆分是否合理（单一职责）
- [ ] 状态管理是否清晰（避免过度全局化）
- [ ] 数据流是否可追踪（Props → State → Event）

## 代码质量
- [ ] TypeScript 类型是否完整（无 implicit any）
- [ ] 错误处理是否完善（try-catch、边界条件）
- [ ] 代码复用性（是否存在重复逻辑可提取）

## 性能
- [ ] 是否存在不必要的重渲染
- [ ] 图片/资源是否优化（压缩、懒加载）
- [ ] Bundle 体积影响（新依赖评估）

## 可维护性
- [ ] 命名是否清晰（变量、函数、组件）
- [ ] 注释是否必要且准确
- [ ] 测试覆盖率（新增代码是否包含测试）

## 关键问题（🔴 Blocker）
- [ ] **问题1**：[位置] — [问题描述] — [建议修改]

## 改进建议（🟡 Suggestion）
- [ ] **建议1**：[位置] — [描述] — [理由]

## 表扬（💭 Nit）
- ✅ [位置] — [好的实践] — [为什么好]
```
