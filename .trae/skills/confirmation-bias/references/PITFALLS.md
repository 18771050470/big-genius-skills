# 确认偏误常见陷阱

## 陷阱 1：过度矫正——为反对而反对，导致反向确认偏误

**问题描述**：使用者听说“要克服确认偏误”，于是在论证时**故意选择与初始观点相反的方向**，同等忽略支持原始观点的合理证据。这本质上是另一种确认偏误（反向版），并未真正基于证据。

**❌ 反例**：团队讨论是否采用微服务架构。工程师小李初始观点是“微服务好”，但他想起要克服偏误，于是推翻自己，强行主张“单体架构更好”，并只收集单体架构的优点（如同步简单、部署容易），完全忽略微服务的可扩展性、故障隔离等优点。最终结论偏向单体，但缺乏对微服务优点的公正评估。

**✅ 正例**：小李应该按照本 Skill 流程：先记录初始观点（微服务好），然后公平地列出支持与反对证据。如果反对证据确实更强，则修改结论；如果支持证据仍然充分，则保留原观点。最终，经过对比，小李发现微服务在团队规模和业务阶段下仍是合理选择，但保留了对单体优点的承认（如“单体更适合短期快速迭代”），置信度调整为中高。

**原因分析**：对抗偏误的方法不是“倒向对立面”，而是“同时公平对待双方”。过度矫正引入新的盲区：只反对而不支持，同样是片面的。必须用流程强制正反兼顾。

## 陷阱 2：用确认偏误去指责他人，却忽略自己的确认偏误

**问题描述**：使用本 Skill 时，把自己置于“真理裁判”位置，只检查别人是否存在确认偏误，而对自己的判断免于审视。例如在团队辩论中，不断指出对方的证据筛选问题，但自己列举的反对证据也同样有选择性。

**❌ 反例**：产品经理小赵对运营经理小钱说：“你只看到分享功能的成功案例，这就是确认偏误！你应该看看那些失败案例。” 小赵自己却只收集了留存功能成功的案例，完全没有去了解留存功能可能带来的负面影响（如用户疲劳）。双方互相指责对方有偏误，但各自都只支持自己。

**✅ 正例**：小赵在指出对方之前，先做自我检查：使用自检清单，诚实地审视自己是否也忽略了留存功能的反对证据。他发现自己确实收集了 5 条留存成功案例，但未去看留存导致的推送过度或用户流失数据。于是他先修正自己的立场，再以“我们可能都有偏误，不如一起正反对比”的协作方式邀请小钱共同完成流程。

**原因分析**：确认偏误具有“盲点效应”——人容易发现别人的偏误但看不到自己的。本 Skill 的使用者必须将自己也纳入检查范围，否则会陷入新的偏误（所谓“偏误的偏误”）。正确的做法是先自查，再邀请对方一起使用本流程，构建共享的证据基础。
