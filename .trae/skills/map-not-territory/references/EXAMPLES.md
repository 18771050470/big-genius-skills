# 分析示例

## 示例 1：地图不是疆域——KPI驱动决策

**输入**：这个月用户活跃度KPI达标了，说明产品很健康，继续保持。

```
Step 1: 识别模型依赖 → 产出：发现"KPI达标=产品健康"表述，将指标等同于现实
Step 2: 评估模型-现实差距 → 产出：KPI是简化指标，可能遗漏：用户满意度下降、核心功能使用率低、竞品威胁
Step 3: 分析模型局限 → 产出：Goodhart定律：当KPI成为目标后，可能被优化而非反映真实健康度
Step 4: 校正模型偏差 → 产出：建议用多维度指标+定性反馈校准KPI
```

**输出**：
```
## 分析结论
### 核心发现
1. KPI达标不等于产品健康，KPI是简化模型
2. Goodhart定律：指标变目标后可能失真（如刷活跃度）
3. 需要结合定性反馈（用户访谈、NPS）校准

### 建议（按优先级）
1. [高优先级] 增加定性指标：用户访谈、NPS、留存原因分析
2. [高优先级] 检查KPI是否被"优化"而非自然增长
3. [中优先级] 建立多维度健康度仪表盘

### 关键假设
- KPI可能被操纵
- 定性反馈可获取

### 风险
- 仅看KPI可能错过早期预警信号
- 需要平衡定量和定性

### 下一步
建议调用 [critical-thinking] 质疑KPI设计假设
```

## 示例 2：地图不是疆域——架构模型

**输入**：按微服务架构理论，拆成10个服务是最优的，我们应该这样设计。

```
Step 1: 识别模型依赖 → 产出：发现"按理论...最优"表述，将架构模型当作绝对真理
Step 2: 评估模型-现实差距 → 产出：微服务理论假设：团队独立部署、服务边界清晰、运维能力充足；现实可能不满足
Step 3: 分析模型局限 → 产出：微服务适合大规模团队，小团队可能因运维复杂度反而降低效率
Step 4: 校正模型偏差 → 产出：建议基于团队规模和业务复杂度选择架构
```

**输出**：
```
## 分析结论
### 核心发现
1. 微服务是模型，不是绝对真理，适用条件：大团队、复杂业务
2. 现实约束：团队规模、运维能力、业务阶段可能不满足理论假设
3. 架构选择应基于现实而非理论最优

### 建议（按优先级）
1. [高优先级] 评估团队规模：<5人建议单体，>10人考虑微服务
2. [高优先级] 评估业务复杂度：简单业务单体更高效
3. [中优先级] 渐进式拆分，而非一步到位

### 关键假设
- 团队规模<10人
- 业务处于早期

### 风险
- 过早微服务可能导致运维负担
- 需要平衡理论和现实

### 下一步
建议调用 [occams-razor] 评估架构复杂度是否必要
```
