# 常见陷阱

## 陷阱 1：用当前状态预测他人体验

**问题**：因自己熟悉而无法想象他人的困惑。

❌ **Before（错误做法）**
```markdown
## 分析结论
这个界面我觉得很简单，用户肯定能轻松上手。
建议：不需要可用性测试。
```
*问题：共情差距使开发者无法预测用户的困惑*

✅ **After（正确做法）**
```markdown
## 分析结论
### 认知状态对比
| 角色 | 熟悉度 | 认知负荷 |
|------|--------|---------|
| 开发者 | 高(每天使用) | 低 |
| 首次用户 | 低(第一次接触) | 高 |

### 建议
1. 开发者熟悉度≠用户易用性
2. 需要真实用户测试验证
```
*优势：揭示认知状态差异*

**原因**：Loewenstein(1996)冷热共情差距：冷静时无法预测情绪状态下的行为，熟悉时无法预测新手的困惑。

## 陷阱 2：冷热状态转换失败

**问题**：冷静时制定的计划，情绪波动时无法执行。

❌ **Before（错误做法）**
```markdown
## 分析结论
我现在很冷静，这个投资决策很理性，明天执行。
建议：按计划执行。
```
*问题：共情差距使冷静状态无法预测情绪状态下的行为*

✅ **After（正确做法）**
```markdown
## 分析结论
### 状态依赖分析
| 状态 | 决策倾向 | 风险 |
|------|---------|------|
| 冷静 | 理性分析 | 低 |
| 恐慌 | 非理性卖出 | 高 |
| 贪婪 | 过度买入 | 高 |

### 建议
1. 预设决策规则，情绪波动时自动执行
2. 书面记录决策逻辑，情绪时回顾
```
*优势：预设规则降低情绪影响*

**原因**：冷热共情差距使人们低估情绪对决策的影响。预设规则是有效应对策略。

## 陷阱 3：知识诅咒

**问题**：知道某事后无法想象不知道的状态。

❌ **Before（错误做法）**
```markdown
## 分析结论
这个功能逻辑很简单，用户一看就懂。
建议：不需要写说明文档。
```
*问题：知识诅咒使无法预测用户的理解难度*

✅ **After（正确做法）**
```markdown
## 分析结论
### 知识状态对比
| 维度 | 开发者 | 用户 |
|------|--------|------|
| 产品知识 | 完整 | 零 |
| 使用经验 | 丰富 | 无 |
| 理解难度 | 低 | 高 |

### 建议
1. 知道后无法想象不知道的状态
2. 提供清晰的引导和文档
```
*优势：识别知识诅咒，提供用户支持*

**原因**：Camerer(1989)知识诅咒：知道信息后，无法准确预测不知道信息的人的理解难度。
