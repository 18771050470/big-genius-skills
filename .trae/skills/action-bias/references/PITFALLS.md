# 常见陷阱

## 陷阱 1：将"立即行动"等同于"负责任"

**问题**：认为快速响应就是好，忽视等待可能产生更好结果。

❌ **Before（错误做法）**
```markdown
## 分析结论
用户投诉应该立即回复，越快越好。
建议：5分钟内必须响应。
```
*问题：行动偏差使将响应速度等同于服务质量，忽视解决质量*

✅ **After（正确做法）**
```markdown
## 分析结论
### 核心发现
1. 快速响应≠高质量解决，30%立即回复因信息不足导致二次沟通
2. 等待收集完整信息后回复，首次解决率提升40%
3. 用户满意度与首次解决率相关性(r=0.7)远高于响应速度(r=0.3)

### 建议（按优先级）
1. [高优先级] 建立"行动vs等待"决策树
2. [高优先级] 将首次解决率纳入KPI
```
*优势：基于数据区分响应速度和解决质量，识别最优策略*

**原因**：行动偏差源于规范理论——行动是规范，不行动被视为失职。Bar-Eli等(2006)研究证明守门员扑点球时94%选择跳向一侧，尽管统计显示留在中央扑救率最高。

## 陷阱 2：为行动而行动

**问题**：因汇报压力或可见性偏好而行动，非因实际需求。

❌ **Before（错误做法）**
```markdown
## 分析结论
这个版本必须上新功能，不然显得团队没产出。
建议：规划3个新功能。
```
*问题：行动偏差使将功能数量等同于产出价值*

✅ **After（正确做法）**
```markdown
## 分析结论
### 产出价值对比
| 方案 | 开发成本 | 预期收益 | 风险 |
|------|---------|---------|------|
| 3个新功能 | 15人日 | 留存+2% | 中(新功能bug) |
| 优化核心功能 | 8人日 | 留存+15% | 低 |

### 建议
1. 优化核心功能ROI更高(15%/8人日 vs 2%/15人日)
2. 向老板汇报留存提升数据，而非功能数量
```
*优势：量化比较揭示优化优于新功能，打破"新=好"假设*

**原因**：行动偏差使偏好可见产出（新功能），忽视隐性改进（优化）。量化ROI提供客观比较。

## 陷阱 3：忽视"不行动也是决策"

**问题**：将不行动视为不作为，而非主动选择。

❌ **Before（错误做法）**
```markdown
## 分析结论
我们决定暂时不修改这个模块。
建议：保持现状。
```
*问题：将"不修改"描述为被动维持，而非主动决策*

✅ **After（正确做法）**
```markdown
## 分析结论
### 主动决策记录
决策：本版本不修改X模块
理由：
1. 修改风险>收益（回归测试成本10人日，预期收益<1%）
2. 下季度有架构升级，届时一并处理更高效
3. 当前模块稳定性99.9%，无紧急修改需求

### 建议
1. 记录此决策及依据，便于后续回顾
2. 设置触发条件：若稳定性<99.5%则重新评估
```
*优势：将不行动框架化为主动决策，有依据有触发条件*

**原因**：行动偏差使不行动被框架化为"不作为"。主动决策框架将不行动视为经过分析的选择。
