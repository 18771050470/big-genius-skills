# 分析示例

## 示例 1：聚光灯效应——公开演讲焦虑

**输入**：我明天要在大会上演讲，万一忘词了所有人都会记住我的尴尬，太丢人了。

```
Step 1: 识别聚光灯信号 → 产出：发现"所有人都会记住"表述，高估他人对自己的关注度
Step 2: 评估关注程度 → 产出：听众更关注内容而非演讲者表现，忘词会被快速遗忘
Step 3: 分析聚光灯机制 → 产出：自我中心偏差：自己是自己世界的中心；透明度错觉：认为自己的紧张很明显
Step 4: 校正聚光灯效应 → 产出：建议用数据校准关注度估计，准备应对策略
```

**输出**：
```
## 分析结论
### 核心发现
1. "所有人都会记住"是聚光灯效应，实际听众关注度远低于预期
2. Gilovich等(2000)实验：穿尴尬T恤的人估计50%人注意到，实际只有20%
3. 听众更关注内容价值，而非演讲者小失误

### 建议（按优先级）
1. [高优先级] 准备忘词应对策略：看笔记、停顿、过渡句
2. [高优先级] 理解听众注意力有限，不会持续关注你的表现
3. [中优先级] 练习降低紧张感，但接受适度紧张是正常的

### 关键假设
- 听众更关心内容而非表现
- 小失误会被快速遗忘

### 风险
- 过度焦虑可能影响表现
- 需要平衡准备和放松
```

## 示例 2：聚光灯效应——代码审查焦虑

**输入**：我的代码被review了，如果有bug所有人都会觉得我不专业。

```
Step 1: 识别聚光灯信号 → 产出：发现"所有人都会觉得"表述，高估他人对自己的负面评价
Step 2: 评估关注程度 → 产出：reviewer关注代码质量而非个人能力，bug是正常现象
Step 3: 分析聚光灯机制 → 产出：自我中心偏差：认为自己的错误很突出；持久性错觉：认为错误会被长期记住
Step 4: 校正聚光灯效应 → 产出：建议理解bug是开发常态，关注改进而非完美
```

**输出**：
```
## 分析结论
### 核心发现
1. "所有人都会觉得不专业"是聚光灯效应，reviewer更关注代码而非个人
2. 行业数据：平均每个PR有3-5个评论，bug是正常开发过程
3. reviewer自己也有bug被指出，理解这是协作改进过程

### 建议（按优先级）
1. [高优先级] 将code review视为学习机会，而非能力评判
2. [高优先级] 关注具体反馈，改进代码质量
3. [中优先级] 理解reviewer不会长期记住你的bug

### 关键假设
- 团队文化是建设性review
- reviewer理解bug是常态

### 风险
- 焦虑可能导致回避code review
- 需要区分建设性反馈和人身攻击
```

## 反例：不该触发此 Skill 的场景

**原因**：这是真实的高关注度事件（CEO公开批评），不是聚光灯效应。聚光灯效应指高估不存在的关注。
