# 分析示例

## 示例 1：绩效评估中的基本归因错误

**输入**：小李这个月业绩不达标，肯定是他不够努力，态度有问题。

```
Step 1: 识别归因信号 → 产出：发现"肯定是他不够努力"表述，将行为归因于个人特质
Step 2: 评估归因模式 → 产出：忽视情境因素（市场变化、资源支持、任务难度）
Step 3: 分析归因机制 → 产出：焦点效应：人比情境更显著；认知捷径：特质归因比情境分析更容易
Step 4: 校正归因错误 → 产出：建议分析情境因素，收集完整信息
```

**输出**：
```
## 分析结论
### 核心发现
1. 业绩不达标可能由多种因素导致，非仅个人努力
2. 情境因素：本月市场下滑15%、小李负责的区域竞品降价30%、新员工培训占用时间
3. 小李历史业绩：过去6个月平均达标率110%，本月是首次不达标

### 建议（按优先级）
1. [高优先级] 分析情境因素对业绩的影响程度
2. [高优先级] 与小李沟通了解具体困难
3. [中优先级] 提供资源支持（竞品应对策略、时间管理）

### 关键假设
- 历史业绩反映真实能力
- 情境因素可量化分析

### 风险
- 错误归因可能导致人才流失
- 忽视情境因素可能错过系统性问题
```

## 示例 2：项目失败归因中的基本归因错误

**输入**：这个项目失败完全是产品经理能力不行，需求没想清楚就开干。

```
Step 1: 识别归因信号 → 产出：发现"完全是产品经理能力不行"表述，将失败归因于个人
Step 2: 评估归因模式 → 产出：忽视系统因素（流程、资源、协作、决策机制）
Step 3: 分析归因机制 → 产出：基本归因错误：高估个人因素，低估情境因素
Step 4: 校正归因错误 → 产出：建议系统分析失败原因，而非归咎个人
```

**输出**：
```
## 分析结论
### 核心发现
1. 项目失败通常由多因素导致，非单一个人问题
2. 系统因素：需求评审流程缺失、跨部门协作不畅、资源不足、时间压力
3. 产品经理个人因素：需求分析能力确实有提升空间，但非唯一原因

### 建议（按优先级）
1. [高优先级] 系统分析失败原因：流程、资源、协作、个人
2. [高优先级] 改进需求评审流程，避免类似问题
3. [中优先级] 为产品经理提供培训支持

### 关键假设
- 系统改进比个人追责更有效
- 失败是学习机会，非追责工具

### 风险
- 单纯归咎个人可能掩盖系统性问题
- 团队可能因恐惧而不敢创新
```

## 反例：不该触发此 Skill 的场景

**输入**：小李连续3个月业绩垫底，且多次违反公司规定。

**原因**：这是基于持续行为模式的评估，有充分证据支持个人因素，不是基本归因错误。
